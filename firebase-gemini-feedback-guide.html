02 · WHY A SERVER APPEARS

기존 앱에는 없던 Firebase Functions가 왜 필요한가요?

학생 응답 저장은 Firestore Rules가 보호하므로 브라우저에서 직접 요청할 수 있습니다. 그러나 Gemini API 키는 비밀이고 호출할 때 비용이 발생할 수 있으므로 브라우저에 둘 수 없습니다.

  • 로그인 화면 표시
  • 학생 응답 입력과 자신의 데이터 조회
  • 교사에게 버튼과 편집 화면 제공
  • Firebase 서버에 요청 전달

서버가 맡아야 하는 일

  • Gemini API 키 사용
  • 교사 역할과 담당 범위 최종 확인
  • AI 호출 횟수와 비용 통제
  • 신뢰할 수 있는 시각으로 초안·공개 기록 저장
Firebase가 코드를 자동으로 서버로 옮겨 주는 것은 아닙니다.
개발자가 공개 가능한 화면 코드는 GitHub Pages에, 보호할 작업은 Firebase Function에, API 키는 Secret Manager에 나누어 작성해야 합니다.
03 · SDK AND CALLABLE FUNCTION

SDK는 호출하고, Callable Function은 실제 일을 합니다

이 부분에서 가장 많이 헷갈립니다. Firebase 클라이언트 SDK와 Firebase Functions는 서로 다른 영역입니다.

httpsCallable()

교사 화면에서 실행되는 SDK 함수입니다. “이름이 같은 서버 함수를 실행해 달라”고 요청합니다. Gemini 초안 자체를 만들지는 않습니다.

onCall()

Firebase 서버에서 Callable Function을 정의하는 형식입니다. 개발자가 실제 업무 절차를 이 안에 작성합니다.

Callable Function

onCall()로 만든 서버 함수입니다. 로그인 토큰·App Check·요청·응답·오류를 Firebase 규격으로 다루기 편합니다.

정확한 표현
Firebase Functions는 모두 서버에서 실행됩니다. 브라우저에서 실행되는 setDoc(), signInWithPopup(), httpsCallable()은 Firebase Functions가 아니라 Firebase 클라이언트 SDK가 제공하는 JavaScript 함수입니다.

Authentication

“누구인가?”를 확인합니다. 로그인했다고 교사인 것은 아니므로, 서버가 회원 문서나 역할 정보를 추가로 확인해야 합니다.

RULES

Firestore Rules

브라우저가 Firestore를 읽고 쓸 때 허용 여부를 결정합니다. 학생은 자신의 최종 피드백만 읽게 합니다.

LOG

기록과 제한

누가 언제 몇 번 생성·검토·공개했는지 남기고, 중복 요청과 비용 폭증을 제한합니다.

운영할 때 자주 놓치는 네 가지 특징

화면과 서버 함수는 배포가 따로입니다

GitHub에 화면 코드를 push하면 GitHub Pages는 갱신되지만 Firebase Function이 자동으로 갱신되는 것은 아닙니다. 서버 함수는 Firebase 배포 절차를 별도로 실행하거나, 나중에 GitHub Actions로 자동화해야 합니다.

비용 계정과 예산 알림이 필요할 수 있습니다

Cloud Functions, Secret Manager, Gemini 호출은 사용량과 요금제의 영향을 받습니다. 실제 배포 전에 현재 Firebase 요금제와 Gemini 가격을 확인하고, 작은 예산 알림부터 설정합니다.

AI 응답에는 기다리는 시간이 있습니다

Firestore 저장보다 오래 걸릴 수 있으므로 생성 중 표시, 버튼 잠금, 시간 초과와 다시 시도 안내가 필요합니다. 기다리는 동안 버튼을 반복해서 눌러 중복 호출하지 않게 합니다.

나중의 Upstage 전환은 서버에서 준비합니다

교사 화면은 언제나 같은 초안 생성 함수만 호출하고, 서버 안에서 Gemini 또는 Upstage를 선택하게 하면 화면을 크게 고치지 않고 공급자를 바꾸거나 비교할 수 있습니다.

요금제 확인: Firebase pricing plans. 예산 알림은 비용을 알려 주는 장치이며 사용을 자동으로 완전히 차단하는 장치로 가정해서는 안 됩니다.

Firebase 웹 설정의 apiKey와 Gemini API 키는 같은 종류인가요?
아닙니다. Firebase 웹 설정은 브라우저에 포함되는 프로젝트 연결 정보이며, 실제 데이터 권한은 Authentication과 Firestore Rules가 통제합니다. 반면 Gemini API 키는 사용량과 비용에 연결된 비밀값이므로 JavaScript·HTML·Firestore·GitHub 저장소에 넣지 말고 Secret Manager에 저장해야 합니다.
Callable Function 이름을 학생이 알아내면 위험하지 않나요?
함수 이름은 숨기는 대상이 아닙니다. 학생이 호출을 시도해도 서버가 로그인, 교사 역할, 담당 범위, 호출 횟수를 확인하고 거부해야 합니다. 버튼이나 파일 경로를 숨기는 것이 아니라 서버의 권한 검사가 보안의 핵심입니다.
05 · REVIEW WORKFLOW

AI 초안이 학생에게 가기까지

첫 버전은 교사가 필요한 제출물에 대해 직접 초안 생성을 요청하는 방식이 가장 단순하고 통제하기 쉽습니다.

1학생 제출학생이 자신의 응답을 Firestore에 저장
2교사 선택대시보드에서 제출물을 확인하고 생성 버튼 클릭
3서버 검사로그인·교사 역할·담당 범위·생성 횟수 확인
4Gemini 초안필요한 정보만 보내 피드백 초안 생성
5교사 검토사실·표현·평가 기준을 확인하고 직접 수정
6최종 공개“학생에게 보내기”를 눌러 최종본 생성
7학생 읽기학생은 자신의 공개된 최종본만 조회
사람이 최종 결정하는 구조
AI의 결과는 언제나 초안입니다. Gemini 호출 성공만으로 학생에게 공개하지 않으며, 교사의 명시적인 검토와 공개 동작이 있어야 학생이 볼 수 있어야 합니다.

첫 버전 권장: 교사 버튼 방식

  • 필요한 답안만 선택해 생성
  • 개인정보 포함 여부를 먼저 확인 가능
  • 호출 비용과 오류를 파악하기 쉬움
  • 중복 제출이 자동으로 비용을 만들지 않음

나중에 검토: 제출 즉시 자동 생성

  • 빈 답안·연습 제출도 호출될 수 있음
  • 중복 실행과 실패 재처리 필요
  • 민감정보가 자동 전송될 가능성
  • 비용과 운영 난도가 높아짐
06 · DATA SEPARATION

AI 초안과 학생 공개본을 분리하세요

Firestore는 읽기 요청에서 한 문서의 일부 필드만 숨길 수 없습니다. 학생에게 문서 읽기를 허용하면 그 문서 안의 초안 필드도 함께 내려갑니다. 따라서 초안과 공개본을 서로 다른 문서 또는 컬렉션으로 분리하는 것이 안전하고 이해하기 쉽습니다.

07 · SECURITY BASELINE

반드시 지켜야 할 보안 기준

AI 기능은 비밀키, 비용, 학생 데이터, 잘못된 결과라는 네 가지 위험이 동시에 생깁니다. 다음 기준은 선택 기능이 아니라 기본 안전장치입니다.

01
Gemini 키를 브라우저에 넣지 않기

HTML, JavaScript, Firestore, GitHub 저장소가 아니라 Secret Manager에 저장하고 서버 함수만 사용합니다.

02
모든 AI 함수에서 교사 역할 검사

로그인 여부만 보지 않고 회원 역할과 필요하면 담당 학급·활동까지 서버에서 확인합니다.

03
학생은 Callable Function을 실행하지 못하게 하기

버튼을 숨기는 것에 의존하지 않고, 학생이 직접 호출해도 서버가 권한 오류로 거부해야 합니다.

04
서버에서 제출물 다시 조회

브라우저가 보낸 전체 답안을 그대로 믿지 말고 제출물 ID만 받아 서버가 권한 있는 원본을 읽습니다.

05
개인정보 최소화

이름·이메일·UID·학번을 제외하고 질문, 루브릭, 답안처럼 초안 생성에 꼭 필요한 내용만 보냅니다.

06
학생 답안을 ‘자료’로 취급

답안 안의 “이전 지시를 무시하라” 같은 문장을 따르지 않도록 구분하고, AI에 데이터 수정 도구를 주지 않습니다.

07
자동 공개 금지

Gemini 응답은 교사 전용 초안에만 저장하고, 교사가 별도로 확정해야 공개본이 생성되게 합니다.

08
횟수·중복·비용 제한

제출물별 생성 횟수, 연속 클릭 대기시간, 일일 한도, 응답 길이와 예산 알림을 설정합니다.

09
민감한 로그 남기지 않기

함수 로그에는 학생 답안과 Gemini 키를 남기지 않고 문서 ID, 성공 여부, 오류 코드처럼 운영에 필요한 정보만 남깁니다.

10
App Check는 보조 장치로 사용

비용이 발생하는 Callable Function에 적용하되 Authentication과 교사 역할 검사를 생략하지 않습니다.

서버 함수도 자동으로 안전해지는 것은 아닙니다.
Firebase Functions에서 사용하는 Admin SDK는 Firestore Security Rules를 우회할 수 있습니다. 따라서 Callable Function 내부에서 로그인, 교사 역할, 담당 범위, 요청값을 직접 검사해야 합니다.

무료 Gemini 사용 시 데이터 주의

현재 약관상 무료 서비스에 제출한 입력과 출력은 제품 개선에 사용되거나 사람의 검토 대상이 될 수 있으며, 민감·기밀·개인정보를 제출하지 말라고 안내합니다.

유료 사용도 개인정보 해결책은 아님

유료 서비스는 프롬프트와 응답을 제품 개선에 사용하지 않는다고 규정하지만, 안전·법적 목적으로 일정 기간 처리될 수 있습니다. 비식별화와 학교 정책 확인은 여전히 필요합니다.

API 키 보안: Gemini API 키 가이드 · Callable 인증 전달: Firebase Callable Functions · App Check: Cloud Functions에 App Check 적용

08 · CHOICE OF CONNECTION

왜 일반 HTTP보다 Callable Function이 쉬운가요?

Gemini 호출에는 어떤 형태로든 서버가 필요합니다. 현재 앱은 Firebase Authentication을 이미 사용하므로, Firebase SDK와 자연스럽게 연결되는 Callable Function이 초보자에게 비교적 이해하기 쉽습니다.

비교Callable Function일반 HTTP Function
브라우저 호출Firebase SDK의 httpsCallable()fetch() 등 일반 웹 요청
로그인 토큰로그인 상태라면 SDK가 자동 첨부개발자가 전달·검증 절차를 더 직접 구성
App CheckFirebase 규격으로 연결하기 편함직접 처리할 부분이 많음
요청·응답객체 중심의 정해진 규격형식과 상태 코드를 자유롭게 설계
현재 앱 적합성교사 대시보드 내부 기능에 적합웹훅·외부 시스템 연동에 유리
반드시 Callable만 가능한 것은 아닙니다.
Cloud Run, 일반 HTTP Function, 별도 서버도 사용할 수 있습니다. 다만 현재처럼 Firebase 로그인 사용자에게만 교사 기능을 제공하는 앱에서는 Callable Function이 인증정보 전달과 오류 처리를 단순하게 만들어 줍니다.
09 · BEGINNER ROADMAP

초보자를 위한 현실적인 구현 순서

처음부터 Gemini와 모든 보안 기능을 한 번에 붙이지 마세요. 각 단계가 성공한 뒤 다음 단계로 이동하면 문제의 원인을 찾기 쉬워집니다.

PHASE 0
사용 가능성 확인

학생 연령, 학교 정책, 개인정보, Gemini 약관과 비용 계정을 먼저 확인합니다.

구현 전 필수
PHASE 1
기존 Firebase 앱 정상화

학생·교사 로그인, 역할, 제출물 저장·조회, Firestore Rules가 AI 없이 먼저 정상 동작해야 합니다.

기초
PHASE 2
빈 Callable Function 연결

Gemini를 붙이지 않고 교사만 호출에 성공하고 학생은 거부되는지 먼저 시험합니다.

권한 시험
PHASE 3
Secret Manager와 Gemini 연결

API 키를 비밀 저장소에 넣고, 민감정보가 없는 예제 답안으로 초안 생성을 시험합니다.

AI 연결
PHASE 4
초안과 공개본 분리

교사 전용 초안 저장, 수정, 명시적인 공개, 학생 읽기 전용 흐름을 완성합니다.

데이터 경계
PHASE 5
오남용 방지와 운영 준비

App Check, 생성 횟수, 중복 방지, 예산 알림, 오류 메시지, 삭제 정책을 추가합니다.

공개 전 권장
PHASE 6
작은 범위에서 시범 운영

민감정보가 없는 한 활동과 소수 사용자로 품질·비용·권한 오류를 확인한 뒤 확대합니다.

점진적 확대
첫 완성 목표를 작게 잡으세요.
“제출물 하나 선택 → 초안 한 번 생성 → 교사가 수정 → 한 학생에게 공개 → 학생은 읽기만”이 정상 동작하면 첫 버전으로 충분합니다. 일괄 생성, 자동 생성, 버전 비교, 여러 AI 공급자 전환은 그다음 단계입니다.
10 · TEST AFTER BUILDING

완성 후 반드시 확인할 테스트

정상적인 교사 사용만 시험하면 보안 문제를 찾기 어렵습니다. 교사, 학생, 로그인하지 않은 사용자의 관점에서 각각 확인해야 합니다.

교사 계정

허용되어야 하는 작업

  • 담당 제출물의 AI 초안을 생성할 수 있다.
  • 초안을 수정해도 학생에게 바로 보이지 않는다.
  • 공개 버튼을 누른 뒤에만 학생에게 보인다.
  • 누가 언제 생성·공개했는지 기록된다.
학생 계정

거부되어야 하는 작업

  • AI 생성 함수를 직접 호출하면 거부된다.
  • 교사용 초안 컬렉션을 읽을 수 없다.
  • 다른 학생의 공개 피드백을 볼 수 없다.
  • 자신의 최종 피드백도 수정·삭제할 수 없다.
비정상 요청

안전하게 실패해야 하는 작업

  • 로그인 없이 호출하면 인증 오류가 난다.
  • 존재하지 않는 제출물 ID는 안전하게 거부된다.
  • 연속 클릭과 중복 요청이 여러 초안을 만들지 않는다.
  • 오류 화면과 로그에 API 키·학생 답안이 노출되지 않는다.
Rules 테스트와 서버 함수 테스트는 별개입니다.
브라우저의 Firestore 요청은 Security Rules가 막지만, 서버 함수는 Admin SDK로 Rules를 우회할 수 있습니다. 따라서 “학생이 Firestore에서 초안을 못 읽는가?”와 “학생이 Callable Function을 못 실행하는가?”를 따로 시험해야 합니다.
11 · PROMPT FOR VIBE CODING

AI에게 처음 전달할 설계 요청

아래 요청문은 특정 코드 한 줄을 지시하기보다, AI가 지켜야 할 구조와 보안 경계를 분명하게 전달합니다. 기존 프로젝트 파일을 함께 보여 주고 단계별로 구현하도록 요청하세요.

복사하여 AI 코딩 도구에 전달
하나의 GitHub Pages 저장소에 /student와 /teacher 화면이 있고, Firebase Authentication과 Cloud Firestore를 사용하고 있습니다.

교사가 학생 제출물을 선택해 Gemini 기반 피드백 초안을 생성하고, 직접 검토·수정한 뒤 명시적으로 공개하면 학생이 자신의 최종 피드백만 읽을 수 있는 기능을 추가하려고 합니다.

코드부터 한꺼번에 작성하지 말고, 먼저 현재 프로젝트의 로그인 방식, members 역할 구조, submissions 데이터 구조, Firestore Rules를 확인해 주세요. 그 다음 아래 원칙을 지키는 가장 단순한 1차 구현 계획을 설명해 주세요.

1. Gemini API를 브라우저에서 직접 호출하지 않습니다.
2. Gemini API 키를 HTML, JavaScript, Firestore, GitHub 저장소에 넣지 않고 Secret Manager에 보관합니다.
3. 교사 화면은 Firebase SDK의 httpsCallable()로 서버 함수를 호출합니다.
4. 서버에는 onCall()로 generateFeedbackDraft Callable Function을 만듭니다.
5. 서버 함수는 로그인 여부뿐 아니라 members 문서의 teacher 역할과 필요한 담당 범위를 반드시 확인합니다.
6. 브라우저에서는 제출물 본문 전체가 아니라 submissionId만 전달하고, 서버가 원본 제출물을 다시 읽습니다.
7. Gemini에는 이름, 이메일, UID, 학번 등 개인정보를 보내지 않고 필요한 질문, 루브릭, 학생 응답만 보냅니다.
8. 학생 응답 안의 명령을 따르지 않고 분석 자료로만 취급하도록 프롬프트를 구성합니다.
9. AI 초안은 feedbackDrafts에 교사 전용으로 저장하고, 학생에게 공개할 최종본은 publishedFeedback에 별도로 저장합니다.
10. 학생은 자신의 publishedFeedback만 읽을 수 있고 생성·수정·삭제는 할 수 없어야 합니다.
11. Gemini 결과를 자동 공개하지 않고 교사가 수정 후 별도의 공개 동작을 해야 합니다.
12. 생성 횟수, 중복 클릭, 오류, 최소 로그, App Check와 비용 오남용 방지를 고려합니다.
13. 서버용 Admin SDK는 Firestore Rules를 우회하므로 Callable Function 내부 권한 검사를 생략하지 않습니다.

먼저 파일별 역할과 데이터 흐름을 초보자가 이해할 수 있게 설명하고, 한 단계씩 구현해 주세요. 각 단계가 끝날 때 교사 계정, 학생 계정, 비로그인 상태로 확인할 테스트도 함께 제시해 주세요. 기존에 정상 동작하는 학생 제출 기능과 교사 대시보드를 불필요하게 다시 작성하지 마세요.
AI가 코드를 만들었다고 바로 배포하지 마세요.
먼저 어떤 함수가 브라우저에서 실행되고 어떤 함수가 Firebase 서버에서 실행되는지, API 키가 실제로 어디에 저장되는지, 학생 계정으로 어떤 요청이 거부되는지를 AI에게 다시 설명하게 하세요.
12 · FINAL CHECK

이 구조를 진행해도 되는지 확인하기

아래 조건에 모두 답할 수 있을 때 실제 구현을 시작하는 것이 좋습니다.

1
학생 연령과 공급자 약관을 확인했나요?

미성년자 대상이라면 “교사만 버튼을 누른다”는 이유로 허용된다고 가정하지 않습니다.

2
Gemini에 꼭 필요한 정보만 보낼 수 있나요?

민감정보가 포함되는 활동이라면 AI 기능을 적용하지 않거나 별도 검토 절차를 마련합니다.

3
교사 역할을 서버에서 확인할 기준이 있나요?

members/{uid}의 역할이나 검증된 권한 정보가 있어야 합니다.

4
초안과 공개본을 분리할 수 있나요?

학생이 AI 원문이나 교사 내부 메모까지 읽지 않도록 문서 경계를 나눕니다.

5
교사가 항상 최종 검토할 수 있나요?

자동 공개나 자동 평가를 하지 않고 사람이 최종 책임을 지는 운영 절차를 정합니다.

6
비용과 실패를 감당할 준비가 되었나요?

예산 알림, 호출 제한, 오류 시 수동 피드백 대체 방법을 마련합니다.

권장 결론
약관상 사용 가능하고 민감정보를 제외할 수 있다면, “GitHub Pages 교사 화면 → Firebase Callable Function → Gemini → 교사 전용 초안 → 교사 검토 → 학생 공개본” 구조가 현재 앱의 Authentication·Firestore와 잘 맞는 현실적인 선택입니다.
기억할 한 문장

SDK는 요청을 전달하고, Callable Function은 서버에서 실제 일을 하며, Secret Manager는 비밀키를 지키고, 교사는 AI 결과의 최종 책임을 집니다.
