Authentication · OAuth · Authorization Security Audit

로그인과 OAuth는 어떻게 다르고,
어떤 방식이 더 안전할까요?

GitHub Pages와 Firebase Firestore를 사용하는 학생·교사 웹 앱을 기준으로, Firebase 이메일·비밀번호 로그인과 Google 로그인의 작동 원리·보안 책임·운영 난도를 비교하고 실제 배포 상태를 점검합니다.

개념 관계도 두 로그인 흐름 비교 위험과 운영 책임 대화형 보안 감사
가장 중요한 구분로그인은 목적, OAuth는 가능한 방법 중 하나
공통 기반Firebase UID + ID Token
최종 권한Firestore Security Rules
개념 정리 관계도 로그인 흐름 두 방식 비교 권한 분리 보안 위험 선택 기준 수동 점검 자동 감사
01 · CORE CONCEPTS

네 단어를 먼저 분리해서 이해하기

이 용어들은 서로 경쟁하는 같은 종류의 기술이 아닙니다. 사용자가 앱에 들어오는 전체 과정, 신원 확인, 외부 계정 이용, 데이터 권한 판정이라는 서로 다른 층을 설명합니다.

로그인 · Login

사용자가 앱에서 자신의 신원을 증명하고 로그인 상태를 만드는 전체 행위와 결과입니다.

인증 · Authentication

“이 사용자가 누구인가?”를 확인하는 과정입니다. Firebase Authentication이 담당합니다.

OAuth 2.0

다른 서비스의 권한을 비밀번호 전달 없이 위임하기 위한 표준입니다. 로그인에도 활용됩니다.

인가 · Authorization

확인된 사용자가 “무엇을 할 수 있는가?”를 판단합니다. Firestore Rules가 담당합니다.

정확한 표현: Google 로그인은 보통 OAuth 2.0 기반의 OpenID Connect를 이용합니다. OAuth 2.0이 권한 위임의 틀을 제공하고, OpenID Connect의 ID Token이 로그인한 사용자의 신원 정보를 전달합니다.

03 · SIGN-IN FLOWS

두 로그인 방식은 어디에서 비밀번호를 확인할까요?

Firebase 이메일·비밀번호

OAuth 사용 안 함

  1. 사용자가 앱에 이메일과 비밀번호를 입력합니다.
  2. Firebase SDK가 Firebase Authentication으로 로그인 요청을 보냅니다.
  3. Firebase가 자체 계정의 비밀번호를 확인합니다.
  4. 성공하면 Firebase UID와 ID Token을 발급합니다.
  5. Firestore Rules가 토큰의 UID와 역할로 접근을 판단합니다.

비밀번호를 아는 곳: 사용자와 Firebase Authentication

Google 계정 로그인

OAuth 2.0 + OpenID Connect

  1. 사용자가 앱에서 Google 로그인을 선택합니다.
  2. Google이 계정 로그인과 필요한 동의를 처리합니다.
  3. Google이 확인된 신원 정보를 담은 ID Token을 제공합니다.
  4. Firebase가 Google 인증 결과를 Firebase 사용자와 연결합니다.
  5. Firebase UID와 ID Token으로 Firestore Rules를 통과합니다.

Google 비밀번호를 아는 곳: 사용자와 Google

이메일·비밀번호 로그인 코드

const credential = await signInWithEmailAndPassword(
  auth,
  email,
  password
);

console.log(credential.user.uid);

Google 로그인 코드

const provider = new GoogleAuthProvider();
const credential = await signInWithPopup(auth, provider);

console.log(credential.user.uid);
둘 다 브라우저에서 사용할 수 있지만 정답 비밀번호를 코드에 저장하지 않습니다. 이메일·비밀번호 방식은 Firebase가, Google 로그인은 Google이 자격 증명을 확인합니다. JavaScript의 if (password === "1234")와는 완전히 다른 구조입니다.

  • Google 계정 사용이 정책상 어려움
  • 별도 앱 계정이 필요함
  • 계정 발급·재설정·종료를 관리할 수 있음
  • 학생이 실제 이메일을 보유하거나 관리자 운영이 가능함

Google 로그인이 적합한 경우

  • 모든 학생이 학교 Google 계정을 사용함
  • 학교 계정의 비밀번호 정책과 비활성화를 활용하고 싶음
  • 교사가 별도 비밀번호를 관리하고 싶지 않음
  • 팝업·리디렉션과 승인 도메인 설정이 가능함
로그인 방법만으로 교사 권한이 생기지는 않습니다. Google 계정으로 로그인했다고 모두 교사가 아니며, 강한 비밀번호로 로그인했다고 교사가 되는 것도 아닙니다. 교사 권한은 Custom Claims, 안전한 역할 문서 또는 교사 UID로 별도 판정해야 합니다.
05 · AUTHORIZATION

로그인 후에도 Firestore 권한 검사가 필요합니다

Authentication은 사용자를 확인하지만, 학생이 자기 자료만 볼지 교사가 전체 자료를 볼지는 결정하지 않습니다. 이 결정은 Authorization, 즉 Firestore Security Rules의 역할입니다.

function signedIn() {
  return request.auth != null;
}

function isTeacher() {
  return signedIn()
    && request.auth.token.teacher == true;
}

match /submissions/{submissionId} {
  // 학생은 자신의 문서만, 교사는 허용된 문서를 읽음
  allow read: if resource.data.ownerUid == request.auth.uid
              || isTeacher();
}

06 · RISK PROFILE

두 방식에서 특히 점검해야 할 보안 위험

Email/Password 주요 위험

  • 추측하기 쉬운 비밀번호와 여러 서비스의 비밀번호 재사용
  • 학생·교사가 같은 계정을 공유하여 UID 분리가 무너짐
  • 공개 회원가입으로 외부인 또는 허위 계정 생성
  • 실제 이메일이 없어 비밀번호를 스스로 재설정할 수 없음
  • 학기 종료 후 계정을 비활성화하지 않음
  • 공용 PC에서 로그인 상태가 지속됨

Google 로그인 주요 위험

  • 개인 Google 계정까지 허용하여 허가되지 않은 사용자 가입
  • Google 로그인 성공만으로 교사 역할을 부여
  • 필요하지 않은 Drive·Calendar 등의 추가 scope 요청
  • 승인된 도메인과 리디렉션 설정 오류
  • 공용 PC에 Google 계정 자체가 로그인된 채 남음
  • 학교 계정이 비활성화되어도 앱 데이터·역할 정리가 누락됨

두 방식에 공통으로 생기는 위험

Rules가 공개됨

인증이 안전해도 allow read, write: if true라면 데이터는 보호되지 않습니다.

교사 역할을 클라이언트가 결정

localStorage, URL, HTML 클래스, 사용자가 쓸 수 있는 role 필드는 신뢰할 수 없습니다.

계정 수명주기 없음

학기 종료, 전학, 교사 변경, 계정 탈취 상황에서 비활성화와 권한 회수 절차가 필요합니다.

Google 로그인에서 scope를 구분하십시오. 기본 로그인에 필요한 신원 정보와 Google Drive 파일 읽기 권한은 다른 요청입니다. 로그인만 필요하다면 불필요한 Google API scope를 요청하지 않는 것이 좋습니다.

08 · MANUAL SECURITY CHECK

방식별 수동 보안 점검표

공통 필수 항목

  • 학생과 교사는 사용자별로 서로 다른 UID를 갖는다.
  • 교사 역할을 Custom Claims 또는 신뢰 가능한 UID로 판정한다.
  • 학생 A가 학생 B의 데이터를 읽거나 수정할 수 없다.
  • 교사 화면을 우회해도 Rules가 데이터 요청을 거부한다.
  • 학생이 교사 피드백과 역할 필드를 수정할 수 없다.
  • 공용 기기의 로그아웃과 인증 지속 방식을 확인했다.
  • App Check·사용량·비정상 요청을 관찰한다.
  • 학기 종료와 권한 회수 절차가 있다.

절대 피해야 할 항목

  • 공통 학생 계정 또는 공통 교사 계정을 공유
  • JavaScript에 정답 비밀번호를 하드코딩
  • Google 로그인 성공을 곧 교사 권한으로 간주
  • 이메일 문자열이나 화면 경로만으로 역할 판정
  • 테스트 모드의 공개 Firestore Rules 유지
  • 학생이 자신의 role을 생성·수정
  • 비밀 API 키·Admin 자격 증명을 GitHub Pages에 포함
  • 실제 학생·교사 계정으로 공격 테스트 없이 배포

Email/Password 추가 점검

  • Firebase 비밀번호 정책을 설정했다.
  • 공개 회원가입 또는 허위 계정 생성을 통제한다.
  • 비밀번호 재설정 방법이 작동한다.
  • 가상 이메일 계정의 분실 처리 담당자를 정했다.
  • 학기 종료 계정 비활성화를 시험했다.

Google 로그인 추가 점검

  • Firebase Auth의 승인된 도메인을 확인했다.
  • 개인 계정 허용 여부와 학교 계정 범위를 정했다.
  • 로그인에 불필요한 Google API scope를 요청하지 않는다.
  • 교사 역할을 Google 로그인과 별도로 판정한다.
  • 공용 PC에서 Google 계정 세션까지 확인한다.
09 · FINAL RECOMMENDATION

핵심 결론

Email/Password와 Google 로그인은 Firebase Authentication에 들어오는 서로 다른 입구입니다. 어느 입구를 선택하든 학생별 UID, 신뢰할 수 있는 교사 역할, 엄격한 Firestore Rules, 계정 종료와 공격 테스트가 갖춰져야 안전합니다.

Email/Password를 선택하면

OAuth 설정은 줄어들지만 비밀번호 정책·재설정·계정 발급·학기 종료를 직접 운영해야 합니다.

Google 로그인을 선택하면

비밀번호 운영은 학교 계정 체계를 활용할 수 있지만 승인 계정 범위·scope·교사 역할을 별도로 통제해야 합니다.

선택한 방식에 따라 결과에서 우선 개선 항목을 설명합니다.

1. 학생과 교사가 사용자별로 서로 다른 Firebase UID를 갖습니까?핵심공유 계정은 학생별 소유권과 책임 추적을 무너뜨립니다.
2. 비밀번호를 브라우저 JavaScript에서 직접 비교하지 않습니까?핵심Firebase 또는 신뢰할 수 있는 인증 서버가 자격 증명을 검사해야 합니다.
3. Firestore Rules가 학생의 자기 문서와 교사의 담당 범위를 구분합니까?핵심로그인 성공과 데이터 접근 권한은 별개입니다.
4. 교사 역할을 신뢰할 수 있는 UID·Custom Claims·관리자 역할 문서로 판정합니까?핵심URL, 이메일 문자열, 화면 변수와 localStorage는 신뢰할 수 없습니다.
5. 학생이 다른 학생 자료와 교사 피드백 필드를 변경하지 못하는지 시험했습니까?핵심UI가 아니라 직접 Firestore 요청으로 거부 여부를 확인해야 합니다.
6. Email/Password 사용 시 비밀번호 정책·재설정·계정 종료 절차가 있습니까?Google 로그인만 사용한다면 해당 없음으로 선택할 수 있습니다.
7. Google 로그인 사용 시 승인 계정 범위·도메인·요청 scope를 검토했습니까?Email/Password만 사용한다면 해당 없음으로 선택할 수 있습니다.
8. 공개 회원가입 또는 허가되지 않은 Google 계정의 가입을 통제합니까?로그인은 성공하지만 수업 구성원이 아닌 계정이 생길 수 있습니다.
9. 공용 PC에서 명확한 로그아웃과 세션 지속 방식을 확인했습니까?Google 계정 자체의 로그인 상태도 함께 고려합니다.
10. App Check·사용량·비정상 로그인과 비용을 관찰합니까?공개 웹 앱의 자동 요청과 남용을 줄이는 운영 항목입니다.
11. 학생·교사 계정의 학기 종료·전학·담당 변경 시 권한 회수 절차가 있습니까?인증 방식과 무관하게 계정 수명주기가 필요합니다.
12. 학교의 개인정보·계정 정책과 수집 최소화 원칙을 확인했습니까?중요기술적으로 안전해도 기관 정책과 맞지 않을 수 있습니다.
