00 · READ THIS FIRST

“개선된 맞춤형 인증 구조”를 우선하고, 적용 범위는 저위험 앱으로 제한합니다

직접 만든 접근 코드 인증은 Google 로그인이나 Firebase Authentication 같은 인증 전문 서비스보다 운영 책임이 큽니다. 그래도 학교 계정 사용이 어렵고 Apps Script를 유지해야 한다면, 데이터 민감도와 개발 역량에 따라 아래 세 단계 중 하나를 선택할 수 있습니다.

A · 맞춤형 권장

무작위 접근 코드 + HMAC-SHA-256

평문 코드를 저장하지 않고 salt·pepper·검증값, 역할 세션과 요청별 권한 검사를 사용합니다.

  • 학생별 무작위 코드
  • 교사 코드 별도·더 강하게
  • 인증 시트가 유출되어도 평문 코드 없음
B · 제한적 대안

저위험 최소 구조

서버의 비공개 저장소에 앱 전용 코드를 두되, 세션·역할 검사와 최소 데이터는 생략하지 않습니다.

  • 비민감·소규모·단기 활동
  • 개별 코드와 짧은 세션
  • AI 생성 코드의 보안 시험 필수
C · 전문 인증 필요

직접 만들지 말아야 할 범위

성적, 상담, 건강, 연락처, 장기 누적 기록, 다수 교사 운영은 Google 계정이나 Firebase Auth를 우선합니다.

HMAC을 사용해도 전문 인증 서비스와 같아지는 것은 아닙니다. HMAC은 저장된 접근 코드의 보호를 강화하지만, 사용자 복구·다중 인증·기기 관리·감사 기록을 자동으로 제공하지 않습니다. 민감한 학생 정보라면 맞춤형 코드 인증의 범위를 넘어섭니다.

01 · ARCHITECTURE

웹 화면은 공개되어도 Sheets는 비공개로 유지할 수 있습니다

02 · IMPROVED CUSTOM PROCESS

개선된 맞춤형 프로세스: 평문 코드를 저장하지 않습니다

이 구조에서는 교사가 학생별로 충분히 긴 무작위 접근 코드를 발급합니다. 서버는 원래 코드를 저장하지 않고, 학생별 salt와 서버 비밀 pepper를 사용해 HMAC-SHA-256 검증값을 만든 뒤 인증용 Sheet에 저장합니다.

개별 코드 발급학생마다 다른 앱 전용 무작위 코드를 교사가 개별 전달
검증값 등록서버에서 salt + 사용자 ID + 코드와 pepper로 HMAC 계산
코드 전송GET URL이 아니라 google.script.run 또는 POST 사용
서버 재계산입력 코드로 같은 HMAC을 만들고 저장된 검증값과 비교
역할 세션 발급student 또는 teacher, 대상 ID, 만료 시각을 서버에 저장
매 요청 재검사데이터를 읽고 쓰기 전에 세션·역할·소유권 확인
verifier = HMAC-SHA-256( key = AUTH_PEPPER_V1, message = "v1 | subjectId | salt | issuedAccessCode" ) 인증용 Sheet 저장: subjectId + salt + verifier + role + status + credentialVersion 저장하지 않음: 원래 접근 코드, pepper
salt · 사용자별 공개 난수

같은 코드를 가진 사용자가 있어도 검증값이 같아지지 않게 합니다. 검증값과 함께 인증용 Sheet에 저장해도 됩니다.

pepper · 앱 전체 비밀키

인증용 Sheet와 분리해 Script Properties에 둡니다. Sheet만 유출된 경우 대량 추측을 어렵게 만드는 추가 방어입니다.

HMAC-SHA-256 · 키가 있는 검증 함수

입력과 비밀키로 고정 길이 검증값을 만듭니다. 원래 코드를 복원하는 암호화가 아닙니다.

중요한 전제: HMAC-SHA-256은 빠른 함수입니다. 교사가 발급한 충분히 긴 무작위 코드에는 실용적인 방어가 될 수 있지만, 학생이 직접 만든 짧고 흔한 비밀번호를 저장하는 표준 방식은 아닙니다. 사용자 선택 비밀번호가 필요하면 Argon2id·scrypt·PBKDF2 같은 비밀번호 전용 KDF 또는 인증 전문 서비스를 사용해야 합니다.
개선 후에도 남는 위험이 있습니다. 브라우저 스크립트가 침해되면 세션 토큰이 탈취될 수 있고, 로그인 시도 제한은 타인이 계정을 잠그는 공격에 악용될 수 있습니다. 코드 초기화·복구·퇴직자 권한 회수는 직접 운영해야 하며, Apps Script의 할당량·감사 로그 한계와 맞춤형 인증 코드의 구현 오류도 남습니다. 따라서 이 구조는 비민감·소규모·단기 교육 활동에 한정합니다.
권장 구조의 HMAC 검증값 함수 예시
function createCodeVerifier_(subjectId, salt, accessCode) {
  const props = PropertiesService.getScriptProperties();
  const pepper = props.getProperty('AUTH_PEPPER_V1');

  if (!pepper) throw new Error('SERVER_CONFIG_ERROR');

  const code = String(accessCode || '').trim();
  if (code.length < 12 || code.length > 128) {
    throw new Error('INVALID_CREDENTIALS');
  }

  const message = ['v1', subjectId, salt, code].join('|');
  const bytes = Utilities.computeHmacSha256Signature(
    message,
    pepper,
    Utilities.Charset.UTF_8
  );

  return Utilities.base64EncodeWebSafe(bytes).replace(/=+$/, '');
}

이 코드는 핵심 형태를 보여 주는 예시입니다. 실제 앱에는 일정 시간 비교, 입력 길이 제한, 실패 횟수 제한, 오류 메시지 통일, pepper 버전 교체 절차가 추가되어야 합니다.

공개 함수가 반드시 가져야 하는 권한 검사 형태
function teacherLoadDashboard(sessionToken) {
  const session = requireTeacher_(sessionToken); // 가장 먼저
  return readTeacherDashboard_(session);         // 그 다음 Sheet 접근
}

function teacherSaveFeedback(sessionToken, feedbackData) {
  const session = requireTeacher_(sessionToken); // 가장 먼저
  validateFeedback_(feedbackData);
  return saveFeedbackToSheet_(session, feedbackData);
}

function studentSaveMyWork(sessionToken, workData) {
  const session = requireStudent_(sessionToken); // 가장 먼저
  validateWork_(workData);

  // 브라우저가 보낸 studentId가 아니라 세션의 subjectId 사용
  return saveWorkToSheet_(session.subjectId, workData);
}

function requireTeacher_(sessionToken) {
  const session = findValidSession_(sessionToken);
  if (!session) throw new Error('AUTH_REQUIRED');
  if (session.role !== 'teacher') throw new Error('FORBIDDEN');
  return session;
}

04 · MINIMUM VIABLE SECURITY

바이브 코딩 교사를 위한 최소 버전

암호학을 직접 설계하기 어렵다면, AI에게 구현을 맡기되 앱의 범위를 비민감·소규모·단기로 제한합니다. 최소 버전에서는 접근 코드를 비공개 서버 저장소에서 평문 비교할 수 있지만, 이것은 인증 저장소가 유출되면 모든 코드가 즉시 노출되는 약한 방식입니다.

최소 버전에서도 생략할 수 없는 것: 로그인 화면, POST 전송, 파일 분리만으로는 부족합니다. 서버가 세션을 발급하고, 모든 교사용·학생용 데이터 함수가 역할과 소유권을 먼저 확인해야 합니다. 이를 구현하거나 시험할 수 없다면 교사용 기능을 공개 배포에 넣지 말고 Google 계정 제한 배포로 분리하세요.
1 · 저장

서버에만 코드 보관

교사 앱 전용 코드는 Script Properties, 학생별 코드는 비공개 인증용 Sheet에 둡니다. HTML과 데이터용 Sheet에는 두지 않습니다.

2 · 접속

서버 로그인 후 세션

google.script.run으로 전송하고, 성공하면 짧은 수명의 임시 세션을 발급합니다. URL에는 코드와 세션을 넣지 않습니다.

3 · 권한

함수마다 서버 검사

학생·교사 데이터 함수는 가장 먼저 역할을 확인합니다. 화면 숨김과 클라이언트 변수는 보안 판단에 사용하지 않습니다.

05 · COPY & PASTE FOR AI

AI에게 전달할 구체적인 기술 요구사항

Google Apps Script와 Google Sheets로 학생 활동 웹 앱과 교사 대시보드를 만들어 주세요.

- 학생 수: [예: 30명]

[저장소]

복사용 프롬프트 2 · 개선된 맞춤형 구조로 승격최소 버전이 작동한 뒤 HMAC 구조를 요청할 때 사용
현재 Apps Script 접근 코드 인증을 다음의 개선된 맞춤형 구조로 보완해 주세요.

- 학생 및 교사 접근 코드를 평문으로 저장하지 마세요.
- 학생별로 안전하게 생성한 고유 salt를 사용하세요.
- AUTH_PEPPER_V1은 Script Properties에서 읽고 인증용 Sheet에는 저장하지 마세요.
- Apps Script의 Utilities.computeHmacSha256Signature()을 사용해 HMAC-SHA-256 검증값을 만드세요.
- 검증 메시지는 version, subjectId, salt, 정규화한 접근 코드가 모호하지 않게 결합되도록 설계하세요.
- 인증용 Users 탭에는 subjectId, salt, verifier, verifierVersion, role, status, credentialVersion만 저장하세요.
- HMAC 검증값은 가능한 경우 일정 시간 비교 함수로 비교하세요.
- 세션 원문 토큰 대신 세션 토큰의 검증값을 서버에 저장하세요.
- AUTH_PEPPER와 SESSION_SIGNING_KEY는 서로 다른 Script Property를 사용하세요.
- 로그인 실패 횟수 제한, 짧은 세션 만료, 로그아웃 취소, 코드 재발급 시 credentialVersion 증가와 기존 세션 무효화를 구현하세요.
- 모든 공개 학생·교사 함수의 첫 단계에서 서버 세션과 역할을 검사하는 기존 구조를 유지하세요.
- 원래 접근 코드, pepper, 세션 원문을 실행 로그에 남기지 마세요.
- HMAC-SHA-256이 빠른 함수이므로 이 구조는 교사가 발급한 충분히 긴 무작위 접근 코드에만 사용한다고 코드 주석과 설치 문서에 명시하세요.
- 사용자가 직접 만드는 일반 비밀번호라면 이 구현을 권장하지 말고 PBKDF2/Argon2id 또는 인증 전문 서비스를 안내하세요.

수정된 파일 전체, 데이터 마이그레이션 절차, Script Properties 키, 코드 재발급 절차, 기존 세션 폐기 테스트까지 제공하세요.

- 근거가 되는 함수명을 적으세요.
- FAIL인 항목은 위험을 한 문장으로 설명하고 코드를 수정하세요.
- 수정 뒤 동일한 18개 항목을 다시 평가하세요.
- 실제 Apps Script 배포 화면에서 교사가 직접 확인해야 할 항목은 별도 체크리스트로 정리하세요.

가장 중요한 실패

학생 토큰으로 교사 함수가 실행되거나, 토큰 없이 Sheet를 읽는 함수가 하나라도 있으면 배포하지 않습니다.

AI의 “확인 완료”만 믿지 않기

시크릿 모드, 학생 기기, 만료 토큰으로 실제 운영 /exec URL을 교사가 직접 시험합니다.

수정 후 다시 전체 시험

한 함수의 보안 수정을 다른 함수가 우회할 수 있으므로 18개 항목을 처음부터 다시 실행합니다.

FINAL RECOMMENDATION

초보자에게도 설명할 수 있어야 운영할 수 있습니다

개선된 맞춤형 구조는 평문 코드를 저장하지 않는 HMAC 검증, 서버 세션, 요청마다 역할 검사를 함께 사용합니다. 최소 구조는 HMAC 저장 보호를 잠시 단순화할 수 있지만, 세션과 서버 권한 검사는 단순화할 수 없습니다. 두 구조 모두 전문 인증 서비스와 동등하지 않으며 저위험·소규모 앱에만 적용합니다.

저장 보호

가능하면 salt + pepper + HMAC 검증값. 최소 버전의 평문 코드는 저위험·단기 앱에서만 제한적으로 사용합니다.

접근 통제

교사·학생 세션을 구분하고 모든 서버 함수가 Sheet 접근 전에 역할과 소유권을 확인합니다.

범위 통제

앱 전용 ID와 최소 데이터만 사용합니다. 민감하거나 장기적인 학생 기록은 인증 전문 서비스로 옮깁니다.
