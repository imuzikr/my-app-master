00 · HOW TO USE THIS INDEX

이 파일이 앞으로 전체 프로젝트의 기준점입니다

이 인덱스는 개별 문서의 내용을 대신하지 않습니다. 어떤 질문을 어느 문서에서 다루었는지 알려 주고, 이미 확정한 원칙을 반복해서 다시 논의하거나 서로 다른 문서가 충돌하는 일을 줄이는 역할을 합니다.

찾기

기술·배포·인증 주제로 필요한 문서를 검색하고 바로 엽니다.

연결하기

화면 구조 문서와 그 구조에 적용되는 인증 문서를 함께 확인합니다.

결정하기

확정된 원칙과 아직 결정하지 않은 선택을 구분합니다.

통합하기

나중에 하나의 종합 가이드로 합칠 때 사용할 목차를 미리 유지합니다.

업데이트 원칙: 새 HTML 문서를 만들면 문서 카드, 관련 문서, 결정 기록과 향후 통합 목차를 함께 갱신합니다. 기존 문서의 결론이 바뀌면 새 복사본을 늘리기보다 기존 문서와 이 인덱스를 같은 시점에 수정합니다.

05 · CONFIRMED PRINCIPLES

지금까지 모든 문서에서 일관되게 확정한 원칙

화면 분리는 보안 경계가 아니다

별도 URL, HTML 파일, 숨김 화면과 ?page=teacher는 사용자 경험과 코드 구조일 뿐 교사 인증을 만들지 않습니다.

브라우저는 신뢰하지 않는다

클라이언트가 보낸 역할, 이메일, 학번, UID, 문서 ID와 행 번호는 서버 또는 Security Rules가 검증해야 합니다.

공동 비밀번호는 개인 인증이 아니다

유출자를 찾거나 한 사용자만 차단할 수 없으므로, 민감한 학생 데이터와 교사 기능에는 개별 계정을 사용합니다.

로그인·인증·OAuth·역할 권한은 다르다

계정을 확인한 뒤에도 우리 앱의 학생·교사 역할과 데이터 접근 범위를 별도로 결정해야 합니다.

데이터 계층이 최종 권한을 검사한다

Firestore에서는 Security Rules, Apps Script에서는 각 서버 함수가 최종적으로 읽기·쓰기 권한을 확인합니다.

식별 실패는 접근 거부가 기본이다

사용자 이메일이나 UID를 확인하지 못했을 때 추측하거나 기본 역할을 부여하지 않고 안전하게 실패시킵니다.

교사 권한은 개별적으로 회수할 수 있어야 한다

교사 허용 목록, 커스텀 클레임 또는 서버 역할 명단으로 특정 계정의 권한을 변경하고 기록할 수 있어야 합니다.

최소 권한과 최소 데이터를 유지한다

필요한 OAuth 범위, Firestore 컬렉션, Sheet 열, 로그와 보존 기간만 사용합니다.

최소 버전도 서버 권한 검사는 생략하지 않는다

HMAC 저장 보호를 단순화하는 저위험 대안에서도 세션, 역할, 학생 데이터 소유권은 모든 서버 요청에서 확인합니다.

JWT Payload는 암호화된 비밀 저장소가 아니다

Firebase ID Token의 내용은 사용자가 읽을 수 있습니다. 서버는 서명·발급자·대상·만료를 검증하고 Payload에는 비밀번호·API 키·학생 답안을 넣지 않습니다.

Custom Claims는 작고 안정적인 역할 정보만 담는다

권한 있는 서버만 설정하며, 프로필·학급 목록·답안 같은 일반 데이터는 Firestore 문서에 저장합니다.

App Check는 인증과 Rules를 보완한다

정상 앱에서 온 요청인지 추가로 확인하지만 사용자가 누구인지 또는 어떤 데이터를 읽을 수 있는지는 Auth와 Rules가 판단합니다.

07 · SHARED GLOSSARY

앞으로 모든 문서에서 동일하게 사용할 용어

로그인 Login

사용자가 계정 정보를 입력하거나 제공자를 선택해 인증 절차를 시작·완료하는 사용자 경험입니다.

인증 Authentication

사용자가 누구인지 신뢰할 수 있는 방식으로 확인하는 과정입니다.

역할 권한 App Authorization

인증된 사용자가 학생·교사·관리자 중 누구이며 어떤 기능과 데이터에 접근할 수 있는지 결정합니다.

OAuth

사용자 또는 앱이 Google 같은 서비스의 자원에 어느 범위까지 접근할 수 있는지 위임하는 표준입니다.

배포 Deployment

작성한 코드를 실제 사용자가 접근할 수 있는 URL과 실행 환경으로 공개하는 단위입니다.

화면 View / Page

학생 활동 또는 교사 대시보드처럼 사용자가 보는 UI 단위이며, 배포나 보안 경계와 동일하지 않습니다.

Firestore Security Rules

인증 사용자와 데이터 조건에 따라 Firestore 읽기·쓰기를 허용하거나 거부하는 서버 측 규칙입니다.

Apps Script 서버 함수

google.script.run 등으로 호출되는 서버 코드입니다. 호출될 때마다 사용자·역할·입력을 검사해야 합니다.

HMAC-SHA-256

입력과 서버 비밀키로 검증값을 만드는 키 기반 함수입니다. 교사가 발급한 무작위 접근 코드의 저장 보호에 활용할 수 있습니다.

salt와 pepper

salt는 사용자별로 다른 공개 난수이고, pepper는 인증 데이터와 분리해 보관하는 앱 전체 비밀키입니다.

서버 세션

로그인 뒤 서버가 사용자 ID·역할·만료를 기억하는 임시 상태입니다. 이후 모든 데이터 요청의 권한 판단에 사용합니다.

JWT · Firebase ID Token

Header, Payload, Signature로 구성된 서명된 토큰입니다. Firebase 로그인 뒤 UID를 전달하며 서버는 서명·발급자·대상·만료를 검증합니다. Payload는 암호화가 아닙니다.

Custom Claims

권한 있는 서버가 Firebase 사용자 토큰에 넣는 작은 역할 정보입니다. 교사·관리자처럼 접근 제어에 필요한 값만 사용합니다.

App Check

요청이 등록된 정상 앱 환경에서 왔는지 추가로 확인하는 남용 방어 계층입니다. 사용자 인증과 데이터 권한 검사를 대신하지 않습니다.

Callable Function

Firebase Functions에서 onCall()로 만든 서버 함수입니다. 클라이언트의 httpsCallable() 호출과 연결되어 인증 토큰·App Check·오류 형식을 다루기 쉽게 해 줍니다.
