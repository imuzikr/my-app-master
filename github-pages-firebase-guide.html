Architecture & Security Guide

학생용·교사용 GitHub Pages를 따로 배포하고 Firestore를 함께 사용하는 방법

두 개의 정적 웹사이트가 하나의 Firebase 프로젝트를 이용하는 구조를 기능성, 작업 난도, 운영 편의, 보안 관점에서 정리한 실무 가이드입니다.

01 · SUMMARY

한눈에 보는 종합 평가

학생용 사이트와 교사용 사이트를 각각 GitHub Pages에 배포하고, 두 사이트에서 같은 Firestore를 사용하는 방식입니다. 전통적인 서버를 직접 운영하지 않아도 되지만, 브라우저가 데이터베이스에 직접 접근하기 때문에 Firestore Security Rules가 사실상 서버의 출입 통제 역할을 합니다.

구현 가능성높음정적 웹에서도 Firebase SDK 사용 가능
초기 작업 난도중간Auth·Rules·데이터 구조 학습 필요
기능 확장성매우 높음실시간 현황·피드백·파일 기능 확장
보안 잠재력높음단, Rules를 정확히 설계했을 때
핵심 원칙: 학생용 주소와 교사용 주소가 다르다고 해서 권한이 분리되는 것은 아닙니다. URL과 화면은 공개될 수 있다고 가정하고, 실제 데이터 접근 권한은 Firebase Authentication과 Firestore Security Rules에서 차단해야 합니다.
02 · ARCHITECTURE

전체 구조와 주요 특징

GitHub Pages는 HTML·CSS·JavaScript를 전달하고, Firebase Authentication은 사용자가 누구인지 확인하며, Firestore는 데이터를 저장합니다. Security Rules는 확인된 사용자에게 어느 데이터까지 허용할지를 판단합니다.

교사용 GitHub Pages 현황·조회·피드백
예: teacher-app.github.io

GitHub Pages화면 파일을 인터넷에 공개하고 HTTPS로 제공학생·교사 권한을 안전하게 판정하지 않음

Firestore문서 형태로 데이터를 저장하고 실시간 전달데이터 구조를 대신 설계하거나 권한을 자동 완성하지 않음
Security Rules요청자의 UID·역할·데이터를 기준으로 읽기와 쓰기를 허용·거부화면을 숨기거나 교사용 URL 접속 자체를 막지는 않음
App Check내 웹 앱에서 온 요청인지 확인하여 자동화된 남용을 줄임사용자의 신원 확인이나 교사 권한 판정을 대체하지 않음

두 페이지를 따로 배포할 때의 특징

  • 학생 화면과 교사 화면의 코드, 메뉴, 배포 일정을 완전히 분리할 수 있습니다.
  • 두 사이트가 동일한 Firebase 프로젝트 설정과 Firestore 데이터를 공유할 수 있습니다.
  • 두 사이트의 Firebase 설정값은 브라우저에서 공개됩니다. 이는 정상적인 웹 SDK 구조입니다.
  • 두 주소를 Firebase Authentication의 승인된 도메인에 등록해야 할 수 있습니다.
  • 브라우저가 Firestore와 직접 통신하므로 일반적인 자체 REST 서버를 별도로 만들 필요는 없습니다.
03 · PROS & CONS

장점과 단점

장점

  • 비용 부담이 낮습니다. 소규모 수업은 무료 할당량 안에서 운영할 가능성이 높습니다.
  • 서버 운영이 단순합니다. 서버 설치·패치·포트 관리가 필요하지 않습니다.
  • 실시간 기능이 좋습니다. 학생이 제출하면 교사 대시보드에 바로 반영할 수 있습니다.
  • 화면 책임을 분리할 수 있습니다. 학생 사이트 변경이 교사 대시보드 UI에 직접 영향을 덜 줍니다.
  • 확장이 쉽습니다. 로그인, 파일 업로드, 알림, 통계, AI 피드백 등을 단계적으로 붙일 수 있습니다.
  • GitHub 사용 경험을 살릴 수 있습니다. push 후 Pages에 자동 반영되는 익숙한 배포 방식을 유지할 수 있습니다.

단점

  • Rules 학습이 필요합니다. 화면 코드만 완성해서는 안전한 앱이 되지 않습니다.
  • 두 배포본을 관리해야 합니다. 공통 Firebase 설정, 자료형, 필드명이 서로 어긋날 수 있습니다.
  • 교사 역할 부여가 까다롭습니다. 클라이언트가 스스로 teacher 값을 쓰게 하면 권한 상승 위험이 있습니다.
  • 관리자 작업에는 서버가 필요할 수 있습니다. 일괄 계정 관리, Custom Claims, 비밀 API 호출은 신뢰할 수 있는 서버 환경이 적합합니다.
  • 쿼리와 Rules를 함께 설계해야 합니다. 허용된 문서만 반환되도록 쿼리 조건도 맞춰야 합니다.
  • 과금과 남용을 감시해야 합니다. 공개 사이트는 자동 요청과 불필요한 반복 읽기에 노출될 수 있습니다.

자주 생기는 오해

오해실제 의미필요한 조치
교사용 URL을 알려주지 않으면 안전하다URL은 추측·공유·검색·브라우저 기록 등으로 알려질 수 있음교사 데이터 읽기를 Rules에서 거부
교사 화면을 hidden으로 감추면 된다개발자 도구로 표시하거나 함수를 직접 호출할 수 있음UI 확인과 별도로 서버 측 권한 규칙 적용
Firebase API 키가 보이면 해킹이다웹 설정값은 원래 클라이언트에 포함됨키 은닉보다 Auth·Rules·App Check·할당량 관리
로그인 사용자면 모두 안전하다로그인한 학생도 다른 학생 자료를 요청할 수 있음UID와 문서 소유자를 Rules에서 비교
문서의 role: teacher만 확인하면 된다학생이 해당 역할 문서를 생성·수정할 수 있다면 권한 상승 가능역할 데이터의 쓰기를 관리자만 가능하게 제한
App Check가 있으면 Auth는 없어도 된다App Check는 앱 출처, Auth는 사용자 신원을 확인함민감한 학생 데이터에는 두 기능의 역할을 구분
04 · SECURITY

권장 보안 설계

보안을 한 가지 장치에 맡기지 말고 여러 층으로 구성합니다. 가장 중요한 두 층은 Authentication과 Security Rules이며, App Check와 입력 검증은 이를 보완합니다.

Authentication — 사용자 확인

학교 Google 계정, 이메일 로그인 등으로 학생과 교사에게 고유 UID를 발급합니다. 실제 학생 식별이 필요하다면 익명 로그인은 적합하지 않습니다.

Authorization — 권한 확인

학생은 자기 제출물만, 교사는 담당 학급의 제출물만 읽도록 Security Rules에서 제한합니다.

Data validation — 입력값 검증

허용 필드, 자료형, 문자열 길이, 소유자 UID, 수정 가능한 필드를 Rules에서 확인합니다.

App Check — 앱 출처 확인

내 사이트에서 온 정상 요청인지 확인하여 단순 스크립트와 봇의 남용을 줄입니다. 이것만으로 완전한 방어가 되지는 않습니다.

Monitoring — 사용량 관찰

Firestore 읽기·쓰기 사용량, 오류, 예산 알림을 점검하고 불필요한 실시간 리스너와 반복 쿼리를 줄입니다.

Privacy — 최소 수집과 보존

학습에 필요한 정보만 저장하고 보존 기간, 삭제 방법, 백업 범위, 접근 교사를 정합니다. 민감한 개인정보는 가급적 저장하지 않습니다.

학생 로그인 방식 선택

방식장점주의점적합한 상황
학교 Google 계정UID가 안정적이고 학생이 비밀번호를 새로 만들 필요가 적음개인 Google 계정 접근 허용 여부와 학교 정책 확인 필요학교가 Google Workspace를 사용하는 경우
이메일·비밀번호일반적인 로그인 방식이며 Firebase가 계정을 관리가입·초기 비밀번호·재설정·퇴학/졸업 후 계정 관리 필요별도의 서비스 계정 체계를 운영할 때
익명 로그인학생이 즉시 시작할 수 있어 가장 간단함기기·브라우저 변경 시 동일 학생을 안정적으로 식별하기 어려움개인 식별이 필요 없는 일회성 활동
공통 비밀번호구현이 쉬워 보임공유·노출되기 쉽고 학생별 권한 분리가 불가능민감한 제출 데이터에는 권장하지 않음
교사 역할은 신뢰할 수 있는 위치에서 부여해야 합니다. 소규모·교사 1명 앱에서는 Rules에 교사 UID를 지정하는 방식이 단순합니다. 여러 교사가 사용하는 서비스라면 Admin SDK로 Custom Claims를 부여하거나, 관리자만 쓸 수 있는 역할 문서를 사용해야 합니다.
05 · RULES EXAMPLE

Firestore 구조와 Rules 예시

아래는 원리를 이해하기 위한 최소 예시입니다. 실제 앱에서는 학급 참여 관계, 허용 필드, 값의 길이와 형식, 제출 기한 등을 추가해야 합니다.

단순한 데이터 구조

submissions/{studentUid}
  studentUid       // 제출 학생 UID
  classId          // 학급 ID
  activityId       // 활동 ID
  answer           // 학생 응답
  createdAt        // 생성 시각
  updatedAt        // 수정 시각
  teacherFeedback  // 교사 피드백

교사 1명·소규모 수업용 최소 개념 예시

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() {
      return request.auth != null;
    }

    function isOwner(studentUid) {
      return signedIn() && request.auth.uid == studentUid;
    }

    function isTeacher() {
      return signedIn()
        && request.auth.uid == "여기에_교사_UID";
    }

    match /submissions/{studentUid} {
      // 학생은 자기 문서만 읽을 수 있고, 교사는 전체를 읽을 수 있음
      allow read: if isOwner(studentUid) || isTeacher();

      // 학생은 자기 UID로 된 문서만 만들 수 있음
      allow create: if isOwner(studentUid)
        && request.resource.data.studentUid == request.auth.uid;

      // 실제 앱에서는 학생이 teacherFeedback을 바꾸지 못하도록
      // 변경된 필드를 추가로 검사해야 함
      allow update: if isOwner(studentUid) || isTeacher();

      // 학생 삭제 허용 여부는 수업 정책에 맞게 결정
      allow delete: if isTeacher();
    }
  }
}
이 예시는 완성형 Rules가 아닙니다. 위의 단순한 allow update만 사용하면 학생이 자기 문서의 teacherFeedback까지 수정할 수 있습니다. 실제 배포본에서는 학생이 변경 가능한 필드를 제한해야 합니다.

학생의 수정 가능 필드를 제한하는 원리

allow update: if isOwner(studentUid)
  && request.resource.data.diff(resource.data)
       .affectedKeys()
       .hasOnly(["answer", "updatedAt"]);

// 교사는 별도의 조건으로 피드백 수정 허용
allow update: if isTeacher();

교사용 전체 조회에서 기억할 점

Firestore Rules는 결과를 받아온 뒤 금지 문서를 제거하는 필터가 아닙니다. 쿼리 자체가 Rules가 허용하는 범위와 일치해야 합니다. 여러 교사가 담당 학급만 조회하게 만들려면 문서의 classId, 교사의 학급 소속 정보, 교사용 쿼리 조건을 함께 설계해야 합니다.

06 · DEVELOPMENT & OPERATIONS

작업 난도와 운영상의 주의점

예상 작업 영역

학생 사이트

로그인, 활동 입력, 자기 제출 조회·수정, 제출 완료 표시, 오류 처리

교사 사이트

교사 로그인, 학급별 조회, 실시간 제출 현황, 피드백 작성, 검색·필터

Firebase

Auth 제공자 설정, 승인 도메인, 데이터 구조, Rules, 인덱스, App Check, 사용량 점검

두 GitHub Pages를 따로 운영할 때

  • 각 저장소에서 push하면 해당 GitHub Pages는 별도의 사용자 작성 Actions 없이도 기본 배포 방식으로 반영할 수 있습니다.
  • 학생 사이트와 교사 사이트가 사용하는 Firebase 프로젝트 ID, 컬렉션명, 필드명을 맞춰야 합니다.
  • 공통 Firebase 초기화 코드와 데이터 형식을 복사해 두면 한쪽만 수정되는 문제가 생길 수 있습니다. 공통 규격을 문서화하거나 공유 모듈·패키지를 고려합니다.
  • 두 사이트 모두 로그인한다면 두 GitHub Pages 도메인을 Firebase Authentication의 승인된 도메인 설정에서 확인합니다.
  • React·Vite처럼 빌드가 필요한 앱은 배포용 파일 생성 과정이 필요할 수 있지만, 순수 HTML·CSS·JavaScript는 그대로 배포할 수 있습니다.

비용과 성능

  • 교사 대시보드에서 전체 데이터를 매번 다시 읽지 말고 학급·활동·기간 조건으로 조회합니다.
  • 실시간 리스너는 화면을 떠날 때 해제하고, 대시보드에 필요한 범위에만 연결합니다.
  • 목록은 페이지 단위로 불러오고, 문서에 너무 큰 본문이나 Base64 이미지를 직접 저장하지 않습니다.
  • 파일은 Cloud Storage에 저장하고 Firestore에는 경로와 메타데이터만 저장하는 편이 적합합니다.
  • 사용량과 결제 설정을 정기적으로 확인하고, 필요한 경우 예산 알림을 설정합니다.

권장 구축 순서

  1. 수집 데이터와 권한 표 작성
    학생이 읽고 쓸 필드, 교사만 볼 필드, 보존 기간을 먼저 정합니다.
  2. Authentication 적용
    학생과 교사가 각자 고유 UID로 로그인하게 합니다.
  3. Firestore 데이터 구조 설계
    학급·활동·학생 UID를 기준으로 문서 경로와 필드를 정합니다.
  4. Rules를 먼저 작성하고 테스트
    학생 A, 학생 B, 교사, 비로그인 상태의 허용·거부 사례를 모두 시험합니다.
  5. 학생 페이지 연결
    자기 데이터만 생성·조회·수정되는지 확인합니다.
  6. 교사 대시보드 연결
    권한이 있는 계정만 필요한 범위를 조회하게 합니다.
  7. App Check와 모니터링 추가
    기본 권한 설계가 안정된 뒤 남용 방지와 사용량 관찰을 강화합니다.
07 · CHECKLIST

배포 전 보안 점검표

반드시 확인

  • Firestore가 테스트 모드의 공개 Rules로 남아 있지 않다.
  • 모든 민감한 읽기·쓰기에 request.auth != null 검사가 있다.
  • 학생 A가 학생 B의 문서를 읽거나 수정할 수 없다.
  • 학생이 자신의 역할을 교사로 변경할 수 없다.
  • 학생이 교사 피드백·성적·관리 필드를 수정할 수 없다.
  • 교사 화면을 우회해 직접 요청해도 Rules가 차단한다.
  • 두 GitHub Pages 주소의 로그인 설정을 확인했다.
  • Rules Playground 또는 Emulator로 허용·거부 사례를 시험했다.
  • 불필요한 개인정보를 저장하지 않는다.
  • 사용량과 비용을 관찰할 방법이 있다.

피해야 할 구성

  • allow read, write: if true; 상태로 실제 학생 데이터를 저장
  • 교사용 JavaScript에 공통 비밀번호를 하드코딩
  • HTML의 hidden 클래스만으로 대시보드를 보호
  • 로그인한 모든 사용자에게 전체 컬렉션 읽기 허용
  • 학생이 쓸 수 있는 문서의 role을 권한 판정에 그대로 사용
  • AI·메일·외부 서비스의 비밀 API 키를 GitHub Pages JavaScript에 포함
  • 교사 Admin 기능에 일반 Firebase 웹 SDK만 사용하면서 관리자라고 가정
  • Rules 변경 후 학생·교사 계정으로 실제 동작을 재검증하지 않음
가장 중요한 공격 테스트: 학생 계정으로 교사용 URL을 열고, 개발자 도구에서 교사용 JavaScript를 직접 실행하거나 Firestore를 직접 요청해 보십시오. 화면이 보이느냐보다 데이터 요청이 permission-denied로 거부되는지가 핵심입니다.
08 · RECOMMENDATION

최종 판단

학생 활동 기록, 제출 현황, 교사 피드백, 실시간 대시보드가 필요하다면 이 구조는 기능성과 비용의 균형이 좋습니다. 다만 두 개의 GitHub Pages는 화면과 개발 작업을 분리할 뿐이며, 보안은 반드시 Firebase 쪽에서 완성해야 합니다.

추천하는 경우

  • 학생별 로그인과 데이터 분리가 필요함
  • 교사용 실시간 제출 현황이 필요함
  • 향후 피드백·파일·통계 기능을 확장할 계획임
  • Rules를 학습하고 계정별 접근 테스트를 수행할 수 있음

다른 방법이 더 쉬운 경우

  • 단순 설문·한 번 제출만 필요함 → Google Forms + Sheets
  • 교사가 시트 중심으로 관리하고 실시간성이 낮음 → Apps Script + Sheets
  • 학생 식별이 필요 없는 짧은 활동임 → 개인정보 없는 간단한 정적 앱

권장 최소 구성: 학교 계정 로그인 + 학생 UID 기반 문서 + 교사 UID 또는 안전한 역할 부여 + 엄격한 Firestore Rules + 배포 전 권한 테스트

OFFICIAL REFERENCES

공식 참고 자료

  • Firebase Security Rules 기본 안내
  • Firebase Authentication 안내
  • Cloud Firestore Security Rules 시작하기
  • 웹용 Firebase App Check 안내
  • Cloud Firestore 사용량과 한도

이 문서는 기술 설계를 위한 일반 가이드입니다. 학생 개인정보 처리와 계정 사용은 학교·교육청의 관련 지침과 내부 정책을 함께 확인해야 합니다.

1. 학생의 제출 내용이 교사 대시보드에 거의 실시간으로 나타나야 하나요?새로고침 없는 현황판과 즉시 피드백이 필요한지 확인합니다.
2. 학생이 로그인하고 자기 자료만 조회·수정해야 하나요?학생별 계정과 데이터 소유권 분리가 필요한지 확인합니다.
3. 제출 후 수정, 교사 피드백, 진행 상태와 같은 상호작용이 많은가요?단순 제출을 넘어 지속적으로 데이터를 주고받는지 확인합니다.
4. 학생 사이트와 교사 사이트를 반드시 독립적으로 배포·수정해야 하나요?단순히 주소만 나누는 것이 아니라 저장소와 배포 주기를 분리할 실질적인 이유가 있는지 묻습니다.
5. 교사가 Google Sheets에서 자료를 직접 확인·수정하는 것이 중요한가요?시트가 단순 내보내기 대상이 아니라 주된 관리 화면인지 확인합니다.
6. 활동이 한 번 제출하고 끝나는 단순 설문에 가까운가요?복잡한 웹 앱 없이 양식만으로 해결할 수 있는지 확인합니다.
7. Authentication과 Firestore Rules를 작성하고 계정별로 테스트할 수 있나요?Firebase 방식의 핵심 보안 작업을 맡을 시간과 학습 여건이 있는지 확인합니다.
8. 향후 파일 업로드, 통계, 여러 학급, 여러 교사 등으로 확장할 가능성이 큰가요?현재의 단순함보다 장기적인 기능 확장성이 중요한지 확인합니다.
9. Gemini·OpenAI·메일 등 비밀 API 키가 필요한 기능이 있나요?정적 GitHub Pages에 비밀 키를 넣을 수 없으므로 서버 측 기능이 추가로 필요합니다.
10. 성적, 상담, 건강정보 등 민감도가 높은 학생 정보를 다루나요?그렇다면 어떤 도구를 선택하든 학교 정책, 최소 수집, 접근 기록과 보존 기간을 별도로 검토해야 합니다.
11. 두 저장소의 공통 코드와 데이터 규격을 계속 함께 관리할 수 있나요?두 페이지 방식에서 한쪽만 수정되어 오류가 생기는 운영 위험을 감당할 수 있는지 확인합니다.
