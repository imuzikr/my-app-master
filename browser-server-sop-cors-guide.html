02 · THREE MEANINGS OF FUNCTION

‘함수’라는 같은 단어를 세 가지로 구분하기

혼란의 가장 큰 원인은 JavaScript 함수, Firebase 클라이언트 SDK 함수, Firebase Functions라는 제품 이름이 모두 ‘함수’라는 단어를 사용한다는 점입니다.

브라우저 클라이언트 SDK 함수

signInWithPopup(), getDoc(), setDoc(), httpsCallable()처럼 브라우저에서 Firebase 서비스에 요청하는 JavaScript 함수입니다.

Firebase 서버 Callable Function

onCall()로 정의하여 배포한 서버 함수입니다. 교사 역할 확인, Gemini 호출, 초안 저장 같은 실제 업무를 수행합니다.

서버 내부 비공개 보조 함수

verifyTeacher(), callGemini()처럼 Callable Function 내부에서만 사용하는 일반 서버 함수입니다. 브라우저가 직접 호출할 수 없습니다.

브라우저가 서버 코드를 내려받아 실행하는 것이 아닙니다. 함수 이름과 요청 데이터를 보내고 서버에서 실행한 결과만 돌려받습니다.

경로만 다르며 프로토콜·호스트·포트가 같습니다.

https://school.github.io/app/teacher/ https://region-project.cloudfunctions.net/function

호스트가 다르므로 교차 출처 통신입니다.

한 GitHub Pages의 /student와 /teacher는 같은 출처입니다. 학생이 교사용 JavaScript 파일 경로를 알아낼 수 있으므로 화면 숨김이 아니라 Firebase 서버의 권한 검사가 필요합니다.
04 · SAME-ORIGIN POLICY

SOP: 브라우저가 기본적으로 세우는 출처 경계

SOP, 즉 동일 출처 정책은 한 출처의 JavaScript가 다른 출처의 문서와 응답을 마음대로 읽지 못하도록 브라우저가 기본적으로 제한하는 정책입니다.

school.github.io의 JavaScript 현재 페이지의 DOM과 같은 출처 응답은 정상적으로 접근
SOP 기본 제한
다른 출처의 응답 서버가 명시적으로 허용하지 않으면 JavaScript가 응답을 읽지 못함
WHY

왜 필요한가요?

사용자가 악성 사이트를 열었을 때 그 사이트의 JavaScript가 사용자의 다른 로그인 서비스 응답을 마음대로 읽는 위험을 줄입니다.

LIMIT

어디까지 적용되나요?

브라우저가 집행하는 정책입니다. 서버 프로그램, 명령줄 도구, API 테스트 도구가 똑같이 따르는 서버 방화벽은 아닙니다.

다른 출처로 요청을 전혀 보낼 수 없다는 뜻인가요?
아닙니다. SOP는 주로 다른 출처의 응답을 JavaScript가 읽고 활용하는 것을 제한합니다. 요청 종류에 따라 전송 자체가 일어날 수도 있습니다. 그래서 서버는 SOP만 믿지 말고 인증·권한·CSRF 등 필요한 방어를 별도로 갖춰야 합니다.
05 · CROSS-ORIGIN RESOURCE SHARING

CORS: 서버가 브라우저에 허용 범위를 알려 주는 규칙

CORS는 SOP의 기본 제한을 필요한 범위에서 완화합니다. 서버가 어떤 출처·방식·헤더의 브라우저 요청을 허용할지 응답 헤더로 알려 주고, 브라우저가 그 정책을 집행합니다.

GitHub Pages 브라우저 “이 출처에서 Firebase 서버의 응답을 읽고 싶습니다.”
CORS
허용 정책 확인
Firebase Callable 인프라 허용 출처·메서드·헤더를 브라우저에 응답
Origin어느 웹 출처인가?

예: https://school.github.io

Method어떤 요청 방식인가?

예: Callable의 실제 요청은 POST

Headers어떤 헤더를 보내는가?

예: Content-Type, Authorization, App Check 관련 헤더

CORS는 브라우저와 서버의 협력입니다.
브라우저가 필요 여부를 판단하고 정책을 집행하며, 서버는 무엇을 허용할지 응답합니다. 어느 한쪽만으로 완성되는 규칙이 아닙니다.
06 · OPTIONS PREFLIGHT

OPTIONS 사전 요청은 누가 보내고 누가 답하나요?

개발자가 일반적으로 OPTIONS 요청 코드를 직접 작성하지 않습니다. SDK가 준비한 실제 요청을 브라우저가 살펴보고, 교차 출처이면서 단순 요청 조건을 벗어나면 브라우저가 자동으로 OPTIONS 사전 요청을 보냅니다.

브라우저
판단다른 출처인가?
판단JSON·Authorization 헤더를 쓰는가?
자동 동작필요하면 OPTIONS 전송
OPTIONS 질문
CORS 응답
Callable 인프라
확인출처·방식·헤더 조건
자동 동작허용 CORS 응답
다음 단계허용되면 실제 POST 대기

사전 요청의 개념적인 질문

OPTIONS /generateFeedbackDraft
Origin: https://school.github.io
Access-Control-Request-Method: POST
Access-Control-Request-Headers: authorization, content-type, x-firebase-appcheck

서버의 개념적인 답변

Access-Control-Allow-Origin: https://school.github.io
Access-Control-Allow-Methods: POST
Access-Control-Allow-Headers: authorization, content-type, x-firebase-appcheck

판단·전송 주체

브라우저가 필요 여부를 판단하고 OPTIONS를 자동 전송합니다. Firebase SDK가 직접 OPTIONS 코드를 작성하는 것은 아닙니다.

응답 주체

Firebase Callable 인프라가 onCall() 함수 본문 앞에서 사전 요청을 자동으로 처리합니다.

업무 실행 여부

OPTIONS만으로 교사 검사·Gemini 호출·Firestore 저장은 실행되지 않습니다. 실제 POST가 도착한 뒤 수행됩니다.

OPTIONS는 요청의 정당성을 심사하지 않습니다.
“이 출처와 방식으로 브라우저 통신을 허용하는가?”만 확인합니다. 실제 사용자가 교사인지 판단하는 단계는 뒤의 실제 Callable 요청입니다.
07 · CALLABLE END TO END

교사가 AI 초안을 요청할 때의 전체 순서

SOP·CORS는 서버 함수 실행 전 통신 단계에 있고, Authentication과 역할 검사는 실제 요청 단계에 있습니다.

1SDK 호출교사 화면이 httpsCallable() 실행
2브라우저 판단교차 출처와 요청 헤더 확인
3OPTIONS필요하면 사전 요청과 CORS 응답
4실제 POST요청 데이터와, 로그인 상태라면 Firebase ID Token 전달
5전달된 토큰 검증Callable 인프라가 형식과 포함된 Auth·App Check Token 확인
6서버 함수교사 역할·담당 범위 확인 후 Gemini 호출
7결과 반환초안 저장 후 성공·오류를 SDK에 전달

Callable 서버 내부에서도 입구와 보조 함수를 나눕니다

ENTRY

브라우저가 호출하는 공개 입구

generateFeedbackDraft처럼 onCall()로 내보낸 함수입니다. 요청을 받아 검사를 통과시킨 뒤 내부 함수를 순서대로 실행합니다.

INNER

서버 내부에서만 쓰는 보조 함수

verifyTeacher(), readSubmission(), callGemini(), saveDraft()는 브라우저가 직접 호출할 수 없는 내부 코드입니다.

Callable은 로그인을 자동으로 필수화하지 않습니다.
Authorization과 App Check 헤더는 Callable 프로토콜에서 선택 항목입니다. 전달된 토큰이 잘못되면 인프라가 거부하지만, 토큰이 없으면 핸들러의 request.auth 또는 request.app이 비어 있을 수 있습니다. 교사 전용 함수는 가장 먼저 request.auth의 존재를 확인하고, 이어서 교사 역할과 담당 학급을 검사해야 합니다. App Check도 enforceAppCheck: true를 설정해야 누락된 토큰을 실제로 차단합니다. 
export const generateFeedbackDraft = onCall(
  { enforceAppCheck: true },
  async (request) => {
    if (!request.auth) {
      throw new HttpsError("unauthenticated", "로그인이 필요합니다.");
    }

    await verifyTeacherScope(request.auth.uid, request.data.submissionId);
    // 권한 확인 뒤에만 제출물 조회와 외부 AI 호출을 수행합니다.
  }
);

08 · DIRECT SERVICE VS CUSTOM SERVER

왜 Firestore에는 바로 요청하고 Gemini는 서버를 거치나요?

Firebase 서비스와 통신한다고 모두 Callable Function이 필요한 것은 아닙니다. Firebase가 이미 제공하는 백엔드 서비스는 전용 SDK와 Security Rules로 직접 요청할 수 있고, 개발자가 만든 비밀 서버 작업만 Functions를 거칩니다.

기능브라우저의 함수요청 대상주요 서버 보호
Google 로그인signInWithPopup()Firebase AuthenticationFirebase의 인증 서비스
학생 제출물 저장setDoc()Cloud FirestoreAuthentication + Firestore Rules
학생 피드백 조회getDoc()Cloud Firestore소유자 UID를 검사하는 Rules
AI 초안 생성httpsCallable()Callable FunctionAuth 검증 + 교사 역할 검사
Gemini 실제 호출브라우저에서 하지 않음서버 내부 보조 함수Secret Manager + 호출 제한

Firebase 기본 서비스

Authentication과 Firestore는 브라우저용 SDK, 서버 API, 보안 규칙이 이미 준비되어 있습니다. 따라서 브라우저가 직접 서비스에 요청합니다.

사용자 정의 서버 작업

Gemini 비밀키 사용과 교사 전용 생성 절차는 Firebase에 미리 들어 있는 기능이 아니므로 개발자가 Functions에 작성하고 Callable로 호출합니다.

09 · COMMON MISUNDERSTANDINGS

자주 생기는 오해와 점검 방법

SOP·CORS·Callable의 위치를 잘못 이해하면 통신 오류를 보안 성공으로 착각하거나, 반대로 정상적인 CORS 오류를 데이터 권한 오류로 오해할 수 있습니다.

×
“CORS를 허용했으니 교사만 접근한다.”

출처 허용과 교사 역할은 별개입니다. 같은 Pages의 학생과 교사도 같은 출처입니다.

×
“OPTIONS가 성공했으니 요청은 안전하다.”

OPTIONS는 통신 조건만 확인합니다. 실제 POST에서 사용자와 권한을 검사합니다.

×
“서버 함수는 브라우저로 내려온다.”

브라우저는 서버 코드가 아니라 함수 이름과 요청값을 보내고 결과만 받습니다.

×
“Firebase Functions 일부는 브라우저에서 실행된다.”

Firebase Functions는 모두 서버에서 실행됩니다. 브라우저의 것은 클라이언트 SDK 함수입니다.

×
“CORS 오류는 로그인 오류다.”

CORS는 실제 함수에 도달하기 전에 발생할 수 있습니다. Auth 오류와 단계가 다릅니다.

×
“CORS로 Gemini API 키를 보호한다.”

브라우저에 키를 넣으면 이미 노출된 것입니다. 비밀키는 Secret Manager와 서버 함수에 둡니다.
