01 · THE SIX FINAL QUESTIONS

이 여섯 질문에 답할 수 있어야 배포할 수 있습니다

화면이 정상적으로 보이고 저장 버튼이 작동하는 것만으로는 안전성을 판단할 수 없습니다. 다음 질문에 파일·경로·Rules·테스트 결과로 답할 수 있어야 합니다.

QUESTION 01현재 설계가 안전한가?

공개되는 브라우저와 신뢰할 서버·Firebase 영역의 경계가 분명한가?

QUESTION 02학생은 정확히 무엇을 할 수 있는가?

승인된 학생이 자신의 제출물과 공개된 피드백만 다루는가?

QUESTION 03교사는 정확히 무엇을 할 수 있는가?

로그인한 모든 사용자가 아니라 담당 교사만 학급 데이터를 다루는가?

QUESTION 04데이터를 어떻게 나누어 저장하는가?

서로 다른 공개 범위의 데이터를 같은 문서에 섞지 않았는가?

QUESTION 05Rules는 무엇을 차단하는가?

타인 문서, 역할 변경, 보호 필드 변경과 과도한 목록 조회를 거부하는가?

QUESTION 06배포 전에 무엇을 시험하는가?

허용 요청뿐 아니라 반드시 거부되어야 할 요청을 실제 계정으로 시험했는가?

이 문서의 판정 기준: 학생·교사 URL을 알고 소스코드를 모두 읽은 사용자가 개발자 도구나 직접 요청을 사용하더라도, 인증된 역할과 데이터 소유권이 없으면 Firebase가 데이터를 거부해야 합니다.

학생 화면과 교사 화면은 공개될 수 있습니다

라우트 가드

로그인 전 교사 화면을 숨겨 사용자 경험을 정리합니다. 하지만 개발자 도구로 우회할 수 있으므로 데이터 보안의 최종 장벽이 아닙니다.

Authentication

사용자가 누구인지 확인합니다. 로그인했다는 사실만으로 교사 권한이나 모든 데이터 접근 권한이 생기지는 않습니다.

Security Rules

브라우저를 신뢰하지 않고 Firebase 서버에서 실제 읽기·쓰기 요청을 허용하거나 거부합니다.

API 키도 같은 경계를 따릅니다. Firebase 웹 apiKey는 공개 설정이지만 Gemini·Upstage 키와 Firebase Admin 서비스 계정은 비밀입니다. GEMINI_API_KEY라는 환경변수 이름만으로 안전이 완성되지 않으며, 키를 읽는 코드가 서버 함수에서 실행되고 브라우저 응답·로그로 전달되지 않아야 합니다.

04 · DOCUMENT-LEVEL DATA MODEL

서로 다른 공개 범위는 서로 다른 문서로 나눕니다

Firestore의 읽기는 문서 단위입니다. 한 문서를 읽도록 허용하면 그 문서의 모든 필드를 받습니다. 학생에게 숨길 교사 초안이나 다른 학생 정보는 같은 문서에 넣지 않습니다.

users/{uid} ├─ displayName, studentNumber └─ 개인 프로필 — 본인·허용된 교사만 joinRequests/{uid} └─ requestedClassId, status, requestedAt classes/{classId} ├─ className, teacherUid ├─ members/{studentUid} │ └─ studentNumber, status └─ activities/{activityId} ├─ title, openAt, closeAt └─ submissions/{studentUid} ├─ ownerUid, answer, createdAt, updatedAt ├─ teacherPrivate/draft │ └─ AI 초안·교사 메모 — 학생 읽기 금지 └─ feedback/final └─ 확정 피드백 — 해당 학생 읽기 전용
피해야 할 구조
classes/classA 문서 하나 
{
  students: {
    uid01: { number: "10101" },
    uid02: { number: "10102" }
  },
  teacherDrafts: { ... }
}

문서를 읽으면 여러 학생 정보와 숨겨야 할 필드가 함께 전달될 수 있습니다.

Rules가 문서 경로마다 서로 다른 읽기·쓰기 조건을 적용할 수 있습니다.

UID를 문서 ID로

학생별 문서가 하나라면 UID를 문서 ID로 사용하여 본인 문서를 직접 찾고 중복 제출을 줄일 수 있습니다.

ID는 보안이 아님

UID나 문서 ID를 추측하기 어렵게 만드는 것만으로 보호되지 않습니다. 최종 보안은 Rules가 담당합니다.

쿼리도 구조에 맞게

교사 대시보드는 담당 학급 경로 안에서 조회하고, 학생은 자신의 문서를 직접 조회하도록 설계하면 Rules가 명확해집니다.

Rules는 필터가 아닙니다. 권한 없는 문서가 결과에 포함될 가능성이 있는 쿼리는 일부만 걸러 주는 것이 아니라 요청 전체를 거부합니다. 쿼리의 where·경로·제한 조건이 Rules가 허용하는 범위와 맞아야 합니다.
05 · FIREBASE ID TOKEN & JWT

JWT는 내용을 숨기는 봉투가 아니라 위조를 판별하는 증명서입니다

Firebase 로그인에 성공하면 Firebase Authentication이 UID를 식별하는 ID Token을 발급합니다. 이 ID Token은 JWT 형식이며, 브라우저가 내용을 읽을 수 있지만 Firebase의 유효한 서명을 새로 만들 수는 없습니다.

PART 1Header서명 알고리즘과 공개키 식별자
alg: RS256
kid: ...
PART 2PayloadUID·발급자·대상·만료·Custom Claims
Base64URL 인코딩이며 암호화가 아님
PART 3SignatureHeader와 Payload가 Firebase 발급 후 변조되지 않았는지 서버가 공개키로 검증
xxxxx.yyyyy.zzzzz
Header.Payload.Signature

Payload 예시 — 설명용이며 실제 토큰 값이 아닙니다.
{
  "sub": "teacherUid01",
  "aud": "school-app-project",
  "iss": "https://securetoken.google.com/school-app-project",
  "iat": 1780000000,
  "exp": 1780003600,
  "role": "teacher"
}
JWT Payload에 비밀정보를 넣지 않습니다. 비밀번호, API 키, 학생 답안, 민감 개인정보를 넣는 공간이 아닙니다. 사용자가 Payload를 바꾸어 볼 수는 있지만, Signature가 맞지 않으므로 정상적인 Firebase 서버 검증에서는 거부되어야 합니다.

06 · CUSTOM CLAIMS

교사 역할은 학생이 수정할 수 없는 토큰 안의 권한 표시로 전달합니다

Custom Claims는 이미 Firebase Authentication으로 로그인한 사용자에게 역할과 접근 수준을 부여하는 기능입니다. 로그인 방법을 새로 만드는 Custom Authentication과는 다른 개념입니다.

권한 있는 관리자 서버Firebase Admin SDK
setCustomUserClaims()
→
Firebase Authentication다음 ID Token에
role: teacher 반영
→
Rules·서버 함수request.auth.token.role
검증 후 교사 기능 허용

Claims에 적합한 값

  • role: "teacher"
  • admin: true

  • 학생 이름·학번·상세 프로필

권장 조합: Custom Claims에는 role: "teacher"처럼 작고 전역적인 역할만 두고, 실제 담당 학급과 승인 상태는 classes/{classId} 및 members/{uid} 문서로 확인합니다.

반드시 알아야 할 동작

서버에서만 설정

브라우저가 보낸 role: teacher 값을 믿지 않습니다. Admin SDK가 실행되는 권한 있는 서버 환경에서만 설정합니다.

다음 토큰에 반영

Claims를 변경해도 현재 ID Token이 즉시 바뀌지 않습니다. 재로그인, 자동 갱신 또는 getIdToken(true)가 필요합니다.

덮어쓰기 주의

setCustomUserClaims()는 기존 Claims 전체를 덮어쓸 수 있으므로 기존 값을 읽고 병합할지 설계해야 합니다.

크기와 목적 제한

Claims Payload는 1000바이트 제한이 있으며 접근 제어용으로만 사용합니다.

교사 Claims 설정 예시 — 서버 전용 개념 코드 
import { getAuth } from "firebase-admin/auth";

// 반드시 권한 있는 서버·관리자 도구에서만 실행
await getAuth().setCustomUserClaims(teacherUid, {
  role: "teacher"
});

// 다음 ID Token 발급·갱신 후 role이 반영됩니다.

이 코드를 학생·교사 브라우저 JavaScript에 넣지 않습니다. 누가 이 관리 함수를 호출할 수 있는지도 서버에서 제한해야 합니다.

07 · APP CHECK

App Check는 사용자가 아니라 요청을 보낸 앱을 추가 확인합니다

웹에서는 reCAPTCHA Enterprise 같은 제공자가 요청 환경의 위험도를 평가하고 App Check Token 발급에 사용됩니다. 등록되지 않거나 검증되지 않은 앱에서 Firebase 자원을 남용하는 일을 줄이는 추가 방어 계층입니다.

LAYER 1 · AUTH사용자가 누구인가?

Firebase ID Token과 UID로 학생·교사 사용자를 식별합니다.

LAYER 2 · ROLE어떤 역할과 담당 범위인가?

Custom Claims와 학급 멤버십 문서로 확인합니다.

LAYER 3 · RULES이 문서에 이 작업을 해도 되는가?

소유권·필드·경로·쿼리 조건으로 허용·거부합니다.

LAYER 4 · APP CHECK등록한 앱에서 온 요청인가?

비공식 클라이언트와 자동화된 남용을 줄이는 추가 신호입니다.

App Check가 돕는 것

  • 복제 앱·스크립트의 무단 Firebase 호출 감소
  • Firestore·Functions 등 지원 서비스의 남용 방어
  • 검증·미검증 요청 지표 확인
  • Callable Function에 App Check Token 자동 첨부 가능

App Check가 대신하지 않는 것

  • 학생과 교사의 신원 확인
  • 교사 역할과 담당 학급 확인
  • 학생 본인 문서의 소유권 확인
  • 서버 함수의 입력 검증과 호출 횟수 제한

웹 앱 적용 순서

1등록

운영 도메인을 포함한 웹용 reCAPTCHA Enterprise 키를 만들고 Firebase App Check에 앱을 등록합니다.

2초기화

Firebase 서비스를 사용하기 전에 App Check SDK를 초기화합니다. 사이트 키는 브라우저 공개용입니다.

3모니터링

아직 강제하지 않고 검증된 요청과 미검증 요청 비율을 확인합니다. 로컬 개발은 Debug Provider를 사용합니다.

4강제 적용

정상 사용자와 배포 도메인이 모두 확인된 뒤 Firestore·Functions 등 필요한 서비스에 Enforcement를 켭니다.

처음부터 강제 적용하지 마세요. 설정되지 않은 운영 도메인, 오래된 배포본, 로컬 시험 환경이 모두 차단될 수 있습니다. 지표를 확인한 뒤 서비스별로 적용하고, Debug Token은 비밀처럼 관리하여 저장소나 운영 번들에 넣지 않습니다.
사이트 키는 비밀번호가 아닙니다. 웹 App Check의 reCAPTCHA 사이트 키는 브라우저에서 사용되는 공개 식별값입니다. 보안은 사이트 키 문자열을 숨기는 것이 아니라 제공자의 검증 결과와 Firebase의 App Check Token 검증·강제 적용에서 나옵니다.
08 · FIRESTORE SECURITY RULES

Rules는 로그인·역할·소유권·필드를 함께 검사합니다

아래 코드는 완제품이 아니라 권장 데이터 구조와 권한표를 Rules로 옮기는 방식의 예시입니다. 실제 필드명·학급 배정 방식·제출 마감 정책에 맞게 수정하고 Emulator 테스트를 통과해야 합니다.

기본 거부

허용 조건을 명시하지 않은 경로는 거부합니다. Test Mode의 전체 허용 규칙을 운영에 남기지 않습니다.

역할과 담당 범위 분리

교사 Claims가 있어도 모든 학급을 허용하지 않고 teacherUid 또는 담당 문서를 확인합니다.

경로와 ownerUid 이중 확인

학생 UID 문서 경로와 문서 내부 ownerUid가 모두 로그인 UID와 일치하는지 확인합니다.

변경 필드 허용 목록

diff().affectedKeys().hasOnly()로 학생이 답안·수정 시각 외 필드를 바꾸지 못하게 합니다.

읽기와 목록 구분

학생은 본인 문서를 직접 get하고, 교사의 list는 담당 학급 경로와 쿼리 범위로 제한합니다.

넓은 허용 규칙 금지

여러 Rules가 같은 경로에 맞으면 하나라도 허용할 때 접근됩니다. 더 구체적인 거부 규칙으로 넓은 허용을 취소할 수 없습니다.

설명용 Rules 골격 보기 
rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() {
      return request.auth != null;
    }

    function isTeacher() {
      return signedIn()
        && request.auth.token.role == "teacher";
    }

    function teaches(classId) {
      return isTeacher()
        && get(/databases/$(database)/documents/classes/$(classId))
             .data.teacherUid == request.auth.uid;
    }

    function isApprovedMember(classId) {
      return signedIn()
        && exists(/databases/$(database)/documents/classes/$(classId)
                  /members/$(request.auth.uid))
        && get(/databases/$(database)/documents/classes/$(classId)
               /members/$(request.auth.uid)).data.status == "approved";
    }

    match /users/{userUid} {
      // 전역 사용자 프로필은 본인만 직접 읽습니다.
      // 교사가 필요한 학생 정보는 담당 학급의 members 문서에 둡니다.
      allow get: if signedIn() && request.auth.uid == userUid;
      allow list, create, update, delete: if false;
    }

    match /joinRequests/{requestUid} {
      allow create: if signedIn()
        && request.auth.uid == requestUid
        && request.resource.data.status == "pending"
        && request.resource.data.keys().hasOnly([
          "requestedClassId", "status", "requestedAt"
        ]);

      // 담당 교사가 목록을 읽으려면 requestedClassId로 쿼리해야 합니다.
      allow get, list: if signedIn()
        && (request.auth.uid == requestUid
          || teaches(resource.data.requestedClassId));

      // 승인·거부와 멤버십 생성은 검증된 Callable/Admin 절차에서 처리합니다.
      allow update, delete: if false;
    }

    match /classes/{classId} {
      allow get: if teaches(classId) || isApprovedMember(classId);
      allow list, create, update, delete: if false;

      match /members/{memberUid} {
        allow get: if signedIn()
          && (request.auth.uid == memberUid || teaches(classId));
        allow list: if teaches(classId);
        allow write: if false; // 승인 서버·관리자 절차에서만
      }

      match /activities/{activityId} {
        allow read: if teaches(classId) || isApprovedMember(classId);
        allow create, update, delete: if teaches(classId);

        match /submissions/{studentUid} {
          allow get: if teaches(classId)
            || (isApprovedMember(classId)
              && request.auth.uid == studentUid
              && resource.data.ownerUid == request.auth.uid);
          allow list: if teaches(classId);

          allow create: if isApprovedMember(classId)
            && request.auth.uid == studentUid
            && request.resource.data.ownerUid == request.auth.uid
            && request.resource.data.keys().hasOnly([
              "ownerUid", "answer", "createdAt", "updatedAt"
            ]);

          allow update: if teaches(classId)
            || (isApprovedMember(classId)
              && request.auth.uid == studentUid
              && resource.data.ownerUid == request.auth.uid
              && request.resource.data.ownerUid == request.auth.uid
              && request.resource.data.diff(resource.data).affectedKeys()
                   .hasOnly(["answer", "updatedAt"]));

          allow delete: if teaches(classId)
            || (isApprovedMember(classId)
              && request.auth.uid == studentUid);

          match /teacherPrivate/{draftId} {
            allow read, write: if teaches(classId);
          }

          match /feedback/{feedbackId} {
            allow read: if teaches(classId)
              || (isApprovedMember(classId)
                && request.auth.uid == studentUid);
            allow write: if teaches(classId);
          }
        }
      }
    }

    match /{document=**} {
      allow read, write: if false;
    }
  }
}
이 골격을 그대로 붙여 넣고 끝내지 않습니다. 교사 여러 명, 제출 마감, 필드 자료형·길이, 서버 시각, 가입 승인 절차, 필요한 쿼리와 Rules 호출 제한을 실제 앱에 맞게 보완해야 합니다.

운영 금지

allow read, write: if true;

프로젝트 ID와 경로를 아는 누구나 전체 데이터를 읽고 바꾸거나 삭제할 수 있습니다.

여전히 과도함

allow read, write:
  if request.auth != null;

로그인한 학생 모두에게 전체 컬렉션을 허용할 수 있습니다. UID·역할·담당 범위를 더 확인해야 합니다.

09 · PRIVILEGED SERVER CODE

Admin SDK는 Rules를 우회하므로 서버 함수가 직접 권한을 검사합니다

가입 승인, Custom Claims 부여, AI 초안 생성, 대량 내보내기처럼 권한이 큰 작업은 서버 함수에서 처리할 수 있습니다. 그러나 Admin SDK로 Firestore에 접근하면 클라이언트 Security Rules가 적용되지 않습니다.

브라우저 Firebase SDK

Security Rules 적용

로그인 UID와 Rules 조건에 따라 Firestore가 허용·거부합니다.

서버 Firebase Admin SDK

Security Rules 우회

서버 자격 증명의 권한으로 접근하므로 함수 안에서 역할·담당 범위·입력을 직접 검사합니다.

CHECK 1ID Token 검증

유효한 Firebase 사용자이며 토큰이 해당 프로젝트에서 발급되었는지 확인합니다.

CHECK 2교사 역할 검증

검증된 Token의 Custom Claims에서 교사 역할을 확인합니다.

CHECK 3담당 학급 검증

요청 대상 제출물이 로그인 교사가 담당한 학급에 속하는지 서버에서 확인합니다.

CHECK 4입력·횟수 검증

필드 형식·길이·허용 작업·중복 요청·호출 횟수를 제한합니다.

CHECK 5비밀정보 경계

서비스 계정과 외부 API 키를 서버 환경에만 두고 응답·로그로 반환하지 않습니다.

화면에서 버튼을 숨긴 것으로 서버 함수가 보호되지 않습니다. 학생이 함수 이름과 요청 형식을 알아내 직접 호출하더라도 교사 전용 작업은 서버에서 거부되어야 합니다.
10 · TEST MODE TO PRODUCTION

기능 완성보다 먼저 운영 Rules와 거부 테스트를 준비합니다

Test Mode는 빠른 실험을 위한 임시 상태입니다. 실제 학생에게 URL을 전달하기 전에 데이터 구조, Authentication, Claims, Rules, App Check와 테스트 계정을 운영 기준으로 전환합니다.

1개발 환경 분리

가능하면 개발용과 운영용 Firebase 프로젝트를 나누고 개발 환경에는 가짜 학생 데이터만 사용합니다.

2Rules를 파일로 관리

firestore.rules를 Git으로 관리하고 콘솔의 최신 배포본과 파일이 일치하는지 확인합니다.

3Emulator 테스트

미로그인·학생 A·학생 B·담당 교사·비담당 교사 역할로 허용과 거부를 자동 시험합니다.

4실제 배포 검증

운영 URL과 실제 테스트 계정으로 직접 경로 요청, 역할 변경, 목록 조회와 App Check를 확인합니다.

권장 배포 순서

  1. 운영 데이터 구조와 권한표를 확정합니다.
  2. Authentication 제공자와 승인 도메인을 설정합니다.
  3. 교사 Claims와 학급 멤버십을 권한 있는 절차로 부여합니다.
  4. 운영 Rules를 Emulator에서 테스트하고 배포합니다.
  5. 학생·교사 화면을 배포하고 비로그인 상태부터 확인합니다.
  6. 학생 A·학생 B·담당 교사·비담당 교사로 교차 시험합니다.
  7. App Check를 초기화하고 지표를 확인한 뒤 필요한 서비스에 강제 적용합니다.
  8. 오류·권한 거부·사용량을 확인하고 학생에게 URL을 전달합니다.
화면 배포와 Rules 배포는 별개입니다. Vercel·Netlify·GitHub Pages에서 화면을 되돌려도 Firestore Rules와 데이터가 함께 되돌아가지는 않습니다. Rules 파일과 데이터 복구 절차를 별도로 관리해야 합니다.

12 · PROMPTS FOR AI CODING

AI에게 구조와 거부 테스트를 구체적으로 요청합니다

“보안을 강화해 주세요”라고만 요청하지 말고, 신원·역할·소유권·필드·서버 경계와 거부 테스트를 파일별로 확인하게 합니다.

프롬프트 1 · 최종 설계 및 구현 요구사항

프로젝트 설계·수정 전에 전달
학생 활동 화면과 교사 대시보드가 있는 Firebase 웹 앱을 설계해 주세요.
전문 개발자가 아닌 교사가 유지할 앱이므로 구조는 단순하게 유지하되 다음 보안 기준은 생략하지 마세요.

[기본 구조]
- 학생 화면과 교사 화면은 한 배포의 서로 다른 경로나 별도 배포가 될 수 있습니다.
- 화면 분리는 보안 수단으로 사용하지 않습니다.
- Firebase Authentication이 모든 사용자의 UID와 ID Token을 발급합니다.
- Firestore Security Rules가 실제 데이터 요청을 서버에서 허용하거나 거부합니다.

[역할과 멤버십]
- 교사 역할은 브라우저가 쓰는 users 문서의 role 값을 믿지 않습니다.
- 교사처럼 전역적인 작은 역할은 Firebase Admin SDK가 서버에서 설정한 Custom Claims를 사용합니다.
- Custom Claims는 role 같은 접근 제어 정보에만 사용하고 학생 프로필·학급 목록·답안은 넣지 않습니다.
- 담당 학급과 학생 승인 상태는 classes/{classId} 및 members/{uid} 문서에서 확인합니다.
- Claims 변경 후 새 ID Token 발급 또는 강제 갱신이 필요한 흐름을 설명해 주세요.

[권장 데이터 구조]
- classes/{classId}/members/{uid}
- classes/{classId}/activities/{activityId}
- classes/{classId}/activities/{activityId}/submissions/{studentUid}
- 교사용 비공개 초안은 submissions 아래 teacherPrivate 별도 문서
- 학생 공개용 확정 피드백은 feedback/final 별도 문서
- 한 문서에 여러 학생 개인정보나 교사 비공개 초안을 함께 저장하지 않습니다.

[Rules]
- 기본 거부 원칙을 적용합니다.
- 학생은 승인된 학급의 자신의 문서만 생성·조회·수정·삭제할 수 있습니다.
- 학생은 ownerUid, role, status, 교사용 초안, 확정 피드백을 수정할 수 없습니다.
- 교사는 Custom Claims의 교사 역할과 담당 학급을 모두 확인한 뒤 해당 학급만 처리할 수 있습니다.
- get과 list를 구분하고 Rules는 필터가 아니라는 점을 쿼리에 반영합니다.
- 학생 수정 가능 필드는 diff().affectedKeys().hasOnly() 방식으로 제한합니다.
- Test Mode 전체 허용 규칙을 남기지 않습니다.

[JWT와 서버 함수]
- Firebase ID Token은 JWT이며 Payload는 암호화된 비밀 공간이 아님을 전제로 합니다.
- 자체 서버나 서버리스 함수에서는 Firebase Admin SDK로 ID Token의 서명·발급자·대상·만료를 검증합니다.
- Admin SDK가 Firestore Rules를 우회하므로 서버 함수에서 교사 역할, 담당 학급, 요청 대상과 입력값을 다시 검사합니다.
- 비밀 API 키와 서비스 계정은 브라우저 코드·응답·로그에 포함하지 않습니다.

[App Check]
- App Check는 Authentication이나 Rules를 대신하지 않는 추가 남용 방어 계층으로 사용합니다.
- 웹 배포 도메인을 등록하고 App Check SDK를 초기화합니다.
- 먼저 모니터링하고 정상 요청을 확인한 뒤 Firestore와 필요한 Functions에 강제 적용합니다.
- 로컬 개발은 Debug Provider를 사용하되 Debug Token을 저장소에 커밋하지 않습니다.

먼저 파일 구조, Firestore 경로, 권한표, Rules 설계와 테스트 계획을 보고해 주세요.
아직 코드를 한꺼번에 생성하지 말고, 불명확한 역할·데이터 범위를 먼저 질문해 주세요.

프롬프트 2 · 완성 후 보안 감사

완성된 전체 프로젝트와 함께 전달
