00 · IMPORTANT CLARIFICATION

두 방식의 차이는 “URL 매개변수 사용 여부”가 아닙니다

Apps Script 프로젝트 안의 HTML 파일은 일반 웹 서버처럼 /student.html 주소로 직접 공개되지 않습니다. 웹 앱의 단일 배포 URL에 요청이 들어오면 doGet(e)가 반드시 하나의 HtmlOutput을 반환합니다.

웹 앱 배포 URL.../exec?page=student
.../exec?page=teacher
→
doGet(e)e.parameter.page 확인
반환할 화면 결정
→
HtmlOutput 한 개Student.html 또는 Teacher.html
또는 항상 Index.html
따라서 두 구조 모두 ?page=student 같은 매개변수를 사용할 수 있습니다. 실제 비교 대상은 서버의 doGet이 서로 다른 HTML 파일을 골라 반환하느냐, 아니면 항상 하나의 index.html을 반환하고 그 안에서 화면을 전환하느냐입니다.
구조 A

서버 라우팅 · 여러 HTML 파일

doGet(e)가 Student.html 또는 Teacher.html을 선택해 반환합니다.

구조 B

클라이언트 전환 · 하나의 index.html

doGet(e)는 항상 Index.html을 반환하고, JavaScript가 학생·교사 영역을 표시합니다.

학생·교사 UI와 JavaScript가 한 문서에 있고, URL 매개변수 또는 로그인 결과에 따라 한쪽을 표시합니다.

공통점

  • Apps Script 웹 앱 배포는 하나이며 기본 /exec URL도 하나입니다.
  • 학생 화면과 교사 화면 모두 google.script.run으로 같은 서버 측 Apps Script 함수에 접근할 수 있습니다.
  • 데이터는 같은 Google Sheets에 저장할 수 있습니다.
  • URL의 ?page=teacher는 화면 선택 정보이지 교사 인증 정보가 아닙니다.
  • 서버 함수는 브라우저에서 직접 호출될 수 있다고 가정하고 매번 입력과 권한을 확인해야 합니다.
02 · SERVER-ROUTED MULTI-FILE

구조 A: Student.html과 Teacher.html을 분리

doGet(e)가 요청한 화면과 사용자 권한을 확인한 후 서로 다른 HTML 파일을 반환합니다. 학생 화면과 교사 대시보드의 기능과 레이아웃이 크게 다를 때 이해하기 쉽습니다.

장점

  • 학생과 교사 코드의 책임 범위가 명확함
  • 각 HTML 파일이 작아져 읽고 수정하기 쉬움
  • 학생 요청에 교사용 HTML·JS를 반환하지 않도록 구성 가능
  • 대시보드가 커져도 학생 화면 파일이 복잡해지지 않음
  • 각 화면을 독립적으로 시험하기 쉬움

단점

  • 공통 CSS·헤더·함수가 중복되기 쉬움
  • 화면 이동 때 전체 페이지가 다시 로드됨
  • 두 파일의 데이터 필드와 디자인이 어긋날 수 있음
  • 공통 변경을 여러 파일에 반복할 가능성
  • doGet 라우팅과 템플릿 include 구조가 필요

03 · SINGLE INDEX CLIENT VIEW

구조 B: index.html 하나에서 화면 전환

한 HTML 문서 안에 학생 영역과 교사 영역을 두고, JavaScript가 현재 페이지 상태에 따라 필요한 영역을 표시합니다. 규모가 작고 두 화면이 많은 디자인과 상태를 공유할 때 빠르게 구현할 수 있습니다.

장점

  • 파일 수가 적고 초기 구조가 단순함
  • 공통 CSS·함수·로그인 상태를 자연스럽게 공유
  • 전체 새로고침 없이 빠른 화면 전환 가능
  • 하나의 디자인 시스템을 유지하기 쉬움
  • google.script.history와 URL 상태를 연동할 수 있음

단점

  • 기능이 늘면 하나의 거대한 파일이 되기 쉬움
  • 학생 브라우저에도 교사용 HTML·클라이언트 코드가 전달될 수 있음
  • 숨김 상태와 현재 페이지 상태 관리가 복잡해짐
  • 교사 데이터까지 미리 불러오는 실수가 생길 수 있음
  • 한 곳의 JavaScript 오류가 두 화면에 모두 영향을 줌

기본 구조 예시

<main id="student-view" hidden>
  학생 활동 화면
</main>

<main id="teacher-view" hidden>
  교사 대시보드
</main>

<script>
  const page = initialPage; // student 또는 teacher
  document.getElementById(`${page}-view`).hidden = false;
</script>
hidden은 보안이 아닙니다. 학생이 교사 영역의 숨김 속성을 제거할 수 있습니다. 교사 데이터는 권한 확인이 끝난 뒤에만 서버에서 가져오고, 서버 함수도 교사 권한을 독립적으로 검사해야 합니다.

05 · SECURITY

화면 구성과 실제 보안을 분리해서 생각해야 합니다

두 파일을 사용하면 교사 HTML을 학생에게 반환하지 않도록 만들 수 있어 코드 구성상 이점은 있습니다. 하지만 ?page=teacher를 바꾸는 것만으로 교사 화면을 받을 수 있다면 보안상 분리된 것이 아닙니다.

×
오해: Teacher.html은 별도 파일이므로 안전하다

doGet이 매개변수만 보고 반환하면 누구나 ?page=teacher로 요청할 수 있습니다.

×
오해: index에서 교사 영역을 숨기면 안전하다

개발자 도구로 hidden을 제거하고 교사 함수를 직접 호출할 수 있습니다.

×
오해: 버튼이 없으면 서버 함수를 호출할 수 없다

클라이언트 코드를 수정해 공개된 google.script.run 서버 함수를 호출할 수 있다고 가정해야 합니다.

×
오해: 시트 URL을 숨기면 데이터가 안전하다

서버 함수가 방문자의 입력을 그대로 믿으면 Apps Script가 소유자 권한으로 시트를 읽어 줄 수 있습니다.

권장 방어 계층

1
배포 접근 범위

학교 도메인, 로그인 사용자, 익명 공개 중 실제 수업 환경에 맞는 최소 범위를 선택합니다.

2
doGet의 화면 반환 조건

교사 화면을 요청하면 가능할 때 서버에서 사용자 신원과 교사 허용 목록을 확인한 뒤 반환합니다.

3
각 서버 함수의 독립적인 권한 확인

getTeacherDashboard(), saveFeedback(), exportData()가 각각 교사 권한을 검사합니다.

4
학생 데이터 소유권

클라이언트가 보낸 학번을 신뢰하지 않고 가능한 경우 인증된 사용자와 학생 명단을 연결합니다.

5
입력 검증

허용 필드, 자료형, 길이, 학급·활동 ID를 서버에서 확인하고 수식 주입을 방지합니다.

6
시트·운영 보호

시트를 불필요하게 학생과 공유하지 않고 실행 기록, 할당량, 동시 쓰기와 오류를 관찰합니다.

가장 중요한 원칙: doGet에서 교사 화면을 막았더라도 서버 함수에서 다시 검사해야 합니다. 사용자는 화면을 거치지 않고 클라이언트 코드를 변형해 서버 함수를 호출할 수 있습니다.
06 · DEPLOYMENT IDENTITY

“누구로 실행할 것인가”가 보안에 더 큰 영향을 줍니다

Apps Script 웹 앱은 배포할 때 스크립트 소유자로 실행하거나, 웹 앱에 접속한 사용자로 실행할 수 있습니다. 이 선택은 두 HTML 파일인지 index 하나인지보다 데이터 접근 권한에 더 직접적인 영향을 줍니다.

배포 설정특징장점주의점
나로 실행
USER_DEPLOYING		모든 서버 함수가 배포자의 권한으로 실행학생에게 Google Sheet를 공유하지 않아도 앱이 시트에 저장 가능방문자가 소유자 권한을 간접 이용하므로 모든 서버 함수의 권한·입력 검증이 필수
접속 사용자로 실행
USER_ACCESSING		각 사용자의 Google 권한으로 실행Google 계정 기반 사용자 맥락을 활용할 수 있음사용자별 승인 과정과 할당량이 있으며 시트 권한이 없으면 작업이 실패할 수 있음

Session 이메일의 중요한 제한

Session.getActiveUser().getEmail()은 실행 환경과 보안 정책에 따라 빈 문자열을 반환할 수 있습니다. 특히 웹 앱을 “나로 실행”하여 방문자가 직접 승인하지 않는 상황에서는 방문자 이메일을 항상 얻을 수 있다고 가정하면 안 됩니다.

function requireTeacher_() {
  const email = Session.getActiveUser().getEmail();

  if (!email || !TEACHER_EMAILS.includes(email)) {
    throw new Error('교사 권한이 없습니다.');
  }
}
이 코드는 이메일을 얻을 수 있는 배포·도메인 조건에서만 작동합니다. 배포 전에 실제 학생 계정과 교사 계정으로 getActiveUser().getEmail() 결과를 확인하십시오. getEffectiveUser()는 “나로 실행” 배포에서 배포자 계정을 반환하므로 방문자 식별에 사용할 수 없습니다.

방문자 신원을 얻을 수 없다면

  • 학교 Google Workspace 도메인 내 로그인 사용자만 접근하도록 배포 조건을 조정합니다.
  • 교사용 앱을 별도 배포하여 접근 범위를 더 엄격하게 제한하는 방안을 검토합니다.
  • 개별 사용자를 식별해야 하는 민감한 앱이라면 Firebase Auth 또는 별도 인증 백엔드를 검토합니다.
  • 서버에 저장한 공통 교사 비밀번호는 클라이언트 하드코딩보다 낫지만 개인별 책임 추적·권한 회수에는 한계가 있습니다.
07 · IMPLEMENTATION EXAMPLES

두 구조의 안전한 기본 구현

A. 여러 HTML 파일을 명시적으로 선택

function doGet(e) {
  const page = e?.parameter?.page || 'student';

  // 요청값을 파일 이름으로 그대로 사용하지 않고 허용 목록으로 매핑
  const files = {
    student: 'Student',
    teacher: 'Teacher'
  };

  if (!(page in files)) {
    return HtmlService.createHtmlOutput('잘못된 페이지입니다.');
  }

  if (page === 'teacher') {
    requireTeacher_();
  }

  return HtmlService
    .createTemplateFromFile(files[page])
    .evaluate()
    .setTitle(page === 'teacher' ? '교사 대시보드' : '학생 활동');
}
허용 목록이 중요합니다. createTemplateFromFile(e.parameter.page)처럼 사용자 입력을 파일 이름으로 직접 사용하지 말고, 미리 정한 키만 실제 파일에 연결합니다.

공통 CSS·JS include

function include_(filename) {
  return HtmlService.createHtmlOutputFromFile(filename).getContent();
}

// Student.html 또는 Teacher.html
<style>
  <?!= include_('SharedCSS'); ?>
</style>

<script>

  • “나로 실행” 앱이라면 시트를 학생에게 직접 공유하지 않아도 됩니다.

  • 중복 제출 방지를 서버에서 다시 확인합니다.

10 · RELEASE CHECKLIST

배포 전 점검표

구조·유지보수

  • 두 HTML 파일과 두 배포를 혼동하지 않는다.
  • doGet이 허용된 page 값만 처리한다.
  • 공통 CSS·JS·필드명이 한곳에서 관리된다.
  • index 방식이면 브라우저 뒤로가기와 새로고침을 시험했다.
  • 여러 HTML 방식이면 양쪽 화면의 공통 변경을 함께 시험했다.
  • 운영 배포 버전을 업데이트하고 /exec에서 확인했다.

보안·데이터

  • ?page=teacher를 교사 인증으로 사용하지 않는다.
  • 각 교사 서버 함수가 권한을 독립적으로 확인한다.
  • 실제 학생·교사 계정에서 활성 사용자 이메일을 확인했다.
  • getEffectiveUser()를 방문자 신원으로 착각하지 않는다.
  • 학생이 보낸 학번·역할·행 번호를 서버가 검증한다.
  • 시트가 학생에게 불필요하게 직접 공유되지 않았다.
  • 수식 주입·중복 제출·동시 수정을 처리한다.
  • 화면을 우회해 서버 함수를 호출하는 공격을 시험했다.
배포 전 중단 기준: 방문자를 식별할 수 없는데 교사 대시보드가 민감한 학생 데이터를 반환하거나, 교사 함수가 공통 비밀번호·클라이언트 변수만 믿는다면 화면 구조를 선택하기 전에 인증 설계를 다시 해야 합니다.
11 · RECOMMENDATION

최종 추천

학생 활동 화면과 교사 대시보드가 기능적으로 뚜렷하게 다르다면, 한 배포에서 Student.html과 Teacher.html을 서버가 선택해 반환하고 공통 CSS·JS를 include하는 구조가 가장 이해하기 쉽고 확장에도 유리합니다.

대부분의 수업 앱

여러 HTML + 공통 include + 서버 권한 검사

학생과 교사 화면을 읽기 쉽게 분리하면서 한 배포와 공통 코드를 유지합니다.

작고 단순한 프로토타입

index.html 하나 + 내부 화면 전환

빠르게 시작하되 파일이 커지면 화면과 서버 서비스를 분리할 계획을 세웁니다.

보안상 더 중요한 결정: HTML 파일 개수보다 웹 앱의 접근 범위, 실행 사용자, 방문자 식별 가능 여부와 모든 서버 함수의 권한 검사가 우선입니다.

FINAL · STRUCTURE SELECTOR

어떤 화면 구조가 적합한지 확인하는 체크리스트

아래 질문에 답하면 여러 HTML, 단일 index, 별도 배포 세 가지 대안을 비교합니다. 별도 배포는 접근 정책과 운영 주기를 정말 분리해야 할 때만 후보가 됩니다.

1. 학생 화면과 교사 화면의 기능·레이아웃 차이가 큰가요?입력 중심 화면과 대시보드가 서로 다른 앱처럼 느껴지는지 확인합니다.
2. 두 화면이 로그인 상태·스타일·클라이언트 상태를 많이 공유하나요?한 문서에서 상태를 유지하는 이점이 큰지 확인합니다.
3. 교사 대시보드가 표·검색·통계·피드백으로 계속 커질 예정인가요?단일 index가 장기적으로 복잡해질 가능성을 확인합니다.
4. 화면 전환 때 새로고침 없이 즉시 상태를 유지해야 하나요?SPA와 비슷한 사용자 경험이 중요한지 확인합니다.
5. 학생과 교사에게 서로 다른 배포 접근 범위·실행 권한이 반드시 필요한가요?한 배포의 화면 분리로 해결되지 않는 정책 요구인지 확인합니다.
6. 두 앱의 담당자와 배포 시기가 완전히 다른가요?별도 배포의 운영 비용을 감수할 이유가 있는지 확인합니다.
7. 공통 파일 include와 doGet 라우팅을 배울 수 있나요?여러 HTML 구조의 초기 학습 비용을 감당할 수 있는지 확인합니다.
8. 학생에게 교사용 HTML·클라이언트 코드를 아예 반환하지 않는 것이 운영상 유리한가요?이는 보안 완성책은 아니지만 코드 노출과 실수를 줄이는 데 도움이 될 수 있습니다.
9. 한 파일이 커져도 함수·컴포넌트를 체계적으로 분리할 수 있나요?단일 index 구조의 장기 유지보수 역량을 확인합니다.
10. 실제 방문자의 이메일·역할을 서버에서 안정적으로 확인할 수 있나요?어떤 화면 구조를 선택해도 먼저 해결해야 하는 보안 조건입니다.
11. 교사 서버 함수가 화면과 무관하게 권한을 다시 검사하나요?파일 구조보다 우선하는 필수 보안 항목입니다.

선택 도구의 한계: 구조 점수가 높아도 사용자 신원과 서버 권한 검사가 불확실하면 먼저 인증 설계를 해결해야 합니다. 별도 배포 역시 자동 보안이 아니라 서로 다른 접근 정책을 적용할 수 있는 운영 수단입니다.
