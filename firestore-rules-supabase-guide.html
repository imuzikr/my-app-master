01 · THE BIG PICTURE

데이터베이스를 선택해도 권한 설계는 사라지지 않습니다

Firebase와 Supabase는 데이터 저장 방식이 다르지만, 브라우저에서 데이터베이스를 직접 사용하는 앱이라면 “누가 어떤 데이터를 읽고 바꿀 수 있는가”를 데이터베이스 쪽에서 검사해야 한다는 원칙은 같습니다.

NoSQL

Cloud Firestore

데이터를 컬렉션 → 문서 → 필드로 저장하는 문서형 NoSQL 데이터베이스입니다. Firebase Authentication이 사용자 UID를 확인하고 Security Rules가 문서 요청을 허용하거나 거부합니다.

SQL

Supabase PostgreSQL

데이터를 테이블 → 행 → 열로 저장하는 관계형 SQL 데이터베이스입니다. Supabase Auth, PostgreSQL 권한(Grants), RLS 정책이 함께 접근 범위를 결정합니다.

학생·교사 브라우저 문서를 읽거나 새 데이터를 저장해 달라고 요청합니다.
→
Authentication + 권한 정책 로그인 UID, 역할, 소유자, 학급 소속을 검사합니다.
→
허용된 데이터만 처리 화면을 숨겼는지가 아니라 서버 측 정책을 통과했는지가 기준입니다.
‘프로덕션 모드’도 완성 규칙은 아닙니다. Firestore 테스트 모드는 개발을 쉽게 시작하도록 비교적 열린 규칙을 사용할 수 있고, 초기 프로덕션 모드는 웹·모바일 요청을 우선 막는 출발점입니다. 어느 쪽도 학생·교사 앱에 맞는 권한을 자동으로 완성해 주지는 않습니다.
02 · COLLECTION, DOCUMENT, FIELD

Firestore는 문서를 중심으로 생각합니다

Firestore 경로는 컬렉션과 문서가 번갈아 나타납니다. 실제 읽기·쓰기의 대상은 문서이며, 문서 안에 이름·역할·응답 같은 필드가 들어갑니다.

학급과 학생 회원 구조
classes · 컬렉션
classA · 문서 IDclasses/classA
name: "1학년 1반" · 필드
teacherUid: "teacherUid01" · 필드
members · 하위 컬렉션
studentUid01 · 회원 문서classes/classA/members/studentUid01
회원 문서 안의 필드
문서 ID: studentUid01
studentNumber: "10101"
displayName: "학생 표시명"
role: "student"
joinedAt: Timestamp

컬렉션

같은 종류의 문서를 담는 묶음입니다. 컬렉션 자체가 JSON 문서처럼 읽히는 것이 아니라 그 안의 문서들을 조회합니다.

문서

Firestore의 기본 읽기·쓰기 단위입니다. Rules는 요청한 문서 경로와 조건을 비교합니다.

필드

문서 안의 실제 값입니다. 문자열·숫자·배열·맵·Timestamp 등으로 저장할 수 있습니다.

경로 읽는 법: classes/classA/members/studentUid01은 classes 컬렉션의 classA 문서 아래, members 하위 컬렉션의 studentUid01 문서를 의미합니다.
03 · DOCUMENT AS A SECURITY BOUNDARY

문서 안의 일부 필드만 숨길 수는 없습니다

Firestore에서 읽기는 문서 단위입니다. 사용자가 문서를 읽도록 허용되면 그 안의 모든 필드를 받습니다. Rules로 name은 보여 주고 teacherMemo만 감추는 방식은 만들 수 없습니다.

피해야 할 구조 · 한 문서에 모두 모으기
classes/allClasses
classA: { 학급 정보 전체 }
classB: { 학급 정보 전체 }
classC: { 학급 정보 전체 }
문제: classA만 필요해도 allClasses 문서 전체를 읽음
권장 구조 · 학급마다 독립 문서
classes/classA1학년 1반 기본 정보
classes/classB1학년 2반 기본 정보
classes/classC1학년 3반 기본 정보
장점: 필요한 학급 문서 하나만 정확히 요청

학급 문서를 나눈 뒤에도 공개 범위를 다시 확인합니다

같은 문서에 공개·비공개 정보가 섞임
classes/classA
name: 학생에게 공개 가능
studentIds: 전체 명단 노출 가능
teacherMemo: 교사 전용
aiFeedbackDrafts: 교사 전용
공개 범위에 따라 문서를 분리
classes/classA학급 구성원이 보는 기본 정보
classes/classA/members/{uid}학생별 회원 정보
classes/classA/submissions/{id}학생별 제출물
classes/classA/private/teacherData교사만 읽는 정보
화면에서 감추는 것과 데이터 접근을 막는 것은 다릅니다. JavaScript가 teacherMemo를 화면에 표시하지 않아도 학생에게 학급 문서 읽기가 허용되었다면 브라우저 개발자 도구에서 전체 문서를 확인할 수 있습니다.
실전 판단 기준: 읽을 수 있는 사람이 같고 함께 변경되는 값은 같은 문서에 둘 수 있습니다. 읽을 수 있는 사람이 다르거나 계속 늘어나는 회원·제출물·피드백은 별도 문서 또는 하위 컬렉션으로 분리합니다.

계속 커지는 배열이나 맵을 한 문서에 모으면 권한 분리가 어려울 뿐 아니라 문서 크기 제한과 동시 수정 부담도 커집니다. Firestore 기본 모드의 문서 크기 한도는 1MiB이므로, 학생 수와 제출 횟수에 따라 늘어나는 데이터는 독립 문서로 관리하는 편이 좋습니다.

04 · DOCUMENT ID DESIGN

임의 ID 대신 의미 있는 특정 ID를 사용할 수 있습니다

Firestore가 자동 생성한 문서 ID를 사용하는 방법과 개발자가 안정적인 ID를 지정하는 방법은 모두 정상적인 방식입니다. 학급 ID나 Firebase Authentication UID를 문서 ID로 사용하면 정확한 문서를 직접 찾고 Rules의 비교 조건을 단순하게 만들 수 있습니다.

AUTO

자동 생성 문서 ID

const ref = await addDoc(
  collection(db, "classes"),
  classData
);

console.log(ref.id); // 임의로 생성된 ID

공지·로그·여러 번 제출하는 기록처럼 새 문서를 계속 추가할 때 편리합니다.

SET

개발자가 지정한 문서 ID

const classId = "cls_2026_01";

await setDoc(
  doc(db, "classes", classId),
  classData
);

학급처럼 하나의 안정적인 ID로 한 문서를 직접 찾고 관리할 때 유용합니다.

학생 UID를 회원 문서 ID로 사용

const uid = auth.currentUser.uid;

await setDoc(
  doc(db, "classes", classId, "members", uid),
  {
    studentNumber: "10101",
    role: "student"
  }
);

저장된 경로는 다음과 같습니다.

classes/cls_2026_01/members/studentUid01

학생 화면에서는 전체 회원 컬렉션을 조회하지 않고 자신의 UID로 정확한 문서 하나를 요청합니다.

const memberRef = doc(
  db,
  "classes",
  classId,
  "members",
  auth.currentUser.uid
);

const memberSnapshot = await getDoc(memberRef);

특정 문서 ID가 주는 이점

  • 전체 컬렉션 검색 없이 문서 하나를 직접 조회
  • 문서 ID와 로그인 UID를 Rules에서 바로 비교
  • 한 학생당 하나의 회원 문서처럼 중복을 줄임
  • 학급·활동·학생의 경로 관계가 명확해짐
+

그래도 반드시 필요한 Rules

  • 문서 ID를 아는 것과 접근 권한은 다름
  • 다른 학급 ID로 경로를 바꿔 요청할 수 있음
  • UID는 식별자이지 비밀번호가 아님
  • 로그인·소속·역할을 서버 측 Rules가 검사
“추측하기 어려운 ID라서 안전하다”는 설명은 틀립니다. 특정 문서 ID를 쓰면 보안 규칙을 정확하게 표현하기 쉬워지는 것이지, 문서 주소가 비밀이 되어 보호되는 것은 아닙니다. 실제 보안은 Authentication과 Rules가 담당합니다.
문서 ID 선택 팁: 표시 이름인 1학년 1반보다 cls_2026_01처럼 안정적인 내부 ID를 사용하세요. Firestore 문서 ID는 직접 이름을 바꾸는 기능이 없으므로 자주 바뀌는 값을 문서 ID로 사용하면 이전 작업이 어려워집니다.
05 · RULES LANGUAGE

Rules는 “경로 + 요청 종류 + 조건”으로 읽습니다

Firestore Rules는 화면 버튼의 표시 여부를 결정하는 프런트엔드 코드가 아닙니다. 데이터 요청이 Firebase에 도착할 때 서버 측에서 실행되어 허용 또는 거부를 결정합니다.

rules_version = '2';

service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() {
      return request.auth != null;
    }

    match /classes/{classId}/members/{memberUid} {

      allow get: if signedIn()
        && request.auth.uid == memberUid;

      allow list: if false;
    }
  }
}
rules_version = '2'

Firestore Rules 문법 버전을 지정합니다.

service cloud.firestore

이 규칙이 Cloud Firestore 요청에 적용된다는 뜻입니다.

match /.../{classId}

규칙이 적용될 문서 경로입니다. 중괄호는 실제 문서 ID가 들어갈 자리입니다.

allow get

정확한 문서 하나를 읽는 요청을 조건이 참일 때 허용합니다.

request.auth.uid

Firebase Authentication으로 로그인한 현재 사용자의 UID입니다.

memberUid

요청한 회원 문서의 문서 ID입니다. 두 UID가 같으면 자신의 문서입니다.

요청 종류를 구분합니다

Rules 요청의미대표적인 웹 SDK 호출주의점
get문서 하나 조회getDoc()정확한 문서 경로를 지정
list여러 문서 검색·목록 조회getDocs()Rules는 결과를 자동 필터링하지 않음
create새 문서 생성addDoc(), 새 문서의 setDoc()request.resource.data 검사
update기존 문서 수정updateDoc(), 기존 문서의 setDoc()기존 값과 수정 후 값을 비교
delete문서 삭제deleteDoc()기존 문서의 소유자 확인
readget + list읽기 전체편리하지만 범위가 넓음
writecreate + update + delete쓰기 전체세부 작업별 조건이 다르면 나누기

기존 문서와 저장하려는 문서를 구분합니다

resource.data

Firestore에 현재 저장되어 있는 기존 문서입니다. 현재 소유자가 누구인지 확인할 때 사용합니다.

request.resource.data

생성 또는 수정 후 저장될 문서입니다. 사용자가 ownerUid나 역할을 바꾸려고 하는지 검사할 수 있습니다.

allow update: if signedIn()
  && resource.data.ownerUid == request.auth.uid
  && request.resource.data.ownerUid
     == resource.data.ownerUid;

현재 문서의 소유자가 로그인 사용자이고, 수정 후에도 ownerUid가 그대로일 때만 허용한다는 뜻입니다.

읽기는 문서 전체지만 수정 가능 필드는 제한할 수 있습니다

allow update: if signedIn()
  && resource.data.ownerUid == request.auth.uid
  && request.resource.data
    .diff(resource.data)
    .affectedKeys()
    .hasOnly(["content", "updatedAt"]);

이 규칙은 학생이 content와 updatedAt만 수정하도록 제한합니다. 다른 필드까지 바꾸려 하면 수정 요청 전체가 거부됩니다. 반면 읽기에서는 특정 필드만 감출 수 없으므로 비공개 필드는 애초에 별도 문서로 나누어야 합니다.

Rules는 검색 결과 필터가 아닙니다. 학생이 members 전체 컬렉션을 요청하면 자신의 문서만 골라 반환되는 것이 아닙니다. 전체 요청이 Rules 조건을 만족하지 못하면 거부됩니다. 학생 화면은 자신의 UID로 문서 하나를 직접 조회하고, 목록 조회는 교사에게만 허용하는 방식이 이해하기 쉽습니다.
하위 컬렉션 규칙은 별도로 작성합니다. classes/{classId}에 작성한 규칙이 members와 submissions 하위 컬렉션에 자동으로 적용되는 것은 아닙니다.
06 · CLASS APP RULES EXAMPLE

학급·회원·활동·제출물 구조를 Rules와 함께 봅니다

아래 예시는 한 학급 안에서 학생은 자신의 회원 문서와 제출물만 사용하고, 담당 교사는 학급 전체를 관리하는 구조를 설명하기 위한 축약 예시입니다. 학생 수정 필드는 제한했지만 실제 앱에서는 가입 승인·여러 교사·학년도 이관·자료형과 길이 검사를 추가해야 합니다.

설명용 일부 코드 — 복사·배포 금지. 실제 배포본은 Emulator에서 허용·거부 테스트를 모두 통과해야 합니다.

classes/{classId}
├─ name
└─ teacherUid

classes/{classId}/members/{studentUid}
├─ studentNumber
└─ role

classes/{classId}/activities/{activityId}
└─ title

classes/{classId}/activities/{activityId}/submissions/{studentUid}
├─ ownerUid
├─ content
└─ updatedAt

classes/{classId}/private/teacherData
└─ teacherMemo
데이터해당 학생다른 학생담당 교사비로그인
학급 기본 정보읽기소속 아니면 거부읽기·수정거부
자신의 회원 문서읽기거부전체 관리거부
자신의 제출물CRUD거부전체 CRUD거부
교사용 비공개 문서거부거부읽기·쓰기거부
 
rules_version = '2';

service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() {
      return request.auth != null;
    }

    function isClassMember(classId) {
      return signedIn()
        && exists(
          /databases/$(database)/documents/classes/$(classId)/members/$(request.auth.uid)
        );
    }

    function isClassTeacher(classId) {
      return signedIn()
        && get(
          /databases/$(database)/documents/classes/$(classId)
        ).data.teacherUid == request.auth.uid;
    }

    match /classes/{classId} {
      allow get: if isClassMember(classId)
        || isClassTeacher(classId);

      // 한 학급 중심의 축약 예시라 모든 사용자의 학급 목록 조회를 막습니다.
      allow list: if false;

      // 담당 교사도 teacherUid를 다른 UID로 바꾸지 못하게 합니다.
      allow update: if isClassTeacher(classId)
        && request.resource.data.teacherUid
          == resource.data.teacherUid;

      match /members/{memberUid} {
        allow get: if signedIn()
          && (
            request.auth.uid == memberUid
            || isClassTeacher(classId)
          );

        allow list: if isClassTeacher(classId);
        allow create, update, delete:
          if isClassTeacher(classId);
      }

      match /activities/{activityId} {
        allow read: if isClassMember(classId)
          || isClassTeacher(classId);

        allow create, update, delete:
          if isClassTeacher(classId);

        match /submissions/{studentUid} {
          allow create: if isClassTeacher(classId)
            || (
              isClassMember(classId)
              && request.auth.uid == studentUid
              && request.resource.data.ownerUid
                == request.auth.uid
              && request.resource.data.keys()
                .hasOnly(["ownerUid", "content", "updatedAt"])
            );

          allow get: if isClassTeacher(classId)
            || (
              isClassMember(classId)
              && request.auth.uid == studentUid
              && resource.data.ownerUid
                == request.auth.uid
            );

          // 제출물 목록은 담당 교사만 조회합니다.
          allow list: if isClassTeacher(classId);

          allow update: if isClassTeacher(classId)
            || (
                isClassMember(classId)
                && request.auth.uid == studentUid
                && resource.data.ownerUid
                  == request.auth.uid
                && request.resource.data.ownerUid
                  == resource.data.ownerUid
                && request.resource.data.diff(resource.data)
                  .affectedKeys().hasOnly(["content", "updatedAt"])
              );

          allow delete: if isClassTeacher(classId)
            || (
              isClassMember(classId)
              && request.auth.uid == studentUid
              && resource.data.ownerUid
                == request.auth.uid
            );
        }
      }

      match /private/{privateDocumentId} {
        allow read, write:
          if isClassTeacher(classId);
      }
    }
  }
}

exists()

현재 UID와 같은 회원 문서가 해당 학급에 실제로 존재하는지 검사합니다.

get()

학급 문서의 teacherUid를 읽어 현재 로그인 UID와 비교합니다.

UID 문서 ID

제출물 문서 ID와 ownerUid를 함께 검사하여 경로와 내용이 모두 학생 본인인지 확인합니다.

역할 필드를 학생이 수정하게 두지 마세요. role: "student"를 회원 문서에 저장하더라도 학생이 자신의 역할을 teacher로 바꿀 수 있으면 권한 상승 문제가 생깁니다. 역할·학급 소속·교사 UID 변경은 교사 또는 통제된 서버 작업만 허용해야 합니다.
여러 번 제출해야 한다면: 제출물 문서 ID를 학생 UID 하나로 고정하지 말고 자동 ID를 사용하면서 문서의 ownerUid를 Rules로 검사하거나, attempts/{attemptId} 하위 컬렉션을 추가합니다. 특정 ID는 데이터의 실제 관계에 맞을 때 사용해야 합니다.
교사가 여러 학급을 목록으로 조회해야 한다면: 위의 allow list: if false를 그대로 사용할 수 없습니다. 담당 교사의 UID가 일치하는 학급만 허용하도록 Rules를 설계하고, 웹 코드의 쿼리에도 where("teacherUid", "==", 현재 교사 UID) 조건을 넣어야 합니다. Rules는 검색 결과를 나중에 골라 주지 않기 때문입니다.
07 · ARE PUBLIC RULES A PROBLEM?

Rules가 보인다고 해서 보안이 무너지는 것은 아닙니다

배포된 Firestore Rules는 Firebase 서버에서 실행되며 일반 웹 앱의 JavaScript처럼 브라우저에 내려받아 실행하는 코드가 아닙니다. 다만 firestore.rules 파일을 공개 GitHub 저장소에 올리거나 문서에 예시로 게시하면 규칙 원문은 누구나 볼 수 있습니다.

안전한 Rules는 공격자가 규칙과 데이터 경로를 알고 있어도 권한 없는 요청을 거부해야 합니다.

공개되어도 보통 괜찮은 것

  • 컬렉션·문서 경로와 Rules 구조
  • request.auth.uid 비교 방식
  • 역할·소유자·소속 확인 로직
  • Firebase 웹 앱의 일반 구성 정보
  • Firebase 서비스용으로 제한된 웹 API 키

절대 넣거나 공개하면 안 되는 것

  • Gemini·Upstage 등 외부 서비스의 API 비밀 키
  • 서비스 계정 JSON과 private key
  • Supabase service_role·secret key
  • 교사용 비밀번호와 접근 코드
  • 학생 개인정보나 인증용 비밀값

공개 Rules가 위험해지는 경우

상황판단이유
안전하게 작성한 Rules가 공개 저장소에 있음그 자체는 보통 문제 아님서버가 UID와 권한 조건을 실제로 강제함
allow read, write: if true가 공개됨매우 위험공개 여부가 아니라 규칙 자체가 누구나 접근을 허용함
교사 UID를 규칙에 직접 하드코딩권장하지 않음UID는 비밀번호가 아니지만 식별 정보 노출·교체 어려움·유지보수 문제가 생김
규칙 안에 비밀번호 문자열을 넣고 비교금지Rules는 비밀 저장소가 아니며 사용자 인증을 대신하지 못함
넓은 허용 규칙 뒤에 좁은 거부 규칙 추가위험여러 허용 규칙 중 하나라도 참이면 접근이 허용될 수 있음
공개 저장소의 장점도 있습니다. Rules를 코드와 함께 버전 관리하면 언제 어떤 권한이 바뀌었는지 검토하고 이전 버전과 비교하기 쉽습니다. 다만 Firebase Console과 CLI 양쪽에서 따로 수정하면 한쪽이 다른 쪽을 덮어쓸 수 있으므로 관리 기준을 하나로 정해야 합니다.
Rules를 숨기는 것으로 취약점을 해결할 수 없습니다. 컬렉션 이름과 경로를 어렵게 만들거나 Rules 파일을 비공개로 해도 allow true, 로그인 여부만 확인하는 넓은 권한, 사용자 수정 가능한 역할 필드는 그대로 위험합니다.
08 · SUPABASE VIEW

Supabase에서는 같은 문제를 테이블·행·RLS로 표현합니다

Supabase는 PostgreSQL 기반이므로 Firestore처럼 컬렉션과 문서를 중첩하기보다 테이블을 만들고 기본키·외래키로 관계를 연결합니다. 학생별 접근 제한의 중심은 RLS(Row Level Security)입니다.

2026년 기준 API 키 용어: 공개 클라이언트에는 sb_publishable_... 형식의 publishable key를 사용하고, 서버 전용에는 sb_secret_... 형식의 secret key를 사용합니다. 기존 anon·service_role JWT 키는 각각 publishable·secret key의 레거시 버전이며 2026년 말 폐기 예정입니다. 여기서 PostgreSQL의 anon 역할은 레거시 API 키 이름과 별개의 데이터베이스 역할입니다.
Supabase 보안은 세 층으로 이해합니다 RLS가 핵심이지만 RLS 하나만으로 모든 보안이 완성되는 것은 아닙니다.
1
Supabase AuthJWT를 통해 로그인한 사용자의 신원과 auth.uid()를 확인합니다.
2
PostgreSQL Grantsanon, authenticated 같은 역할이 어떤 테이블·함수에 접근할 수 있는지 정합니다.
3
RLS Policies허용된 테이블 안에서 어떤 행을 읽고 생성·수정·삭제할 수 있는지 제한합니다.

09 · TEST TO PRODUCTION

연수가 끝나기 전에 운영용 권한으로 전환합니다

테스트 모드는 학습을 시작하기 위한 임시 설정입니다. 만료 날짜가 있는 테스트 규칙은 시간이 지나면 자동으로 앱에 맞는 안전한 규칙으로 바뀌는 것이 아니라 조건이 거짓이 되어 요청이 막힐 수 있습니다. 실제 Rules 탭의 내용을 확인해야 합니다.

Firestore 운영 전환

  1. 실제 컬렉션·문서·필드 목록을 작성합니다.
  2. 학생·교사·비로그인 사용자의 권한표를 만듭니다.
  3. Authentication을 연결하고 UID 기준을 정합니다.
  4. 열린 테스트 규칙을 경로별 Rules로 교체합니다.
  5. 학생 A·학생 B·교사 계정으로 실패 사례까지 시험합니다.
  6. firestore.rules를 버전 관리하고 배포 방법을 기록합니다.

Supabase 운영 전환

  1. 브라우저에 노출되는 스키마와 테이블을 확인합니다.
  2. 공개 테이블의 RLS를 활성화합니다.
  3. SELECT·INSERT·UPDATE·DELETE 정책을 각각 점검합니다.
  4. Grants를 필요한 역할과 작업으로 제한합니다.
  5. 비로그인·학생·교사 계정으로 정책을 시험합니다.
  6. secret key·레거시 service_role 키가 프런트엔드에 없는지 확인합니다.

AI에게 “작동하게 만들어 달라”고만 요청하면 오류를 없애기 위해 Rules를 과도하게 열 수 있습니다. 현재 데이터 구조를 먼저 분석하고, 권한표와 실패 테스트를 제시한 뒤 코드를 만들도록 요청합니다.

[목표 권한]
1. 비로그인 사용자는 모든 학급·회원·제출 데이터에 접근할 수 없습니다.

[문서 ID 설계]
- 클래스마다 안정적인 classId를 문서 ID로 사용합니다.
- 회원 문서는 Firebase Authentication UID를 문서 ID로 사용합니다.
- 한 활동당 학생 제출이 하나라면 학생 UID를 제출 문서 ID로 사용할 수 있습니다.
- 문서 ID는 비밀번호가 아니므로 소속·역할·소유자를 Rules에서 반드시 확인합니다.

- 교사용 메모와 AI 초안이 학생이 읽는 문서에 섞여 있는 경우
- Rules 또는 공개 저장소에 비밀번호·외부 API 키·서비스 계정 정보가 있는 경우

결과를 위험도 높음·보통·낮음으로 나누고, 각 문제마다 공격 또는 실수 상황, 수정할 Rules, 수정할 JavaScript 쿼리, 테스트 방법을 함께 제시해 주세요. 단순히 화면에서 숨기는 해결책은 제안하지 마세요.
테스트 성공 사례보다 “원래 거부되어야 하는 요청이 실제로 거부되는가”를 더 중요하게 확인합니다.
CONCLUSION

좋은 Rules는 좋은 데이터 구조에서 시작합니다

한 학급을 하나의 문서로 만들고, 학생 회원과 제출물을 UID 중심의 별도 문서로 나누며, 교사용 비공개 정보를 다시 분리하면 Rules가 훨씬 명확해집니다. 그 위에서 Authentication UID, 학급 소속, 소유자, 교사 역할을 요청마다 검사해야 실제 보안이 완성됩니다.

문서 경계

읽는 사람이 다르면 문서를 나눕니다. 필드를 화면에서 숨기는 것은 보안이 아닙니다.

문서 ID

특정 ID는 정확한 경로와 Rules 비교를 돕지만, 주소 자체가 접근 권한은 아닙니다.

서버 측 정책

Firestore Rules 또는 Supabase RLS가 권한 없는 요청을 최종적으로 거부해야 합니다.

최종 기준: Rules와 데이터 경로가 공개되어도 학생 A가 학생 B의 문서와 교사용 데이터를 읽거나 변경할 수 없어야 합니다.
