01 · EXECUTIVE SUMMARY

이 구조를 한 문장으로 설명하면

한 GitHub 저장소의 정적 파일을 하나의 GitHub Pages로 배포하되, 학생과 교사에게 서로 다른 경로와 화면을 제공하고, 두 화면이 같은 Firebase 프로젝트를 사용하도록 만드는 방식입니다.

1

코드 저장

GitHub 저장소 하나에서 학생·교사·공통 코드를 함께 관리합니다.

2

화면 호스팅

GitHub Pages 한 곳에서 두 개의 경로를 제공합니다.

3

사용자 확인

Firebase Authentication이 학생과 교사에게 UID를 발급합니다.

4

데이터·권한

Firestore가 저장하고 Security Rules가 접근을 허용·거부합니다.

가장 중요한 구분: /teacher/라는 별도 화면을 만드는 것은 사용자 경험을 나누는 일입니다. 학생이 교사용 URL이나 코드를 발견해도 데이터를 읽을 수 없도록 만드는 것은 Firestore Security Rules의 일입니다.

03 · URL & ROUTING

화면을 나누는 세 가지 방법

순수 HTML인지 React·Vite 앱인지에 따라 적합한 경로 방식이 달라집니다. GitHub Pages는 일반 서버처럼 모든 경로를 자동으로 index.html로 돌려주는 rewrite 기능이 없다는 점이 중요합니다.

방식주소 예시GitHub Pages 적합성특징
폴더별 HTML/student/
/teacher/		매우 좋음각 폴더에 index.html을 둠. 초보자에게 가장 추천
파일별 HTML/student.html
/teacher.html		좋음가장 단순하지만 URL과 파일 구조가 그대로 드러남
SPA 라우팅/student
/teacher		추가 처리 필요직접 주소 접속 시 404가 생길 수 있어 HashRouter, 404 처리 또는 다른 호스팅 검토

GitHub Pages 주소에서 저장소 이름도 경로입니다

https://사용자명.github.io/저장소명/student/
                           └───────┘└───────┘
                            기본 경로   학생 화면

JavaScript와 CSS 경로를 /shared/app.js처럼 루트 기준으로 쓰면 사용자명.github.io/shared/app.js를 찾게 되어 실패할 수 있습니다. 프로젝트 Pages에서는 ../shared/app.js 같은 상대 경로나 빌드 도구의 base 설정을 정확히 사용해야 합니다.

React Router를 사용한다면: GitHub Pages에서 브라우저 주소가 깔끔한 History Router는 새로고침 시 404가 될 수 있습니다. 간단하게는 HashRouter를 사용하고, 깔끔한 경로와 rewrite가 중요하면 Firebase Hosting이나 Vercel 같은 호스팅을 검토합니다.
04 · BENEFITS & LIMITATIONS

두 GitHub Pages로 나눌 때와 비교한 장단점

주요 장점

  • 배포가 한 번입니다. 한 저장소에 push하면 학생과 교사 화면이 함께 반영됩니다.
  • 공통 코드 관리가 쉽습니다. Firebase 설정, 로그인 처리, 데이터 필드명을 공유할 수 있습니다.
  • 버전 불일치가 줄어듭니다. 학생이 새 필드를 보내는데 교사 화면이 모르는 문제를 줄일 수 있습니다.
  • 도메인 설정이 단순합니다. 학생과 교사 화면이 같은 GitHub Pages 도메인을 사용합니다.
  • 로그인 흐름이 자연스럽습니다. 같은 출처의 경로이므로 화면 이동과 인증 상태 관리가 비교적 간단합니다.
  • 운영 비용이 낮습니다. GitHub Pages와 Firebase의 소규모 무료 범위부터 시작할 수 있습니다.

주요 단점

  • 두 화면이 함께 배포됩니다. 교사 화면만 수정해도 전체 Pages 배포가 다시 진행됩니다.
  • 한 오류가 공통 기능에 영향을 줄 수 있습니다. 공유 인증 코드가 깨지면 학생과 교사 모두 영향을 받습니다.
  • 교사 코드가 같은 사이트에 있습니다. 학생이 파일과 경로를 찾을 수 있으므로 코드 은닉에 의존할 수 없습니다.
  • 독립 운영에는 불리합니다. 학생 앱과 교사 앱의 담당자·배포 주기가 완전히 다르면 저장소 분리가 나을 수 있습니다.
  • 서버 기능은 없습니다. 비밀 API 키, Admin SDK, 일괄 관리자 작업에는 별도 서버 측 기능이 필요합니다.
  • SPA 경로에 제약이 있습니다. 직접 접속과 새로고침을 별도로 처리해야 할 수 있습니다.

구조별 빠른 비교

구조배포 관리공통 코드독립성추천 상황
Pages 1개 + Firebase가장 단순공유하기 쉬움중간한 교사가 운영하는 소규모·중간 규모 수업 앱
Pages 2개 + Firebase두 곳 관리중복 가능성높음담당자와 배포 주기가 완전히 다른 두 앱
Firebase Hosting + FirebaseFirebase 중심공유하기 쉬움중간SPA rewrite, 미리보기, Functions 연동이 중요한 앱

05 · SECURITY MODEL

한 사이트여도 보안 원칙은 동일합니다

교사 화면에 들어가지 못하게 리디렉션하는 코드는 사용 편의를 위한 1차 처리입니다. 개발자 도구와 직접 요청을 막는 실제 통제는 Authentication과 Security Rules에서 이루어져야 합니다.

1
Firebase Authentication

학생과 교사가 학교 Google 계정 또는 개별 계정으로 로그인하고 고유 UID를 받습니다.

2
교사 역할 판정

소규모 앱은 교사 UID 허용 목록, 확장형 앱은 Admin SDK로 부여한 Custom Claims 등을 사용할 수 있습니다.

3
Firestore Security Rules

학생은 자기 제출물만, 교사는 담당 범위만 읽고 수정하도록 데이터베이스에서 거부합니다.

4
입력값과 변경 필드 검증

학생이 ownerUid, role, teacherFeedback 같은 관리 필드를 바꾸지 못하게 합니다.

5
App Check와 사용량 관찰

자동화된 남용을 줄이고 Firestore 읽기·쓰기 사용량, 오류와 예산을 지속적으로 확인합니다.

6
개인정보 최소화

학습에 필요한 데이터만 저장하고 접근자, 보존 기간, 삭제 방법과 학교 지침을 함께 정합니다.

교사용 화면에서 권장되는 흐름

  1. 로그인 상태 확인
    로그인하지 않았다면 로그인 화면을 표시합니다.
  2. 교사 역할 확인
    교사가 아니면 학생 화면이나 접근 거부 안내로 이동시킵니다.
  3. 교사용 쿼리 실행
    권한이 확인된 뒤 필요한 학급과 활동 범위만 조회합니다.
  4. Rules에서 다시 확인
    클라이언트 검사가 조작되더라도 Firestore가 최종적으로 요청을 거부합니다.
화면 리디렉션만으로는 보호되지 않습니다. 학생이 /teacher/에서 실행되는 함수나 Firestore 쿼리를 복사해 학생 화면 또는 콘솔에서 실행할 수 있습니다. 이때 permission-denied가 나와야 안전합니다.

Firebase 웹 설정과 비밀 키의 차이

정보GitHub Pages 포함 여부보호 방법
Firebase apiKey·projectId웹 앱에 포함되는 공개 설정Auth·Rules·App Check·사용량 관리로 데이터 보호
Gemini·OpenAI 비밀 API 키절대 포함하면 안 됨Apps Script, Cloud Functions 등 서버 측 환경 변수에서 사용

Firebase Admin SDK 서비스 계정절대 포함하면 안 됨신뢰할 수 있는 서버 환경에서만 보관
06 · FIRESTORE RULES

학생과 교사의 데이터 권한 예시

다음은 원리를 보여 주는 예시입니다. 학생은 자기 응답만 수정하고, 교사는 전체를 읽고 피드백만 수정할 수 있도록 변경 필드를 구분합니다.

예시 데이터

submissions/{submissionId}
  ownerUid         // 학생 Firebase UID
  classId          // 학급 ID
  activityId       // 활동 ID
  answer            // 학생 답변
  updatedAt         // 학생 수정 시각
  teacherFeedback   // 교사 피드백
  feedbackAt        // 피드백 시각

교사 1명인 소규모 앱의 개념 예시

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() {
      return request.auth != null;
    }

    function isTeacher() {
      return signedIn()
        && request.auth.uid == "교사의_FIREBASE_UID";
    }

    function isOwner() {
      return signedIn()
        && resource.data.ownerUid == request.auth.uid;
    }

    match /submissions/{submissionId} {
      // 학생은 자기 문서, 교사는 모든 제출물 읽기
      allow read: if isOwner() || isTeacher();

      // 학생은 자기 UID가 들어간 문서만 생성
      allow create: if signedIn()
        && request.resource.data.ownerUid == request.auth.uid;

      // 학생은 답변과 수정 시각만 변경
      allow update: if isOwner()
        && request.resource.data.diff(resource.data)
             .affectedKeys()
             .hasOnly(["answer", "updatedAt"]);

      // 교사는 피드백 관련 필드만 변경
      allow update: if isTeacher()
        && request.resource.data.diff(resource.data)
             .affectedKeys()
             .hasOnly(["teacherFeedback", "feedbackAt"]);

      allow delete: if isTeacher();
    }
  }
}
완성형 Rules는 아닙니다. 실제 앱에서는 생성할 때 허용할 필드 목록, 문자열 길이와 자료형, 유효한 학급·활동인지, 제출 기한, 교사의 담당 학급 범위까지 추가해야 합니다. 여러 교사라면 UID 하나를 직접 쓰기보다 Custom Claims 또는 안전하게 관리되는 역할·학급 문서를 설계합니다.

Rules는 필터가 아닙니다

Firestore는 전체 문서를 먼저 가져온 뒤 허용된 것만 남겨 주지 않습니다. 쿼리가 반환할 수 있는 범위 전체가 Rules에 허용되어야 합니다. 따라서 교사가 담당 학급만 조회하게 한다면 classId 조건이 들어간 쿼리와 담당 학급 권한 구조를 함께 설계해야 합니다.

07 · BUILD & OPERATE

권장 구축 순서와 운영 방법

  1. 권한표부터 작성
    비로그인, 학생, 교사가 각 컬렉션에서 무엇을 읽고 만들고 수정할 수 있는지 표로 정합니다.
  2. 저장소와 폴더 구성
    student/, teacher/, shared/를 만들고 공통 데이터 규격을 정의합니다.
  3. Firebase Authentication 연결
    학교 계정 환경에 맞는 로그인 방식을 설정하고 GitHub Pages 도메인을 승인 목록에서 확인합니다.
  4. Firestore 구조와 Rules 작성
    화면보다 먼저 학생 A·학생 B·교사의 허용·거부 사례를 설계합니다.
  5. 학생 화면 구현
    학생이 자기 문서만 제출·조회·수정할 수 있는지 시험합니다.
  6. 교사 대시보드 구현
    교사 역할 확인 후 필요한 학급·활동만 조회하고 피드백을 저장합니다.
  7. GitHub Pages 배포
    Settings → Pages에서 대상 브랜치와 폴더를 지정합니다. 단순 정적 파일은 push 후 기본 배포로 반영됩니다.
  8. 실제 공격 관점 테스트
    학생 계정으로 교사 쿼리 실행, 다른 학생 UID 접근, 관리 필드 변경을 시도하고 모두 거부되는지 확인합니다.
  9. App Check·사용량·삭제 정책 보완
    기본 권한이 안정된 뒤 남용 방지와 운영 정책을 강화합니다.

GitHub Actions가 필요한 경우

프로젝트 형태별도 Actions 필요성이유
순수 HTML·CSS·JavaScript대체로 필요 없음지정 브랜치의 완성 파일을 GitHub Pages가 기본 방식으로 배포
React·Vite 빌드 결과를 저장소에 포함선택 사항로컬에서 dist를 만들어 직접 올릴 수 있으나 반복 작업이 생김
React·Vite 소스만 저장유용함Actions가 설치·빌드·Pages 배포를 자동화
테스트 성공 후만 배포유용함검사와 빌드를 통과한 코드만 공개 가능

비용과 성능 관리

  • 교사 대시보드에서 전체 학년 데이터를 매번 읽지 말고 학급·활동·기간별로 조회합니다.
  • 실시간 리스너는 필요한 화면에서만 사용하고 화면을 떠날 때 해제합니다.
  • 목록은 페이지 단위로 불러오고, 큰 이미지나 Base64를 Firestore 문서에 직접 저장하지 않습니다.
  • 파일은 Cloud Storage에, 파일 경로와 설명은 Firestore에 저장합니다.
  • Firestore 사용량과 결제 설정을 확인하고 필요한 경우 예산 알림을 구성합니다.
08 · RELEASE CHECK

배포 전 최종 점검표

구조·기능 점검

  • /student/와 /teacher/를 직접 열어도 정상 작동한다.
  • 저장소 이름이 포함된 GitHub Pages 기본 경로를 반영했다.
  • 공통 Firebase 설정과 데이터 필드가 한곳에서 관리된다.
  • 학생과 교사 화면의 변경 사항을 함께 시험한 뒤 push한다.
  • React Router를 사용한다면 새로고침과 직접 URL 접속을 확인했다.
  • 모바일 화면과 교사용 데스크톱 화면을 각각 시험했다.

보안 점검

  • Firestore 테스트 모드의 공개 Rules를 제거했다.
  • 학생 A가 학생 B의 문서를 읽거나 수정할 수 없다.
  • 학생이 교사 피드백과 역할 필드를 변경할 수 없다.
  • 학생 계정으로 교사 쿼리를 실행하면 거부된다.
  • 비밀 API 키와 Admin SDK 자격 증명이 정적 코드에 없다.
  • Rules Playground 또는 Emulator로 권한별 사례를 확인했다.

절대 보안으로 착각하면 안 되는 것

교사용 URL 비공개

주소는 언젠가 알려질 수 있습니다. URL 비공개는 데이터 권한 통제가 아닙니다.

hidden 처리

개발자 도구로 표시할 수 있습니다. 화면 숨김은 사용자 경험 처리일 뿐입니다.

클라이언트의 isTeacher

브라우저 코드는 수정할 수 있습니다. Rules가 동일한 조건을 독립적으로 확인해야 합니다.

09 · RECOMMENDATION

이 상황에서의 최종 평가

한 명 또는 소수의 교사가 하나의 수업 앱을 운영하면서 학생 화면과 교사 화면 모두를 직접 관리한다면, GitHub Pages 한 곳에 두 경로를 두고 Firebase를 사용하는 방식이 두 사이트를 별도로 배포하는 것보다 일반적으로 단순하고 안정적입니다.

이 방법을 추천

  • 학생·교사 화면이 같은 데이터 구조와 디자인을 공유함
  • 배포 담당자와 수정 시점이 거의 같음
  • 실시간 대시보드와 학생별 권한이 필요함
  • Auth와 Rules를 테스트할 수 있음

다른 방법을 검토

  • 두 앱의 담당자·배포 주기가 완전히 다름 → Pages 2개
  • SPA rewrite와 서버 기능이 중요함 → Firebase Hosting 등
  • 교사에게 Google Sheets가 중심임 → Apps Script + Sheets
  • 한 번 제출하는 단순 활동임 → Google Forms + Sheets
FINAL · METHOD SELECTOR

이 방법이 최선인지 확인하는 선택 체크리스트

아래 항목에 답하면 다섯 가지 구조의 적합도를 비교합니다. 최소 8개 이상 선택해야 결과가 표시됩니다.

1. 학생별 로그인과 자기 데이터만 접근하는 기능이 필요한가요?개별 UID와 세밀한 데이터 권한의 필요성을 확인합니다.
2. 실시간 제출 현황과 교사 피드백이 중요한가요?Firestore의 실시간 기능이 실제로 필요한지 확인합니다.
3. 학생·교사 화면이 같은 담당자, 디자인, 데이터 구조를 공유하나요?한 저장소로 관리할 때의 이점이 큰지 확인합니다.
4. 학생·교사 앱을 서로 다른 일정과 담당자로 독립 배포해야 하나요?두 GitHub Pages로 나눌 실질적인 이유가 있는지 확인합니다.
5. React SPA의 깔끔한 URL, rewrite 또는 서버 함수 연결이 중요한가요?GitHub Pages보다 Firebase Hosting이 편리한 기능인지 확인합니다.
6. 교사가 Google Sheets에서 직접 자료를 관리하는 것이 중요한가요?시트가 주된 관리자 화면인지 확인합니다.
7. 활동이 한 번 제출하고 끝나는 단순 설문에 가까운가요?맞춤형 웹 앱이 실제로 필요한지 확인합니다.
8. Authentication과 Firestore Rules를 작성하고 사용자별로 테스트할 수 있나요?Firebase 방식의 핵심 보안 작업을 수행할 수 있는지 확인합니다.
9. Gemini·OpenAI 등 비밀 API 키가 필요한 기능이 있나요?정적 Pages만으로 처리할 수 없어 서버 측 기능이 필요합니다.
10. 향후 파일, 통계, 여러 학급과 여러 교사로 확장할 가능성이 큰가요?장기적인 기능 확장성을 확인합니다.
11. 성적·상담·건강정보처럼 민감도가 높은 정보를 다루나요?어떤 도구든 학교 정책과 개인정보 보호 검토가 별도로 필요합니다.

점수보다 우선할 조건: Firebase를 선택해도 학생·교사 화면을 굳이 두 저장소로 나눌 이유가 없다면 한 GitHub Pages가 더 단순합니다. 반대로 비밀 API 키, 복잡한 관리자 기능, SPA rewrite가 핵심이라면 정적 GitHub Pages만으로 완결하려 하지 말고 서버 측 기능이나 다른 호스팅을 함께 검토하십시오.
