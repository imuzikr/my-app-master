01 · FIRST DISTINCTION

“비밀번호를 사용한다”는 말부터 구분해야 합니다

겉으로는 모두 비밀번호 입력창처럼 보이지만, 비밀번호를 어디에서 검사하고 로그인 후 누구에게 어떤 UID를 주는지에 따라 보안 수준이 완전히 달라집니다.

위험

브라우저가 공통 비밀번호 비교

password === "teacher1234"처럼 JavaScript에서 확인하고 화면만 엽니다. 소스코드와 실행 흐름을 사용자가 볼 수 있으므로 인증으로 인정할 수 없습니다.

제한적

학생·교사가 계정을 하나씩 공유

Firebase Auth를 사용해도 학생 전체가 같은 계정, 교사 전체가 같은 계정을 쓰면 개인별 UID와 책임 추적이 사라집니다.

권장

사용자별 Firebase 계정

각 학생과 교사가 자기 이메일·비밀번호로 Firebase Auth에 로그인하고 고유 UID를 받습니다. Rules에서 이 UID와 역할을 확인합니다.

중요: OAuth를 사용하지 않는 것과 Firebase Authentication을 사용하지 않는 것은 다릅니다. Google 로그인 없이도 Firebase의 이메일·비밀번호 인증을 사용할 수 있으며, 이것이 비밀번호만 사용하는 구조에서 가장 현실적인 기본 선택입니다.
02 · FIVE PASSWORD MODELS

비밀번호 기반 구조 5가지 비교

교사·학생 페이지를 GitHub Pages 한 곳 또는 두 곳으로 배포하는지는 아래 인증 방식의 안전성을 바꾸지 않습니다. 핵심은 Firebase가 사용자를 개별적으로 인증하고 Rules가 요청자의 UID와 역할을 확인하는가입니다.

보안 매우 낮음

A. JavaScript에 공통 비밀번호 하드코딩

입력한 비밀번호를 브라우저 코드에서 비교해 학생 또는 교사 화면을 표시합니다.

  • 비밀번호와 우회 조건을 소스에서 찾을 수 있음
  • 화면을 열어도 Firebase에는 인증된 UID가 없음
  • 데이터를 읽으려면 Rules를 공개하는 잘못된 구성이 되기 쉬움
난도 낮음실제 서비스 금지민감 데이터 금지
권장하지 않음

B. 학생용·교사용 Firebase 계정을 하나씩 공유

예를 들어 모든 학생이 student@class.test 계정 하나를 같이 사용합니다.

  • 비밀번호 검사는 Firebase가 하므로 A보다는 나음
  • 모든 학생의 UID가 같아 개인별 소유권 분리 불가능
  • 한 명에게 유출되면 전체 비밀번호를 교체해야 함
  • 누가 어떤 작업을 했는지 구분하기 어려움
난도 낮음개인별 데이터 부적합교사 계정 공유 위험
기본 권장

C. 사용자별 Firebase 이메일·비밀번호 계정

각 학생과 교사가 별도 계정으로 로그인하고, Firebase가 고유 UID와 인증 토큰을 발급합니다.

  • OAuth 팝업이나 Google 계정이 필요 없음
  • 학생은 자기 UID의 문서만 접근 가능
  • 비밀번호 정책과 재설정 이메일 사용 가능
  • 교사 역할은 UID 또는 Custom Claims로 별도 부여
난도 중간개인별 권한 가능대부분의 수업 앱
고급 선택

D. 학번·비밀번호 + 자체 인증 서버

서버가 학번과 비밀번호를 검증한 뒤 Firebase Custom Token을 발급합니다.

  • 이메일 없이 학번 형태의 로그인 가능
  • 비밀번호 저장·해시·재설정·공격 방어를 직접 설계
  • Admin SDK와 서비스 계정은 서버에만 보관
  • 정적 GitHub Pages만으로는 구현할 수 없음
난도 높음서버 필수전문 운영 필요
조건부

E. 수업 참여 코드 + 익명 계정

학생이 공통 수업 코드를 입력하되 Firebase에서는 각 브라우저에 별도 익명 UID를 발급합니다.

  • 참여 코드는 신원 인증이 아니라 수업 입장 코드임
  • 기기·브라우저가 바뀌면 같은 학생을 찾기 어려움
  • 코드를 안전하게 검증하려면 서버 기능이 필요할 수 있음
  • 개인 식별이 필요 없는 저위험 단기 활동에 한정
난도 중간일회성 활동개인 자료 부적합
방식학생별 UID서버 필요개인별 자료교사 대시보드종합 판단
A. 클라이언트 공통 암호없음없음불가안전하지 않음사용 금지
B. 역할별 공유 계정역할별 하나없음사실상 불가책임 추적 불가권장하지 않음
C. 개별 Email/Password사용자별기본 기능은 불필요가능가능기본 권장
D. 자체 학번 로그인사용자별필수가능가능고급·조직형

03 · RECOMMENDED ARCHITECTURE

권장 구조: 개별 비밀번호는 Firebase Auth가 검사

GitHub Pages는 로그인 입력 화면만 제공합니다. 브라우저 코드가 정답 비밀번호를 가지고 있지 않으며, Firebase Authentication에 로그인 요청을 보내고 성공하면 UID와 인증 토큰을 받습니다.

GitHub Pages학생·교사 로그인 화면
이메일과 비밀번호 입력
→
Firebase Authentication비밀번호 검사
UID·ID Token 발급
→
Security Rules → FirestoreUID·교사 역할 확인
허용된 데이터만 접근

로그인 코드의 핵심

import {
  getAuth,
  signInWithEmailAndPassword
} from "firebase/auth";

const auth = getAuth();

async function login(email, password) {
  const credential = await signInWithEmailAndPassword(
    auth,
    email,
    password
  );

  // 로그인 성공 후 얻는 Firebase 사용자 UID
  console.log(credential.user.uid);
}
중요한 차이: 이 코드 안에는 정답 비밀번호가 없습니다. 사용자가 입력한 비밀번호는 Firebase Authentication이 검사하며, 앱은 성공 또는 실패 결과와 인증된 사용자의 UID를 받습니다.

페이지 분리와의 관계

GitHub Pages 한 곳

/student/와 /teacher/에서 같은 Firebase Auth를 사용합니다. 코드와 배포를 함께 관리하기 쉽습니다.

GitHub Pages 두 곳

학생 사이트와 교사 사이트 모두 같은 Firebase 프로젝트에 로그인할 수 있습니다. 두 도메인의 Auth 설정과 코드 버전을 함께 관리해야 합니다.

어느 방식이든 학생 계정으로 교사 페이지 주소에 접속하는 것은 가능할 수 있습니다. 그러나 교사용 데이터를 읽는 요청은 Rules에서 거부되어야 합니다.

04 · SHARED PASSWORD RISK

학생용·교사용 공통 비밀번호가 위험한 이유

가장 위험한 구현

비밀번호 입력예: teacher1234
→
브라우저 JavaScript 비교소스 확인·우회 가능
→
교사 화면 표시인증된 교사 UID는 없음
// ❌ 보안 인증이 아닙니다.
if (password === "teacher1234") {
  document.querySelector("#teacher-dashboard")
    .classList.remove("hidden");
}

코드에서 발견 가능

소스 보기, 개발자 도구, 내려받은 JavaScript, 빌드 결과에서 비교 값이나 우회 지점을 찾을 수 있습니다.

화면만 열릴 뿐

브라우저에서 화면을 표시해도 Firestore 입장에서는 인증된 교사 계정이 아닙니다.

Rules를 약화시키기 쉬움

인증 UID가 없으니 개발자가 데이터를 읽기 위해 공개 Rules를 설정하는 심각한 실수로 이어집니다.

해시·난독화·환경 변수로 숨기면 괜찮을까요?

시도왜 해결되지 않는가
비밀번호를 SHA-256으로 해시브라우저에 비교할 해시와 비교 로직이 남습니다. 공격자는 원문을 몰라도 동일한 검사를 통과하도록 코드를 우회할 수 있습니다.
JavaScript 난독화·압축분석을 잠시 어렵게 할 뿐이며, 실행되는 코드와 네트워크 요청은 사용자 브라우저에 전달됩니다.
GitHub 저장소를 비공개GitHub Pages가 전달한 HTML·JavaScript는 방문자의 브라우저에 내려오므로 실행 코드를 확인할 수 있습니다.
빌드 환경 변수에 저장클라이언트 번들에 포함되는 순간 브라우저에서 추출할 수 있습니다.
localStorage에 로그인 결과 저장사용자가 직접 값을 만들거나 바꿀 수 있습니다. 서버가 서명한 인증 토큰이 아닙니다.

공유 Firebase 계정도 근본 문제는 남습니다

Firebase Auth에서 학생 공통 계정으로 실제 로그인하면 비밀번호가 코드에 노출되는 문제는 줄어듭니다. 그러나 학생 30명이 모두 같은 UID를 가지므로 Rules는 어느 학생인지 구별할 수 없습니다.

// 모든 학생이 같은 공유 계정으로 로그인했다면
request.auth.uid == "shared-student-uid"

// 학번 입력값은 학생이 바꿔 보낼 수 있으므로
// 이것만으로 문서 소유자를 신뢰할 수 없습니다.
request.resource.data.studentNumber == "10203"
교사 공통 계정은 특히 피해야 합니다. 유출 시 모든 학생 기록을 읽거나 수정할 수 있고, 누가 접근했는지 구분하기 어려우며, 한 교사만 퇴직·변경해도 전체 비밀번호를 바꿔야 합니다.

공통 비밀번호를 사용할 수 있는 제한적 범위

공통 문자열은 “보안 인증”이 아니라 “수업 참여 코드” 정도로만 사용할 수 있습니다. 개인 식별이나 민감한 자료가 없고, 코드가 알려져도 피해가 작으며, 각 사용자는 별도 익명 UID를 갖는 일회성 활동에 한정해야 합니다. 교사 권한에는 사용하지 않는 것이 안전합니다.

05 · ACCOUNT OPERATIONS

개별 이메일·비밀번호 계정을 어떻게 운영할까요?

OAuth를 없애면 로그인 화면은 단순해지지만, 계정 생성·초기 비밀번호 전달·분실 처리·학기 종료 비활성화 같은 운영 부담은 교사 또는 관리자가 맡게 됩니다.

계정 생성 방식 비교

방식장점위험·불편추천 상황
교사가 Firebase Console에서 미리 생성허가된 학생만 계정 보유, 공개 회원가입 불필요학생 수가 많으면 반복 작업, 초기 비밀번호 전달 필요한두 학급의 소규모 운영
Admin SDK로 일괄 생성명단 기반 자동 생성·비활성화 가능신뢰할 수 있는 서버 또는 관리자 스크립트 필요여러 학급·반복 운영
학생이 직접 회원가입교사의 생성 작업 감소외부인·중복·허위 계정 방지와 이메일 검증 필요학생이 실제 이메일을 보유하고 검증 가능할 때
가상 이메일 형식 계정실제 이메일 없이 Email/Password 사용 가능재설정 이메일을 받을 수 없어 분실 처리를 교사가 담당관리자가 계정 수명주기를 직접 운영할 수 있을 때

학생에게 실제 이메일이 없으면

Firebase의 기본 Password Provider는 이메일 주소를 계정 식별자로 사용합니다. 학번만 입력하게 만들고 싶다면 다음 두 선택지가 있습니다.

내부적으로 가상 이메일로 변환

화면에는 학번만 받고 앱이 10203@class.example 같은 이메일 형식으로 변환할 수 있습니다. 구현은 쉽지만 메일 기반 본인 확인과 비밀번호 재설정을 사용할 수 없습니다.

자체 인증 서버 + Custom Token

서버가 학번·비밀번호를 확인하고 고유 UID의 Firebase Custom Token을 발급합니다. 올바른 구조지만 비밀번호 저장과 서버 보안까지 직접 책임져야 합니다.

가상 이메일은 기술적 우회일 뿐 완전한 계정 체계가 아닙니다. 비밀번호 분실, 학생 전학·졸업, 계정 재사용, 보호자 요청, 관리자 변경까지 포함한 운영 절차를 먼저 정해야 합니다.

필수 계정 수명주기

  1. 계정 발급
    허가된 명단에서 사용자별 계정을 만들고 임시 비밀번호를 서로 노출되지 않게 전달합니다.
  2. 첫 로그인과 비밀번호 정책
    충분한 길이와 복잡도를 요구하고 초기 비밀번호를 계속 공유하지 않도록 운영합니다.
  3. 분실·잠금 처리
    실제 이메일이면 재설정 메일을, 가상 이메일이면 관리자 확인 절차를 사용합니다.
  4. 권한 변경
    학생을 교사로 바꾸는 기능을 클라이언트에 두지 않고 Admin SDK 또는 관리자가 처리합니다.
  5. 학기 종료
    필요한 데이터를 보존·삭제하고 사용하지 않는 계정을 비활성화하거나 제거합니다.
  6. 유출 대응
    비밀번호 변경, 세션 폐기, 피해 범위 확인과 Rules·로그 점검 절차를 준비합니다.
06 · SECURITY DESIGN

비밀번호 방식의 보안 설계

1
사용자별 계정

학생과 교사가 서로 다른 Firebase UID를 갖도록 합니다. 역할별 공유 계정을 사용하지 않습니다.

2
비밀번호 정책

Firebase Console에서 최소 길이와 문자 조합을 정하고, 추측하기 쉬운 공통 초기 비밀번호를 피합니다.

3
교사 역할의 안전한 부여

교사 1명은 Rules의 UID 허용 목록, 여러 교사는 Admin SDK의 Custom Claims 또는 관리자만 쓰는 역할 구조를 사용합니다.

4
Firestore Security Rules

학생은 자기 UID의 문서만, 교사는 담당 학급만 접근하도록 읽기·쓰기·변경 필드를 제한합니다.

5
세션과 공용 기기

수업 후 명확한 로그아웃 버튼을 제공하고 공용 기기에서는 로그인 지속 방식을 신중히 선택합니다.

6
남용 방지와 관찰

App Check, 가입 제한, 일반화된 로그인 오류, 사용량·실패 로그와 비용 알림으로 자동 공격과 남용을 줄입니다.

비밀번호는 Firestore에 저장하지 않습니다

Firebase Email/Password 사용 시

앱이 비밀번호 목록을 Firestore에 만들 필요가 없습니다. Firebase Authentication이 비밀번호 기반 계정을 관리합니다.

자체 인증 서버 사용 시

평문이나 단순 SHA-256으로 저장하지 않습니다. 검증된 비밀번호 해시 방식, 고유 salt, 재설정 토큰, 속도 제한과 서버 보안이 모두 필요합니다.

교사 역할을 정하는 안전성 순서

방법안전성적합한 범위주의점
Rules에 교사 UID 직접 지정높음교사 1명 또는 소수교사 변경 시 Rules 수정·재배포
Admin SDK Custom Claims높음여러 교사·확장형 앱서버 또는 안전한 관리자 작업 필요
관리자만 쓰는 역할 문서설계에 따라 높음학급·역할이 자주 바뀌는 앱학생이 역할 문서를 만들거나 수정하지 못하게 해야 함
이메일 문자열에 teacher 포함낮음권장하지 않음가입 통제가 없다면 교사처럼 보이는 주소 생성 가능
클라이언트의 role = "teacher"매우 낮음사용 금지브라우저에서 값과 코드를 조작할 수 있음
비밀번호 정책만으로 충분하지 않습니다. 강한 비밀번호를 사용해도 모든 학생이 같은 계정을 쓰거나 Rules가 열려 있으면 학생별 데이터는 보호되지 않습니다. 인증, 역할, Rules, 세션 운영이 함께 맞아야 합니다.
07 · RULES & ROLE EXAMPLE

개별 계정과 교사 역할을 적용한 Rules 예시

아래 예시는 학생은 자기 제출물의 답변만, 교사는 피드백 필드만 수정한다는 원리를 보여 줍니다.

소규모 수업: 교사 UID 직접 지정

rules_version = '2';
service cloud.firestore {
  match /databases/{database}/documents {

    function signedIn() {
      return request.auth != null;
    }

    function isTeacher() {
      return signedIn()
        && request.auth.uid == "교사의_FIREBASE_UID";
    }

    function ownsExistingDocument() {
      return signedIn()
        && resource.data.ownerUid == request.auth.uid;
    }

    match /submissions/{submissionId} {
      allow read: if ownsExistingDocument() || isTeacher();

      allow create: if signedIn()
        && request.resource.data.ownerUid == request.auth.uid;

      allow update: if ownsExistingDocument()
        && request.resource.data.diff(resource.data)
             .affectedKeys()
             .hasOnly(["answer", "updatedAt"]);

      allow update: if isTeacher()
        && request.resource.data.diff(resource.data)
             .affectedKeys()
             .hasOnly(["teacherFeedback", "feedbackAt"]);

      allow delete: if isTeacher();
    }
  }
}

확장형: Custom Claims의 교사 역할

function isTeacher() {
  return request.auth != null
    && request.auth.token.teacher == true;
}

teacher: true는 교사 브라우저가 직접 쓰는 값이 아닙니다. Firebase Admin SDK가 신뢰할 수 있는 환경에서 사용자 계정에 부여한 claim이어야 합니다.

예시 Rules를 그대로 배포하지 마십시오. 실제 앱에서는 생성 가능한 필드, 문자열 길이와 자료형, 학급 구성원 여부, 유효한 활동 ID, 제출 기한, 교사의 담당 학급 범위를 추가로 검사해야 합니다.

08 · IMPLEMENTATION PATH

초보자에게 권장하는 구축 순서

  1. 인증 방식을 확정
    개인별 Email/Password를 기본으로 선택하고, 공통 비밀번호는 제외합니다.
  2. 계정 발급과 복구 정책 작성
    실제 이메일 사용 여부, 초기 비밀번호 전달, 분실 처리, 학기 종료 절차를 정합니다.
  3. Firebase Auth에서 Email/Password 활성화
    비밀번호 정책을 설정하고 필요하지 않다면 공개 회원가입 화면을 만들지 않습니다.
  4. 교사 계정을 먼저 생성
    교사 UID를 확인하고 Rules 허용 목록 또는 Custom Claims로 역할을 부여합니다.
  5. 학생별 계정 생성
    학생마다 서로 다른 UID가 생기는지 확인합니다. 초기 비밀번호를 공통으로 오래 사용하지 않습니다.
  6. Firestore 데이터 구조와 Rules 작성
    학생 A, 학생 B, 교사, 비로그인 사용자별 허용·거부 표를 Rules로 옮깁니다.
  7. 로그인 화면 연결
    signInWithEmailAndPassword()를 사용하고 로그인 성공 후 UID와 역할에 맞는 화면으로 안내합니다.
  8. 우회 공격 테스트
    학생 계정으로 교사 페이지 열기, 다른 UID 조회, 피드백 필드 수정, 직접 Firestore 요청을 시도합니다.
  9. 로그아웃·재설정·비활성화 시험
    공용 기기와 분실 상황까지 실제 운영 흐름을 확인합니다.
  10. App Check와 사용량 감시 보완
    기본 권한이 안정된 후 자동 남용 방지와 비용 관찰을 강화합니다.

09 · RELEASE CHECKLIST

배포 전 비밀번호 인증 점검표

반드시 확인

  • 학생과 교사가 각자 다른 Firebase 계정과 UID를 갖는다.
  • 정답 비밀번호나 비밀번호 해시가 GitHub Pages 코드에 없다.
  • 평문 비밀번호를 Firestore에 저장하지 않는다.
  • 교사 역할은 UID 또는 신뢰 가능한 Custom Claims로 판정한다.
  • 학생이 자신의 역할 문서를 만들거나 수정할 수 없다.
  • 학생 A가 학생 B의 자료를 읽거나 수정할 수 없다.
  • 학생이 교사 피드백과 관리 필드를 바꿀 수 없다.
  • 비밀번호 분실·학기 종료·계정 비활성화 절차가 있다.
  • 공용 기기에서 로그아웃과 세션 지속 방식을 확인했다.
  • Rules Playground 또는 Emulator로 권한별 사례를 시험했다.

피해야 할 구성

  • 교사 비밀번호를 JavaScript 상수로 저장
  • 학생 전체가 하나의 Firebase 계정을 공유
  • 교사 전체가 하나의 관리자 계정을 공유
  • localStorage의 역할 값을 신뢰
  • 학번 입력값만으로 문서 소유자를 판정
  • 로그인 성공 후 Rules 없이 화면만 숨김
  • 학생이 회원가입하며 자신의 역할을 선택
  • 서비스 계정 JSON을 GitHub 저장소에 업로드
  • 실패 횟수 제한·사용량 감시 없이 공개 로그인 운영
  • 교사 권한 유출 시 대응 계획이 없음
가장 중요한 실전 테스트: 학생 계정으로 로그인한 뒤 교사용 페이지와 함수를 강제로 실행해 보십시오. 화면이 잠시 보이는지가 아니라 Firestore 요청이 permission-denied로 거부되는지가 핵심입니다.
10 · FINAL RECOMMENDATION

최종 판단

OAuth를 사용하지 않는 조건이라면, 대부분의 수업용 앱에서 가장 균형이 좋은 방법은 학생과 교사에게 개별 Firebase 이메일·비밀번호 계정을 발급하고, 교사 역할과 데이터 접근은 Security Rules로 판정하는 것입니다.

현실적인 권장안

  • 개인별 Email/Password 계정
  • 교사 1명은 UID 허용 목록
  • 학생은 자기 UID 자료만 접근
  • 초기에는 공개 회원가입 없이 관리자 발급
  • 명확한 로그아웃과 계정 종료 절차

사용하지 말아야 할 안

  • 교사용·학생용 공통 비밀번호를 코드에 입력
  • 공유 계정 하나로 모든 학생을 로그인
  • 화면 잠금만 하고 Firestore Rules를 공개
  • 학번 필드를 믿고 학생 소유권 판정
  • Admin 자격 증명을 정적 코드에 포함
FINAL · AUTH METHOD SELECTOR

어떤 인증 방식이 적합한지 확인하는 체크리스트

아래 질문에 답하면 네 가지 현실적인 대안을 비교합니다. 공통 교사 비밀번호는 보안상 추천 후보에 포함하지 않았습니다.

1. 학생별로 자기 자료만 조회·수정해야 하나요?사용자별 UID가 반드시 필요한지 확인합니다.
2. 학생이 실제로 사용할 수 있는 개별 이메일 주소를 가지고 있나요?Firebase 비밀번호 재설정과 이메일 확인을 활용할 수 있는지 확인합니다.
3. 이메일 대신 학번과 비밀번호만으로 로그인해야 하나요?단순 화면 표시가 아니라 실제 계정 식별 방식의 요구사항입니다.
4. 안전한 인증 서버와 Admin SDK를 운영할 개발 지원이 있나요?자체 학번 로그인을 올바르게 구현할 역량이 있는지 확인합니다.
5. 계정 생성·비밀번호 분실·학기 종료를 교사가 관리할 수 있나요?OAuth를 사용하지 않을 때 늘어나는 운영 부담을 감당할 수 있는지 확인합니다.
6. 학교 Google 계정을 이미 모든 학생이 사용하고 있나요?OAuth를 제외하려는 결정을 다시 검토할 실익이 있는지 확인합니다.
7. 활동이 개인 기록이 없는 단순 일회성 설문인가요?복잡한 로그인 자체가 불필요한지 확인합니다.
8. 교사 대시보드에서 민감한 학생 자료나 평가 정보를 보나요?공유 계정이나 공통 비밀번호를 배제해야 하는 강한 조건입니다.
9. 학생이 비밀번호를 잊었을 때 스스로 복구할 수 있어야 하나요?실제 이메일 또는 기존 학교 계정의 필요성을 확인합니다.
10. 여러 학급과 여러 교사로 장기 확장할 계획인가요?계정 자동화와 역할 관리의 확장성을 확인합니다.
11. OAuth를 제외한 이유가 보안·정책 때문인가요, 단지 복잡해 보이기 때문인가요?정책상 금지가 아니라면 계정 운영 부담까지 포함해 다시 비교할 필요가 있습니다.

공통 비밀번호는 결과 후보가 아닙니다. 교사용 공통 비밀번호 또는 학생 전체 공유 계정은 기능 요구사항과 무관하게 개인별 권한·책임 추적·폐기 관리에 구조적인 한계가 있기 때문입니다.
