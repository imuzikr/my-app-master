01 · THE PRODUCT

먼저 “무엇을 만들 것인가”를 고정합니다

AI에게 기술 이름부터 나열하면 화면은 만들어져도 권한과 데이터 흐름이 흔들릴 수 있습니다. 아래 사용자 경험을 프로젝트의 기준 문장으로 사용합니다.

학생

학생 활동 화면

Google 계정으로 로그인하고, 자신의 활동 데이터를 입력·조회·수정·삭제합니다. 다른 학생의 데이터는 볼 수 없습니다.

교사

교사 대시보드

가입 요청과 학생 제출 현황을 보고 평가·피드백을 작성합니다. 필요하면 학생 대신 데이터를 생성·수정·삭제할 수 있습니다.

보호

Firebase 데이터·권한

Authentication이 사용자를 확인하고, Cloud Firestore가 데이터를 저장하며, Security Rules가 실제 접근을 허용하거나 거부합니다.

완성의 기준: 화면이 예쁘게 열리는 것이 아니라, 학생 A가 학생 B의 데이터와 교사 기능을 직접 요청해도 Firestore에서 거부되는 상태입니다.

이번 문서에서 다루지 않는 것

  • React·Vue 같은 프레임워크와 npm 빌드
  • Gemini·Upstage를 이용한 AI 피드백 호출
  • Firebase Admin SDK를 사용하는 대규모 계정 관리
  • 성적·건강·상담 등 고위험 개인정보 시스템

AI 피드백은 비밀 API 키와 학생 데이터의 외부 전송이 추가되므로, 기본 앱을 검증한 뒤 별도 확장 문서에서 다루는 편이 안전합니다.

03 · WHO DECIDES WHAT

교사·AI·Firebase의 책임을 섞지 않습니다

바이브 코딩의 품질은 AI에게 많은 코드를 요구하는 것보다, 누가 어떤 결정을 맡는지 분명히 하는 데서 높아집니다.

교사가 결정저장할 데이터, 학생 승인 방식, 삭제 정책, 평가 기준, 개인정보 범위
AI가 구현·설명파일 작성, 화면 동작, Firebase SDK 연결, 오류 처리, Rules 초안, 테스트 절차
Firebase가 처리Google 로그인, UID·토큰 발급, 데이터 저장, Rules에 따른 서버 측 허용·거부

AI에게 맡겨도 좋은 것

  • 기존 파일을 읽고 변경 범위를 제안하기
  • 학생·교사 화면과 공통 모듈 작성하기
  • Firebase Modular SDK 사용하기
  • 권한표를 Rules 조건으로 옮기기
  • 오류 메시지와 테스트 절차 만들기

교사가 확인해야 하는 것

  • AI가 필요 이상으로 개인정보를 저장하지 않는가?
  • 학생·교사의 실제 권한이 요구사항과 같은가?
  • 삭제가 즉시 삭제인지 휴지통인지 명확한가?
  • 실제 학생 계정으로 거부 테스트를 했는가?
  • AI가 “안전하다”고 말한 근거가 Rules에 있는가?
04 · FILE MAP

파일은 이해하기 쉬운 책임 단위로만 나눕니다

처음부터 파일을 지나치게 세분화하지 않습니다. 학생·교사·공통이라는 세 영역이면 출발하기에 충분합니다.

저장소/
├─ index.html                 # 시작 안내와 화면 이동
├─ student/
│  ├─ index.html             # 학생 화면
│  ├─ student.js             # 학생 로그인·데이터 동작
│  └─ student.css
├─ teacher/
│  ├─ index.html             # 교사 대시보드
│  ├─ teacher.js             # 승인·조회·평가 동작
│  └─ teacher.css
└─ shared/
   ├─ firebase.js            # Firebase 초기화와 공통 연결
   ├─ common.js              # 공통 로그인·오류 처리
   └─ common.css

student/

학생에게 필요한 화면과 동작만 둡니다. 자신의 UID를 기준으로 데이터를 요청합니다.

teacher/

교사 역할을 확인한 뒤 대시보드를 불러옵니다. 화면을 열었다는 사실만으로 데이터를 요청하지 않습니다.

shared/

두 화면이 함께 사용하는 코드입니다. 실제 데이터가 저장되는 폴더가 아니라 Firebase 연결 모듈을 모아 둔 이름입니다.

이름은 규칙이 아닙니다. shared, firebase.js, common.js는 이해를 돕기 위해 정한 이름입니다. AI가 다른 이름을 제안하면 역할이 겹치지 않는지만 확인하십시오.
05 · FIREBASE FOUNDATION

설정과 코드를 함께 준비해야 합니다

Firebase 콘솔에서 기능을 켜는 것만으로 로그인 화면이 생기지는 않습니다. 교사는 콘솔을 설정하고, AI는 웹 페이지에서 SDK를 불러와 로그인·상태 확인 코드를 연결합니다.

교사가 Firebase 콘솔에서 할 일

  1. Firebase 프로젝트와 웹 앱 등록
  2. Cloud Firestore 생성과 지역 선택
  3. Authentication에서 Google 로그인 활성화
  4. GitHub Pages 도메인을 승인 도메인에 추가
  5. 웹 앱의 firebaseConfig 복사
  6. Rules를 배포하고 테스트 모드 종료

AI가 코드에서 할 일

  1. 공식 CDN에서 Firebase SDK 불러오기
  2. initializeApp()으로 공통 초기화
  3. Google 로그인·로그아웃 연결
  4. onAuthStateChanged()로 상태 확인
  5. UID와 회원 역할에 맞는 화면 표시
  6. Firestore 오류를 사용자에게 안내

Config는 누가 입력하나요?

앱 제작자가 Firebase 콘솔에서 제공되는 설정 객체를 shared/firebase.js에 한 번 입력합니다. 학생이나 교사가 로그인할 때마다 입력하는 값이 아닙니다.

const firebaseConfig = {
  apiKey: "Firebase 콘솔에서 복사",
  authDomain: "Firebase 콘솔에서 복사",
  projectId: "Firebase 콘솔에서 복사",
  storageBucket: "Firebase 콘솔에서 복사",
  messagingSenderId: "Firebase 콘솔에서 복사",
  appId: "Firebase 콘솔에서 복사"
};
Firebase 웹 API 키는 공개 설정입니다. 프로젝트를 식별하는 값이며 관리자 비밀키가 아닙니다. 데이터 보호는 Authentication과 Security Rules가 담당합니다.
절대 넣지 않을 것: Firebase Admin 서비스 계정, private_key, 교사 비밀번호, Gemini·Upstage·OpenAI 비밀 API 키. GitHub Pages의 JavaScript는 방문자가 확인할 수 있습니다.

왜 CDN을 사용하나요?

방식의미현재 앱에 대한 판단
Firebase CDN브라우저가 Google 서버에서 공식 SDK를 불러옴권장 시작점 · 설치와 빌드가 없음
npm + ViteSDK를 설치하고 배포 파일로 빌드앱이 커진 뒤 검토
REST 직접 구현토큰·요청·오류 처리를 직접 작성초보 프로젝트에는 권장하지 않음

CDN은 앱 코드를 대신 만들어 주지 않습니다. SDK가 로그인 토큰과 Firestore 통신을 담당하고, 화면·데이터 흐름·오류 처리는 여전히 우리 JavaScript가 구현합니다.

06 · DATA & PERMISSION MAP

컬렉션 이름보다 먼저 권한표를 만듭니다

데이터 구조는 앱마다 달라질 수 있지만, “누가 무엇을 할 수 있는가”는 코드 작성 전에 합의해야 합니다. 아래는 현재 프로젝트의 기본 권한입니다.

작업해당 학생다른 학생교사
학생 데이터 생성허용거부학생 대신 허용
학생 데이터 조회자기 것 허용거부전체 또는 담당 범위 허용
학생 데이터 수정허용 필드만거부허용
학생 데이터 삭제정책에 따라 허용거부허용
평가·교사 피드백 수정거부거부허용
회원 역할 변경거부거부별도 정책으로 제한

AI가 제안할 수 있는 기본 데이터 지도

joinRequests/{uid}        # 미리 알 수 없는 학생의 가입 요청
members/{uid}             # 승인 여부와 student·teacher 역할
activities/{activityId}   # 교사가 만든 활동
submissions/{id}          # 학생 활동·제출 데이터

중괄호는 실제 이름이 아니라 문서 ID의 자리표시자입니다. 예를 들어 joinRequests/{uid}는 joinRequests 컬렉션 안에 학생 UID를 문서 ID로 둔다는 뜻입니다.

데이터의 주인

ownerUid는 해당 제출물이 어느 학생의 것인지 나타냅니다. 학생 Rules의 핵심 비교값입니다.

실제 작업자

교사가 대신 입력할 수 있다면 createdByUid와 updatedByUid로 데이터 주인과 실제 작업자를 구분합니다.

삭제 정책은 코딩 전에 선택합니다. 즉시 삭제는 단순하지만 복구가 어렵습니다. 수업 기록이라면 deleted: true로 숨겼다가 복구하는 휴지통 방식을 AI에게 검토시킬 수 있습니다.
07 · BUILD IN CHECKPOINTS

한 번에 전체 앱을 요청하지 않습니다

AI가 한 번에 많은 파일과 Rules를 만들면 오류의 원인을 찾기 어려워집니다. 각 단계가 실제 브라우저에서 작동하는지 확인한 뒤 다음 단계로 이동합니다.

01

정적 화면과 경로

학생·교사 폴더와 기본 화면을 만들고 GitHub Pages에서 두 주소가 열리는지 확인합니다.

통과 조건 · 두 경로가 모바일과 PC에서 열림
02

Firebase 연결과 Google 로그인

Config, CDN SDK, 로그인·로그아웃, 인증 상태 표시만 연결합니다. 아직 복잡한 데이터 기능을 넣지 않습니다.

통과 조건 · 새로고침 후 로그인 상태 확인
03

가입 요청과 역할

미승인 사용자는 가입 요청만 보내고, 교사가 승인한 학생과 교사만 각 화면에 들어가도록 합니다.

통과 조건 · 미승인 계정은 데이터에 접근 못함
04

학생 CRUD와 교사 대시보드

학생 자기 데이터와 교사 전체 현황을 연결합니다. 기능마다 로딩·성공·실패 상태를 확인합니다.

통과 조건 · 권한표대로 생성·조회·수정·삭제
05

Rules와 우회 테스트

학생이 다른 UID와 교사 코드를 직접 요청하는 상황을 시험합니다. 화면 테스트가 아니라 Firestore의 거부 결과를 확인합니다.

통과 조건 · 잘못된 요청은 permission-denied
AI와의 작업 규칙: 매 단계에서 “변경할 파일, 변경 이유, 교사가 해야 할 콘솔 작업, 성공 확인 방법”을 먼저 설명하게 하십시오. 기존 파일 전체를 매번 다시 만들게 하지 않는 편이 안전합니다.
08 · COPY & PASTE PROMPT KIT

AI에게는 한 단계씩 요청합니다

첫 프롬프트는 프로젝트의 헌법이고, 이후 프롬프트는 단계별 작업 지시입니다. 대괄호 부분만 현재 상황에 맞게 바꾸십시오.

프롬프트 1 · 프로젝트 맥락과 작업 방식 전달새 대화의 첫 메시지로 사용
나는 개발 경험이 많지 않은 교사이며, AI와 대화하면서 수업용 웹 앱을 만들려고 합니다.

[기본 환경]
- 바닐라 HTML, CSS, JavaScript를 사용합니다.
- React, Vue, npm 빌드 도구는 지금 사용하지 않습니다.
- GitHub 저장소 하나를 GitHub Pages 한 곳으로 배포합니다.
- student 폴더와 teacher 폴더에 서로 다른 화면을 둡니다.
- shared 폴더에는 두 화면이 함께 쓰는 Firebase와 공통 코드를 둡니다.
- Firebase Authentication의 Google 로그인을 사용합니다.
- Cloud Firestore에 데이터를 저장하고 Security Rules로 권한을 검사합니다.

[사용자 경험]
- 학생은 student 화면에서 자신의 활동 데이터를 생성, 조회, 수정, 삭제합니다.
- 학생은 다른 학생의 데이터를 볼 수 없습니다.
- 교사는 teacher 대시보드에서 학생들의 제출 현황을 보고 평가와 피드백을 작성합니다.
- 교사는 필요한 경우 학생 대신 데이터를 생성, 수정, 삭제할 수 있습니다.
- Google 로그인 성공과 학생·교사 권한은 별도로 판단해야 합니다.
- 불특정 학생은 로그인 후 가입 요청을 보내고, 교사가 승인한 뒤 사용하게 하는 흐름을 우선 검토해 주세요.

[AI와 작업하는 방식]
- 전체 앱을 한 번에 만들지 말고 화면, 로그인, 회원 승인, 데이터, Rules, 테스트 순서로 나누어 주세요.

- 코드에 Firebase Admin 서비스 계정, 교사 비밀번호, 외부 AI API 비밀키를 넣지 마세요.

프롬프트 2 · Firebase 연결과 Google 로그인정적 화면이 정상적으로 열린 다음 사용
이제 Firebase 연결과 Google 로그인 단계만 구현해 주세요. 아직 학생 활동 CRUD와 교사 대시보드 데이터는 구현하지 않습니다.

- 공식 Firebase JavaScript SDK의 브라우저용 CDN 모듈을 사용해 주세요.
- 모든 Firebase import는 같은 고정 버전을 사용해 주세요.
- shared/firebase.js는 Firebase 초기화, Authentication, Firestore 연결을 담당하게 해 주세요.
- shared/common.js는 Google 로그인, 로그아웃, onAuthStateChanged 기반 상태 관찰을 담당하게 해 주세요.
- Firebase 콘솔의 config 값을 붙여 넣을 위치는 명확한 자리표시자로 남겨 주세요.
- firebaseConfig의 웹 API 키는 공개 설정이지만 Admin 서비스 계정과 외부 API 키는 넣으면 안 된다고 주석으로 설명해 주세요.
- student와 teacher 화면에 로그인 중, 로그인 필요, 로그인 완료, 오류 상태를 구분해 주세요.
- 인증 상태가 확인되기 전에 학생 또는 교사 데이터를 요청하지 마세요.
- teacher 화면을 열었다는 이유만으로 교사라고 판단하지 마세요.
- GitHub Pages의 저장소 경로에서도 작동하도록 상대 경로를 확인해 주세요.

답변 순서:
1. Firebase 콘솔에서 내가 해야 할 설정
2. 변경할 파일과 이유
3. 변경한 파일의 완전한 코드
4. 로컬과 GitHub Pages에서 확인할 로그인 테스트

내 실제 firebaseConfig 값은 공개 채팅에 다시 출력하지 말고, 내가 로컬 파일에 직접 붙여 넣을 수 있도록 위치만 안내해 주세요.
교사는 Google 로그인 제공자 활성화와 승인 도메인 등록을 직접 해야 합니다. 설정만으로는 로그인 버튼이 생기지 않습니다.
프롬프트 3 · 가입 승인과 학생·교사 데이터로그인이 안정적으로 작동한 다음 사용
Google 로그인까지 정상 작동합니다. 이제 회원 승인과 수업 데이터를 구현해 주세요.

[회원 흐름]
- 학생은 미리 특정 Workspace에 등록되어 있지 않고 개인 Google 계정을 사용할 수 있습니다.
- 로그인했다고 바로 승인된 학생으로 취급하지 않습니다.
- 미승인 사용자는 자신의 UID로 가입 요청을 한 번만 만들 수 있게 해 주세요.
- 교사는 대시보드에서 가입 요청을 보고 학생으로 승인하거나 비활성화할 수 있게 해 주세요.
- 학생이 자신의 role이나 active 값을 바꿀 수 없어야 합니다.
- 초기 교사 계정을 어떻게 등록하는지 Firebase 콘솔 절차를 설명해 주세요.

[수업 데이터]
- 학생은 자신의 활동 데이터를 생성, 조회, 수정, 삭제합니다.
- 학생 데이터에는 소유 학생을 나타내는 ownerUid가 있어야 합니다.
- 교사가 학생 대신 작업할 수 있으므로 createdByUid와 updatedByUid 같은 작업자 기록을 검토해 주세요.
- 학생은 다른 학생 데이터와 교사 평가 필드를 변경할 수 없어야 합니다.
- 교사는 전체 또는 담당 범위의 학생 데이터를 조회하고 생성, 수정, 삭제하며 평가와 피드백을 작성할 수 있습니다.
- 학생 쿼리는 자신의 UID 조건을 포함해야 합니다.

[작업 방식]
- joinRequests, members, activities, submissions 같은 컬렉션 이름은 권장안으로 설명한 뒤 사용해 주세요.
- 한 활동당 제출 하나인지 여러 개인지 먼저 물어보고 addDoc과 setDoc 중 맞는 방식을 선택해 주세요.
- 삭제는 즉시 삭제와 휴지통 방식의 차이를 설명하고 내 선택을 받은 뒤 구현해 주세요.
- 화면 코드와 Rules에서 각각 무엇을 검사하는지 구분해 주세요.
- 이번 단계에서도 변경 파일과 성공 확인 방법을 먼저 설명해 주세요.

먼저 데이터 구조와 권한표를 보여 주고 내 확인을 받은 뒤 코드를 작성해 주세요.
AI가 제안한 컬렉션과 필드는 바꿀 수 있습니다. 중요한 것은 UID·역할·소유권이 일관되게 연결되는지입니다.
프롬프트 4 · Rules와 공격 관점 검토기능이 작동한 뒤 마지막으로 사용
현재 앱의 전체 파일과 Firestore Rules를 보안 관점에서 검토해 주세요.

화면에서 버튼이 숨겨지는지가 아니라, 사용자가 개발자 도구나 직접 Firebase 요청으로 우회했을 때 Firestore가 거부하는지를 기준으로 판단해 주세요.

다음을 확인해 주세요.
1. 비로그인 사용자의 모든 수업 데이터 요청이 거부되는가?
2. 로그인했지만 미승인인 사용자는 가입 요청 외의 데이터에 접근할 수 없는가?
3. 학생 A가 학생 B의 데이터를 읽거나 수정하거나 삭제할 수 없는가?
4. 학생이 ownerUid를 다른 UID로 바꾸어 생성할 수 없는가?
5. 학생이 role, active, teacherFeedback 같은 관리 필드를 변경할 수 없는가?
6. 학생이 teacher 화면의 조회·수정 코드를 직접 실행해도 거부되는가?
7. 교사만 가입 승인과 학생 전체 데이터 작업을 할 수 있는가?
8. 교사가 학생 대신 데이터를 생성할 때 대상 학생과 작업자 정보가 구분되는가?
9. 허용하지 않은 필드와 잘못된 자료형·길이가 거부되는가?
10. Rules에 테스트 모드나 allow read, write: if true가 남아 있지 않은가?
11. 여러 match 규칙이 겹쳐 넓은 권한을 허용하지 않는가?
12. GitHub Pages 코드에 Admin 서비스 계정이나 외부 API 비밀키가 없는가?

결과는 사용자 유형별 허용·거부 표로 보여 주세요. 문제가 있다면 관련 파일, 위험, 최소 수정안을 설명하고 변경할 파일의 전체 코드를 제공해 주세요.

실제로 실행하지 못한 검사는 실행했다고 말하지 말고 Firebase Emulator 또는 수동 테스트 절차를 제공해 주세요.
AI의 “문제없음”보다 학생 A·학생 B·미승인·교사 계정으로 직접 확인한 결과를 신뢰하십시오.

11 · FINAL DIRECTION

교사는 코드를 외우는 사람이 아니라 기준을 지키는 운영자입니다

AI가 코드를 작성하더라도 학생 데이터의 범위와 권한, 삭제 정책과 실제 테스트는 교사가 확인해야 합니다. 아래 네 문장만 흔들리지 않으면 구현 세부는 단계적으로 개선할 수 있습니다.

화면 ≠ 권한

경로와 버튼은 편의를 위한 것이고, 보안은 Rules에서 완성합니다.

로그인 ≠ 교사

Google 로그인 후에도 승인 상태와 역할을 별도로 확인합니다.

소유자 ≠ 작업자

교사가 대신 입력한다면 데이터 주인과 실제 작업자를 구분합니다.

완성 ≠ 검증
