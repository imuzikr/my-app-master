OAuth 동의는 교사 승인 아님

사용자가 Sheets 관련 OAuth 범위를 승인했다는 것은 Google 데이터 접근에 동의했다는 뜻이지, 교사 대시보드 권한을 받았다는 뜻이 아닙니다.

화면 표시도 권한 승인 아님

브라우저에 교사 메뉴를 보였거나 ?page=teacher를 사용했다는 사실은 서버가 데이터를 반환할 근거가 되지 않습니다.

한 문장으로 정리: Google은 계정과 Google 서비스 접근을 도와줄 수 있지만, “이 이메일은 우리 학교의 교사다”라는 앱 내부 판단은 개발자가 서버에서 구현해야 합니다.

03 · PASSWORD-BASED ACCESS

‘비밀번호 사용’도 세 가지 수준이 서로 다릅니다

A. 브라우저 공동 비밀번호

HTML·JavaScript에서 문자열 비교

// 절대 사용하지 말 것
if (input === 'teacher1234') {
  showTeacherView();
}
  • 소스 보기와 개발자 도구로 비밀번호를 확인할 수 있습니다.
  • 숨겨 둔 교사 화면을 다시 표시할 수 있습니다.
  • 서버 데이터 함수가 별도로 보호되지 않습니다.

보안 인증으로 인정하기 어려움

B. 서버 공동 비밀번호

Apps Script 함수가 매번 검사

  • 비밀번호를 브라우저 소스에서 숨길 수 있습니다.
  • 한 명에게 유출되면 모두의 비밀번호를 바꿔야 합니다.
  • 누가 접근했는지 개별적으로 증명할 수 없습니다.
  • 세션 만료, 시도 횟수 제한, 회수 기능을 직접 만들어야 합니다.

입장 코드로만 제한적으로 사용

C. 사용자별 비밀번호 직접 구현

Sheets에 계정 시스템을 직접 제작

안전한 비밀번호 해시·salt, 로그인 제한, 계정 잠금, 세션 토큰, 만료, 로그아웃, 비밀번호 재설정과 감사 기록이 모두 필요합니다.

Apps Script는 웹 앱 사용자용 비밀번호 인증 프레임워크를 기본 제공하지 않으므로, 초보자가 직접 구축하기에는 보안 책임이 너무 큽니다.

직접 구현 비권장

D. 인증 서비스의 이메일·비밀번호

예: Firebase Authentication

  • 공동 비밀번호 대신 사용자별 UID를 가집니다.
  • 한 사용자만 정지하거나 비밀번호를 재설정할 수 있습니다.
  • 토큰 발급과 비밀번호 처리를 인증 서비스가 담당합니다.
  • Firestore Rules와 연결하면 사용자별 데이터 권한을 표현하기 쉽습니다.

Google 계정 없이 개인 인증이 필요할 때 현실적

서버 공동 코드를 꼭 써야 한다면

function saveAnonymousActivity(payload, classCode) {
  const expected = PropertiesService
    .getScriptProperties()
    .getProperty('CLASS_CODE');

  if (!classCode || classCode !== expected) {
    throw new Error('접근할 수 없습니다.');
  }

  const clean = validateActivity_(payload);
  return appendLimitedActivity_(clean);
}
이 예시는 강한 인증이 아닙니다. 비밀번호 확인 결과를 브라우저 변수에 저장하고 이후 서버 함수가 그대로 믿으면 안 됩니다. 보호가 필요한 함수가 코드를 다시 검사해야 하며, 성적·상담·교사 대시보드·학생별 개인정보에는 공동 코드만 사용하지 않는 것이 좋습니다. Script Properties도 프로젝트 편집자는 볼 수 있으므로 편집자 권한을 최소화해야 합니다.
04 · GOOGLE ACCOUNT SIGN-IN

학교 Google 계정이 있다면 가장 자연스러운 기본 선택

Google Workspace 학교 계정으로 웹 앱 접근을 제한하고, 서버에서 실제 이메일을 교사·학생 명단과 대조하면 비밀번호를 직접 관리하지 않고도 개별 사용자를 구분할 수 있습니다.

1
배포 접근 범위 제한

가능하면 학교 도메인 또는 로그인 사용자로 제한하고, 꼭 필요한 경우가 아니면 익명 공개를 피합니다.

2
사용자 이메일 확인

Session.getActiveUser().getEmail()이 실제 학생·교사 계정에서 반환되는지 운영 배포 조건으로 시험합니다.

3
교사·학생 명단과 대조

이메일 도메인만 보고 교사라고 판단하지 말고, 서버의 역할 명단에서 정확한 역할을 찾습니다.

4
각 서버 함수에서 다시 확인

화면 진입 때 한 번만 확인하지 말고 조회·피드백·내보내기 같은 모든 교사 함수에서 역할을 검사합니다.

이메일을 못 얻으면 기본적으로 거부해야 합니다. Apps Script의 getActiveUser().getEmail()은 실행 방식과 보안 정책에 따라 빈 문자열을 반환할 수 있습니다. 특히 웹 앱을 배포자 권한으로 실행하는 상황에서는 방문자 이메일을 항상 얻을 수 있다고 가정할 수 없습니다. 같은 Workspace 도메인에서는 가능한 경우가 많지만, 실제 계정 시험 없이 확정하면 안 됩니다.

장점

  • 교사가 학생 비밀번호를 보관하지 않음
  • 개별 사용자 차단과 역할 변경 가능
  • 이메일 기반 활동 기록 가능
  • 학교 계정 폐기 정책과 연계 가능
  • 공동 비밀번호 유출 문제 감소

주의점

  • 학교 계정이 없는 사용자는 이용하기 어려움
  • 배포 접근 설정과 실행 주체를 정확히 선택해야 함
  • 사용자 이메일이 비어 있는 조건을 처리해야 함
  • 교사·학생 명단을 별도로 관리해야 함
  • 사용자 권한 실행은 OAuth 동의와 시트 권한 문제가 생길 수 있음
05 · OAUTH PERMISSIONS

OAuth는 로그인 비밀번호를 대체하는 말이 아닙니다

Apps Script가 비공개 Google Sheets를 읽거나 Drive 파일을 만들려면 해당 작업에 필요한 OAuth 범위를 승인받아야 합니다. OAuth는 스크립트가 Google 계정의 자원에 어느 범위까지 접근할지를 정하는 권한 위임 체계입니다.

로그인

계정 확인

사용자가 Google 계정의 주인임을 Google이 확인합니다.

앱 역할

학생·교사 결정

우리 앱 서버가 이메일과 역할 명단을 비교합니다.

OAuth

Google 서비스 접근

스크립트가 Sheets·Drive 등에 필요한 범위만 사용하도록 허가합니다.

OAuth 점검 원칙

  • 필요하지 않은 Gmail·Drive·전체 Sheets 권한을 요청하지 않습니다.
  • 공개 웹 앱은 appsscript.json에 필요한 OAuth 범위를 명시적으로 제한하는 방안을 검토합니다.
  • 사용자가 일부 범위만 허용할 수 있는 세분화된 동의 흐름을 고려해 누락된 권한을 처리합니다.
  • 배포자 권한으로 실행할 때 얻은 OAuth 토큰을 브라우저로 보내지 않습니다.
  • 학교 도메인 밖의 사용자에게 민감한 범위를 요청하면 ‘확인되지 않은 앱’ 경고와 OAuth 검증 절차가 관련될 수 있습니다.
두 번의 판단이 필요합니다. “이 사용자가 교사인가?”는 역할 권한 검사이고, “이 스크립트가 Sheet를 읽어도 되는가?”는 OAuth·실행 권한 문제입니다. 어느 하나만 해결해서는 안전한 교사 대시보드가 되지 않습니다.
06 · EXECUTION IDENTITY

Apps Script는 누구의 권한으로 실행되는가?

실행 방식Google 자원 접근 권한주요 장점주요 위험·불편적합한 상황
배포자로 실행
USER_DEPLOYING		배포자 계정학생에게 원본 Sheet를 공유하지 않고 앱이 저장 가능방문자가 배포자 권한을 간접 사용하므로 입력·권한 검사가 매우 중요. 활성 사용자 이메일이 비어 있을 수 있음학교 내부 앱에서 방문자 신원을 실제로 확인할 수 있고 서버 함수가 엄격한 경우
접속 사용자로 실행
USER_ACCESSING		접속한 사용자 계정사용자별 Google 권한과 승인 맥락이 명확사용자별 OAuth 동의가 필요할 수 있고, 학생에게 Sheet 접근 권한이 없으면 작업 실패 가능사용자가 자신의 Google 자원으로 작업하거나 파일 권한 구조가 준비된 경우

getEffectiveUser() 오용 금지

배포자로 실행하는 웹 앱에서 이 값은 방문자가 아니라 배포자입니다. 이를 이용해 모든 방문자를 교사로 처리하면 심각한 권한 상승이 발생합니다.

getActiveUser()는 시험 후 사용

실제 운영 /exec URL을 학생 계정·교사 계정·외부 계정에서 열어 이메일 반환과 접근 거부를 각각 확인합니다.

한 배포와 두 배포의 차이: 한 배포의 학생·교사 화면은 같은 실행 주체와 접근 범위를 공유합니다. 교사용 웹 앱만 교직원 그룹으로 제한해야 하거나 실행 설정을 달리해야 한다면 별도 배포를 검토할 수 있습니다. 그래도 모든 교사 서버 함수의 역할 검사는 유지해야 합니다.
07 · RECOMMENDED PATTERNS

사용 환경에 따른 현실적인 선택

모두 학교 Google 계정 사용

추천: 학교 도메인 또는 로그인 사용자로 접근 제한 → 활성 사용자 이메일 확인 → 서버 명단에서 학생·교사 역할 확인 → 모든 서버 함수에서 재검사

계정 없이 짧은 비민감 활동

제한적 허용: 서버에서 학급 입장 코드 확인 + 저장 필드·횟수 제한. 개인 인증이나 성적 시스템으로 확대하지 않습니다.

Google 계정 없이 개인별 기록 필요

추천: Firebase Authentication 같은 인증 서비스와 UID 기반 데이터 권한을 사용합니다. 가능하면 Firestore까지 같은 보안 모델로 구성합니다.

민감한 교사 대시보드

추천: 공동 비밀번호만 사용하지 않습니다. 교사 개별 계정, 교사 허용 명단, 서버 역할 검사와 필요한 경우 교사용 별도 제한 배포를 조합합니다.

초보자에게 권하지 않는 혼합

Firebase Auth + Apps Script + Sheets를 한 번에

가능은 하지만 Apps Script 서버가 Firebase ID 토큰의 서명·발급자·대상·만료를 올바르게 검증해야 합니다. 인증과 데이터 권한이 두 플랫폼에 나뉘어 초보자에게는 오히려 복잡할 수 있습니다.

Google 로그인 + 교사 공통 비밀번호

Google 사용자를 알 수 있는데도 역할 명단 대신 공통 비밀번호를 쓰면 개별 권한 회수와 감사 가능성을 잃습니다. 교사 이메일 허용 목록이 더 명확합니다.

08 · SERVER-SIDE AUTHORIZATION

화면이 아니라 서버 함수가 최종 문지기입니다

doGet에서 교사 화면을 막는 것만으로 충분하지 않습니다. 브라우저 사용자는 클라이언트 코드를 바꾸거나 공개된 google.script.run 함수를 직접 호출할 수 있다고 가정해야 합니다.

function requireSignedIn_() {
  const email = Session.getActiveUser().getEmail()
    .trim()
    .toLowerCase();

  if (!email) {
    throw new Error('사용자 신원을 확인할 수 없습니다.');
  }
  return email;
}

function requireRole_(allowedRoles) {
  const email = requireSignedIn_();
  const role = lookupRoleOnServer_(email); // teacher / student / admin

  if (!role || !allowedRoles.includes(role)) {
    throw new Error('권한이 없습니다.');
  }
  return { email, role };
}

function getTeacherDashboard() {
  const actor = requireRole_(['teacher', 'admin']);
  return readTeacherDashboard_(actor);
}

function saveStudentWork(payload) {
  const actor = requireRole_(['student', 'teacher']);
  const clean = validateStudentWork_(payload);

  // payload.email이나 payload.role을 믿지 않고 인증된 이메일 사용
  return saveWorkFor_(actor.email, clean);
}
이 패턴의 전제: 현재 배포 조건에서 getActiveUser().getEmail()을 안정적으로 얻을 수 있어야 합니다. 빈 이메일을 학생으로 간주하지 말고 거부합니다. 역할 명단은 브라우저가 아니라 서버에서 읽습니다.

09 · COMMON SECURITY FAILURES

초보자 프로젝트에서 자주 생기는 오해와 결과

×
비밀번호 입력창이 있으니 안전하다

비밀번호가 JavaScript에 있으면 사용자는 비교 문자열과 로직을 확인할 수 있습니다.

×
Google에 로그인했으니 교사다

로그인은 계정만 확인합니다. 교사 역할은 별도 허용 목록으로 검사해야 합니다.

×
OAuth에 동의했으니 앱 권한도 있다

OAuth 범위는 Google 서비스 접근 권한이며 학생·교사 역할과 다릅니다.

×
?page=teacher는 교사 전용 주소다

누구나 주소를 입력할 수 있습니다. 화면 요청 값은 인증 정보가 아닙니다.

×
교사 버튼을 숨기면 함수도 보호된다

브라우저에서 버튼을 다시 표시하거나 서버 함수를 직접 호출할 수 있습니다.

×
클라이언트가 보낸 학번은 본인 것이다

다른 학생의 학번이나 행 번호로 바꿀 수 있으므로 인증된 사용자와 서버에서 연결해야 합니다.

×
getEffectiveUser()가 방문자다

배포자로 실행할 때는 배포자 계정이므로 방문자 인증에 사용하면 안 됩니다.

×
원본 Sheet 링크만 숨기면 된다

서버 함수가 권한 없이 데이터를 반환하면 Sheet를 비공개로 해도 앱을 통해 유출됩니다.

10 · RELEASE CHECKLIST

실제 학생 데이터를 연결하기 전 점검표

인증·역할·배포

  • 인증, 학생·교사 역할, OAuth 권한을 별도로 설명할 수 있다.
  • 배포 접근 범위가 필요한 최소 수준으로 제한되어 있다.
  • 배포자 실행과 접속 사용자 실행 중 선택 이유를 기록했다.
  • 실제 학생·교사 계정에서 활성 사용자 이메일을 시험했다.
  • 이메일이 비어 있으면 접근을 거부한다.
  • 교사 이메일은 서버의 허용 명단과 대조한다.
  • 모든 교사 서버 함수가 역할을 다시 확인한다.
  • getEffectiveUser()를 방문자 신원으로 사용하지 않는다.

비밀번호·데이터·OAuth

  • HTML·JavaScript에 비밀번호나 비밀 키가 없다.
  • 공동 코드를 개인 인증으로 표현하지 않는다.
  • 학생이 보낸 이메일·역할·학번·행 번호를 그대로 믿지 않는다.
  • 원본 Sheet는 학생에게 불필요하게 직접 공유되지 않았다.
  • OAuth 범위는 필요한 Google 서비스로 제한했다.
  • OAuth 토큰이나 서버 설정 값을 브라우저로 보내지 않는다.
  • 로그에 비밀번호, 토큰, 과도한 개인정보를 남기지 않는다.
  • 계정·역할 변경, 데이터 보존과 사고 대응 절차가 있다.

하나라도 해당하면 배포를 멈춰야 하는 항목

  • 교사 비밀번호가 HTML 또는 JavaScript에서 확인된다.
  • 교사 화면 URL만 알면 학생 전체 데이터를 조회할 수 있다.
  • 교사 서버 함수가 브라우저의 role: 'teacher' 값을 믿는다.
  • 사용자 이메일이 비어 있어도 교사 또는 로그인 사용자로 처리한다.
  • 배포자로 실행하면서 getEffectiveUser()를 방문자라고 생각한다.
  • 공동 비밀번호 하나로 성적·상담·개인정보를 조회하거나 수정한다.
11 · BEGINNER RECOMMENDATION

초보자에게 가장 안전한 선택 기준

보안 기능을 직접 많이 만들수록 안전해지는 것이 아니라, 검증된 계정 시스템에 맡기고 앱 서버에서는 역할과 데이터 범위만 명확하게 검사하는 편이 더 현실적입니다.

학교 계정이 있음

Google 로그인 + 학교 접근 제한 + 역할 명단

Apps Script와 Sheets를 유지할 때 가장 자연스러운 기본안입니다.

학교 계정이 없음

Firebase Authentication + UID 기반 권한

개인별 기록과 계정 관리가 필요하면 인증 전용 서비스를 사용합니다.

비민감·일회성 활동

서버 측 학급 코드 + 최소 데이터

입장 편의를 위한 장치로만 사용하고 민감한 기능으로 확대하지 않습니다.

교사 대시보드의 최소선: 개별 교사 계정 식별, 서버의 교사 허용 목록, 모든 조회·수정 함수의 역할 검사입니다. 공동 비밀번호 하나만으로는 이 기준을 충족하지 못합니다.

FINAL · INTERACTIVE SECURITY AUDIT

현재 설계의 인증 보안을 직접 점검해 보세요

12개 항목에 답하면 준비도를 계산하고, 치명적인 위험이 있으면 점수와 관계없이 별도로 경고합니다. ‘일부·모름’은 확인이 필요한 상태이므로 절반 점수로 계산합니다.

1. 웹 앱 접근을 학교 도메인 또는 로그인 사용자로 제한했나요?익명 공개는 사용자 신원을 확인하기 어렵게 합니다.
2. 학생과 교사를 개별 계정 또는 고유 UID로 구분하나요?공동 비밀번호만으로는 누가 행동했는지 증명하기 어렵습니다.
3. 실제 학생·교사 계정에서 활성 사용자 이메일 반환을 시험했나요?스크립트 소유자 계정만 시험해서는 부족합니다.
4. 사용자 이메일이 비어 있으면 접근을 거부하나요?빈 값을 학생이나 교사로 자동 처리하면 안 됩니다.
5. 교사 여부를 서버의 교사 명단으로 확인하나요?도메인, URL, 화면 변수 또는 공통 비밀번호만 믿지 않습니다.
6. HTML·JavaScript에 비밀번호나 비밀 키가 없나요?화면 소스와 개발자 도구에서 확인 가능한 값은 비밀이 아닙니다.
7. 모든 교사 서버 함수가 역할을 독립적으로 확인하나요?화면 진입 때 한 번 확인한 결과만 믿지 않습니다.
8. 서버가 브라우저가 보낸 이메일·역할·학번을 그대로 믿지 않나요?인증된 사용자와 데이터 소유권을 서버에서 연결해야 합니다.
9. 배포자 실행과 접속 사용자 실행 중 선택 이유를 알고 시험했나요?실행 주체는 사용자 식별, Sheet 권한과 OAuth 동의에 영향을 줍니다.
10. getEffectiveUser()를 방문자 신원으로 사용하지 않나요?배포자로 실행할 때는 배포자 계정을 반환합니다.
11. OAuth 범위를 필요한 Sheets·Drive 기능으로 제한했나요?사용하지 않는 민감한 Google 서비스 권한은 제거합니다.
12. 원본 Sheet, 로그, 계정·역할 변경 절차를 보호하나요?앱 코드뿐 아니라 운영 데이터와 관리자 권한도 점검합니다.
